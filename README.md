# 测试三步走 · Three-Step Testing Skills

一套通用的、面向 Web 应用的**系统化测试**技能集（Agent Skills）。把"**接案确认 → 理解项目 → 编写用例 → 执行测试**"标准化成 5 个互相联动的 skill，让 AI 以**代码 + 浏览器双重查证**为准，产出可追溯、不夸大、图文并茂的测试结论。

- **不绑定任何具体项目**——换一个被测系统重新触发即可。
- **不绑定任何具体终端**——这套 skill 用通用的"**子代理（subagent）+ 只读取证 + 完成度闸门**"骨架编写，凡是支持 Agent Skill 与子代理的 AI 编码终端都能用：**Claude Code、Antigravity（反重力）、opencode** 等。各终端的"扇出子代理 / 返回结构"按各自约定类推即可。

---

## 包含哪些 skill

| Skill | 作用 | 主要产出 |
|---|---|---|
| `testing-step0-intake` | **⓪ 接案 + 进度总管/续跑**：收集核实被测系统信息（入口/账号/数据库指向/源码/文档/环境约束），自动发现 + 对话确认；并作为"总管"扫描各步进度、定位续跑点，长任务中断或开新对话也能无缝接续 | 《项目信息确认卡》《进度总览》 |
| `testing-step1-understand-project` | **① 全面理解项目**：通读前后端代码、逐页跑浏览器、看渲染对账、通读需求文档，四线交叉印证 | 《项目理解文档》（API / 页面 / 状态机 / 业务闭环 / 权限 / 数据基线） |
| `testing-step2-write-testcases` | **② 编写测试用例**：按"7 测试面 × 横切维度"矩阵铺用例，穷举可操作元素与交互方式，完整统计状态机回环 | 测试用例集 + 覆盖矩阵 + 追溯 |
| `testing-step2-doc-conformance` | **②的子项(2.1) 文档符合性**：逐原子对照需求文档，判定"有 / 无 / 部分" | 原子级符合性核实表 |
| `testing-step3-execute-tests` | **③ 执行测试**：经浏览器 / 接口 / 数据库回读 / 代码佐证四通道真实执行并取证，执行后做对抗性诚实审计 | 测试报告 + 缺陷台账 + 失败用例 + 诚实审计 |

**5 个 skill 全链路双向联动** ⓪↔①↔②↔②.1↔③：上一步产出是下一步输入；第⓪步作为"总管"贯穿全程，统一把产出落在被测项目下的 `测试产出/` 目录。

## 运行环境与模型建议

这套 skill 是**终端无关**的，但不同"终端 × 模型"组合的实际表现差别很大——核心差异在于**模型会不会"偷懒"**（执行不彻底、跳步、用读代码推断冒充真实执行）。实测建议如下：

| 终端 | 推荐模型 | 表现 / 注意事项 |
|---|---|---|
| **Antigravity（反重力）** | `gemini-3.5-flash` | ✅ **推荐**。不偷懒、很勤劳，会老老实实把步骤跑完，输出内容也相当完善。主要是便宜啦ouo。 |
| **Claude Code** | Claude（Opus / Sonnet） | ✅ 推荐，但**务必拉高执行强度**：用 `/effort` 指令开启 **ultracode** 模式，或在需要广覆盖/对抗审计时**手动使用 workflows** 做多代理编排，才能发挥这套 skill 的完整威力。 |
| **opencode** | 国产模型（国模） | ⚠️ 可用，但**配合国模容易遇到"模型偷懒"问题**——执行半途收尾、跳过取证。需要时多催、拆小任务、显式要求"逐条真跑并贴证据"。 |

## 安装

每个 skill 可单独安装：

```bash
npx skills add fsdbujgvr/three-step-testing-skills --skill testing-step0-intake
npx skills add fsdbujgvr/three-step-testing-skills --skill testing-step1-understand-project
npx skills add fsdbujgvr/three-step-testing-skills --skill testing-step2-write-testcases
npx skills add fsdbujgvr/three-step-testing-skills --skill testing-step2-doc-conformance
npx skills add fsdbujgvr/three-step-testing-skills --skill testing-step3-execute-tests
```

> 建议 5 个一并安装，以获得完整的接案→执行链路。

## 怎么用

安装后，用**自然语言**描述意图即可触发对应 skill（无需记 skill 名）。推荐从第⓪步开始，让它带着你走：

1. **从第⓪步进入**——把项目的说明文档 / `docker-compose.yml` / 需求文档丢给它，或直接说"你先扫下这个项目，看看够不够测"。它会自动发现信息、就缺的/不确定的/危险的反问你，并告诉你接下来该做什么。
   > 例：「我丢个 docker-compose 和需求文档给你，看看信息够不够测」

2. **理解项目**：信息确认后进入第①步，全面理解被测系统。
   > 例：「全面了解一下这个系统，为写测试用例打地基」

3. **编写用例**：基于理解文档铺最全测试用例（含文档符合性）。
   > 例：「基于第①步理解文档，写最全测试用例」

4. **执行测试**：真实执行并取证，产出图文并茂的报告。
   > 例：「把第②步用例跑一遍，出图文并茂的报告」

**续跑**：任务中断或开新对话时，直接说「上次测到哪了、从哪继续」，第⓪步会扫描进度并接续。

## 适用范围

- **最适用**：带前端、有源码、有可登录账号的标准 Web 应用。
- 也适用：纯后端 API、电商交易、纯 CRUD 后台、实时协作等形态——7 面骨架与诚实纪律通用，具体专项按项目特性条件触发。
- 异形态（纯 API / Canvas / 移动端 / WebSocket 等）的降级取证见 `testing-step1-understand-project/references/non-standard-forms.md`。

---

## 设计原则

- **以代码 + 浏览器查证为准**——需求文档只是"声称"，一切以可复现实测为准。
- **诚实不夸大**——区分"真测 / 源码佐证 / 阻塞 / 待补"，不把读代码包装成已执行。
- **探索期默认只读**——不写部署库；确需写库须显式授权并可逆清理。
- **通用而非特例**——7 测试面 + 横切维度是通用骨架，具体字段/公式/文案从被测项目重新派生。
