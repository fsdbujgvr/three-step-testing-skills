# 接案确认 + 进度总管（测试三步走 · 第 0 步）

## 唯一目标
做整个三步走的**入口与总管**，两件事：
1. **首次接案**：把测试所需信息收集齐、核实准 → 产出《项目信息确认卡》。
2. **进度总管/续跑**：扫描 `测试产出/` 下各步产出，判定"上次跑到哪、从哪继续"，维护《进度总览》状态台账，规划并路由到下一步。

**为什么要有总管**：长任务常中断、用户会开新对话——必须有个能一眼看出进度、无缝续跑的入口，不靠人脑记。**所有三步走产出统一落在 `测试产出/`，第0步据此扫描。** 未过第 5 节闸门前不放行；只收集/核实/规划，**不测试、不写库、不深入探索**（探索属第①步）。

## 0. 一进来先判模式（自动）
看被测项目工作目录的 `测试产出/`：
- **不存在 / 空 / 无任何步骤产出** → **首次接案模式**（走 §1~§2）。
- **已有产出** → **续跑/总管模式**（先走 §3 扫进度、报告续跑点，再据缺口决定：补接案 / 直接进某步）。
无论哪种，**结束前都更新《进度总览》**（§4）。先扫一眼 `测试产出/第0步-接案与进度/进度总览.md`（若有）——它是上次留下的状态记忆，优先据它快速定位。
> **★判模式时排除"非正式产出"目录（防误判模式）**：判"有无产出"只认**正式步骤目录**(第0~3步默认名 ∪ 英文别名，见 §6)。`测试产出/` 下以 `_` 开头或名含"小测/演练/probe/bak/备份/审计"等的目录是**沙盒/旁路产物，不计入模式判定**(否则首次接案的项目会因有这类目录被误判成"已有产出→续跑模式"，扫空或路由错)。同理，被测项目根目录散落的既往测试材料(差异点用例、历史报告、截图)**不当步骤产出**，但**作为需求/历史线索登记进确认卡字段4**(它们常揭示真被测对象，别丢)。

## 0.5 面向用户的上手交代（必做 · 别只顾内部路由）
本步不只是收集信息和内部路由，更要**指导用户怎么用这套 skill**——用户可能完全不懂"测试三步走"。所以**用人话主动向用户交代**（不要默默扫完就跳下一步）：
- **首次接案时**，开场先讲清：① 这是"测试三步走 ⓪接案→①理解项目→②写用例→③执行测试"，各步分别干嘛、产出什么；② 现在在第 0 步，做完会进哪一步；③ **需要你(用户)配合的清单**——尤其"**确认数据库是本地还是远程/生产**、给齐各角色账号密码、是否授权写库/造数、有哪些项物理测不了"；④ 你可以**随便丢文档/文件或让我自己扫**，不用手填大表。
  - **★在入口即识别部署形态并向下游传递**：若是**纯后端 API / 微服务（无前端）**，重点确认并记入确认卡：**契约文件位置（OpenAPI/Swagger/proto/GraphQL schema）、认证方式（token/cookie-session/API-key/mTLS/OAuth）、上下游依赖服务、各下游 DB/MQ/缓存指向**——并在《进度总览》标明形态，使第①/②/③步据此做形态分流（如以契约端点 N/N 替代页面 M/M）。
- **续跑/总管时**，先告诉用户：当前整体进度、上次跑到哪、**下一步建议从哪继续**；并提示"以后断了或开新对话，直接说『**上次测到哪了**』『接着继续』我就能接上"。
- **每次要进入下一步前**，先用一句话告诉用户：**接下来进第几步、用哪个 skill、它会干什么、需要你再提供/确认什么**，得到认可或必要授权（尤其写库、连生产库）再走。
> 一句话：**信息收集要让用户看懂全貌、知道自己要配合什么、知道怎么续跑**，而不是把用户蒙在鼓里只在内部跳转。

## 1. 接案 · 接受任意输入形态（先扫后问，别让用户填一大堆）
- **用户丢的**：项目说明 md、需求文档、`docker-compose.yml`、`.env`、截图、口头几句——任何形式都收。
- **主动扫项目**：读 README / `docker-compose*.yml` / `.env*` / `Dockerfile` / `package.json`·`pyproject.toml` / 源码入口 / **文档目录（`文档/`·`docs/`·`需求/` 等——逐个列出找到的每份文档、别漏，这是第②步文档符合性逐条对照的直接输入）**。**默认只读、不改不跑。**
- 能从项目读出来的**先自己读出来**，只就"扫不到/不确定/危险"的反问；用户说"你自己扫"就去扫，别反过来甩表。

## 2. 接案 · 要收集核实的字段 + 危险三项（逐项标 ✅已确认/🟡已发现待确认/❓缺失）
字段去哪找、怎么轻量核实见 `references/discovery-map.md`。
| # | 字段 | 备注 |
|---|---|---|
| 1 | 入口 URL + 部署形态 | 怎么访问、Docker/裸机/云、容器名 |
| 2 | 各角色凭据(账号密码 / token / API-key / 证书) | 每个角色都列全；按认证形态收对应凭据 |
| 3 | **★数据库：连接 + 指向(本地副本/远程/生产-VPN后)** | **最关键**；宿主IP/VPN段 ≠ 本地，用网关/网段佐证 |
| 4 | 需求/产品/测试文档：位置+格式 | **主动扫 `文档/`·`docs/`·`需求/` 等目录、逐个列出找到的每份文档(需求/技术要求/操作步骤/用户手册/测试记录单等)，并问用户"还有别的文档吗、第②步要不要逐条对照"**；无则显式记"无"、不影响其他面 |
| 5 | 前后端源码位置(含契约/OpenAPI/proto 位置) | 判定基准(代码>API>UI>文档) |
| 6 | 环境约束 | VPN / 写库授权范围 / 哪些物理测不了 |
- **轻量核实**（确认信息真且准，不深探）：URL 可达、账号可登(尤其角色级:后端能登≠前端能进)、★DB 指向核实。
  - **★"账号可登"核实与"绝不写库"的调和**：真实登录(POST /login)常会**写 session/last_login**，与第0步只读纪律相抵。优先用**只读手段**核账号：token/API-key 直接打一个只读端点(如 `GET /whoami`、`GET /current-user`)看是否 200、或核 token 是否失效；**只有必须验密码且无只读替代时**才真登录，并在确认卡注明"登录有写 session 副作用、已最小化(只登一次)"。核不动时标 🟡(账号可用性待第①/③步自然验证)，不假装已核。
- **危险三项必须显式确认**（默认用 `AskUserQuestion`）：① DB 连本地还是远程/生产(连生产=高风险，须授权或先切本地副本)；② 写库授权范围；③ 哪些物理测不了。
  - **★无交互场景的降级协议（subagent/批处理/一次性管道里 `AskUserQuestion` 不可用时）**：不得因"问不了"就卡死或假装已确认。改为——**上游(用户/任务)已书面给出的授权信息可代替问答**(在确认卡里注明"据书面授权"+原文出处)；**未书面覆盖的危险项**逐项标 🟡 待确认、写进"待问清单"、并**设为进入写库步骤(第③步)的前置闸门**(即：可继续往纯读步骤如第①步走，但写库前必须由用户补确认)。这样既不违"诚实确认"、也不让无交互场景结构性卡死。
- **🟡按形态附加危险项确认（命中即必确认，不命中整项跳过）**：
  - **电商/交易类**（含支付/订单/库存）：支付/退款=**生产网关 vs sandbox**、测试是否**产生真实资金流**、回调能否**安全重放**(幂等/签名)、订单/库存能否**隔离造数**不污染真实交易。
  - **CRUD 后台/配置类**：能否改**角色/权限配置**（会影响其他在线用户的可见性/可用性）、删除是**软删还是硬删**（硬删不可逆）；对真实业务记录建议**优先 QA 造数**或**测前快照、测后还原**。

## 3. 进度总管 · 扫所有步骤，定位"跑到哪、从哪继续"
扫 `测试产出/` 判定每步 **未开始 / 进行中 / 已完成** + 续跑点（判定信号与定位法见 `references/progress-and-resume.md`）：
- **第⓪步**：有《项目信息确认卡》+ 危险三项确认 + 闸门过？
- **第①步**：有《项目理解文档》+ 完成度闸门(API N/N、页面 M/M、状态机 S/S、闭环图谱…)达标？
- **第②步**：有用例集 + 覆盖矩阵 + 文档符合性核实？面×维度矩阵填满？（缺哪些面）
- **第③步**：有执行记录 + 进度跟踪表？**真测率多少、哪些面/模块已done vs pending**（step3 常是部分完成，要定位到面/模块/TC 续跑点）。
- **关键**：别只看"文件夹/文件存在"就判完成——**要看完成信号/闸门/进度跟踪表的真实数字**（防把"有产出"当"已完成"）。
- **★按形态把 N/A 信号剔出分母（别把本就 N/A 当未完成）**：先读第①步的**形态结论**（确认卡/进度总览里登记的部署形态）。对**纯 API / 无前端形态**，把"**页面 M/M、前端源码八类、UI 截图覆盖、面1 UI 闭环**"等前端专属信号自动视为 **N/A，不计入完成度分母**（改以契约端点 N/N、面2/3 接口与数据等为分母）；含前端的项目这些信号照常计入。形态未知时先确认形态再判完成度，避免对纯后端项目误判"未完成"。
产出：**进度小结 + 下一步建议 + 精确续跑点**（下一步是哪步、从哪个面/模块/TC 接着跑）。
- **★反哺回路触发（闭合 ③→②，别让漏项悬空）**：扫到第③步《_第②步漏项反哺清单.md》**非空、且其"建议补成的 TC"尚未进第②步覆盖矩阵** → 在《进度总览》标 **"待第②步反哺增补"**，并把下一步**路由指回第②步**（按 `case-conventions` 补成正式 TC、回填矩阵，标 added-after），消化完再回第③步——由总管负责"谁何时把反哺清单喂回②"，不靠人记。

## 3.5 ★产出真实性深度审核（总管验真 · 别只信自报 · 出"通过/补救提示词"）
判一步"是否**真**做完、能否放行下一步"时（尤其该步由便宜/易偷懒模型跑过、或用户要求复核）——**别只读执行者自报的《进度跟踪表》/真测率就采信**（实测过：自报真测率 66% 实为 22%、整面空壳凑数、借图、偷换分母）。这是把 §3"看真实数字、防把有产出当已完成"落成可操作的**三层**（清单/命令/提示词模板见 `references/audit-checklist.md`）：
- **① 机械层（确定性 · 无法橡皮图章）**：a) 该步**强制产物**齐不齐（缺=未过闸）；b) **回链分母对账**——③执行数 vs ②用例总数、②用例 vs ①结构分母(API/页面/状态机/模块)，缺的=上层有下层无（防分母被悄缩/偷换，如面4 ②287原子、③只判22条却自报"未测0"凑100%）；c) **空证据/凑数 grep**（`before=,after=` 空、整批同句"阻塞"样板、SKIP/源码佐证 占比）；d) **借图**（`md5sum` 抓同字节图冒充多条声称）；e) **报告自相矛盾**（自报阻塞/截图数 vs 执行记录实计）。
- **② 语义层（读内容辨真假）**：环境支持子代理时**派独立审核子代理**分块读产出，判 真测 vs 空壳/源码佐证、逐项**举反例**（不支持子代理则主循环对照清单亲核；判定依据留痕）。
- **③ 重算真实口径**：据机械数得出**真实真测率/完成度**（不抄执行者自报）。
**输出两态**：
- **过** → 明确告知"✅第X步真达标、可放行进下一步"，更新《进度总览》。
- **不过** → ⓐ《进度总览》记真实数字 + 缺口清单；ⓑ生成一段**可直接粘贴回执行终端的《补救提示词》**（逐条命令式：哪面/哪文件/缺什么/按哪个 skill§ 怎么补/要什么真证据 + "补完重跑审核复审"；模板见 `references/audit-checklist.md`）。**总管只判定 + 出提示词，不亲自补/不写库**——补由执行方做。
> **边界（不与"不深入探索"冲突）**：审核主体 = **审产出文件 + 回链上游已落盘的分母**（只读）；可对分母做**轻量代码 grep 抽检**（如①报 API=N，抽几个回代码核数目对不对），但**不重做第①步探索**。本审核是【外部复核】，与各步内置 §5.5/§8.5/§9 **自审互补**（自审是第一道、易被懒模型自己跳过；本步是**跳不掉的外部兜底**）。**两终端经共享 `测试产出/` 目录协调**（无需插件）：执行终端写产出、审核终端读产出 + 写《进度总览》。**形态裁剪照旧**：纯 API/无前端形态把 UI/截图类强制产物与分母整组标 N/A，不当缺口（见 §3 形态剔除）。

## 4. 《进度总览》状态台账（第0步产出的"记忆"，新对话靠它续跑）
落在 `测试产出/第0步-接案与进度/进度总览.md`——整个三步走的状态记忆，也是用户要的"把收集的所有信息变成记忆放这里"。内容（模板见 `references/progress-and-resume.md`）：
- **项目信息确认卡**（§2 的 6 字段最终值 + 危险三项确认 + 证据）；
- **各步状态表**（未开始/进行中/已完成 + 完成度% + 关键产出路径 + 闸门是否过）；
- **当前续跑点**（下一步该做什么、从哪个面/模块/TC 继续）；
- **环境与续跑须知**（DB 指向、是否需 VPN、override 等当前环境状态）、已知风险/待确认、上次中断原因。
> **写后回读铁律**：每完成一步、或预感要中断前，**更新本台账并回读确认落盘**——这是新对话无缝接续的唯一依据。台账要随进度活更新，不能写一次就废。

## 5. 完成度闸门（禁纯自答）
- **接案模式**：6 字段全 ✅已确认（非仅发现）+ 危险三项显式确认 + 账号可用性已核实 + 无文档显式记"无" + 《项目信息确认卡》落盘。
- **明确给结论（必做，别只靠闸门隐含）**：过闸 → 用一句话明确告知用户**"信息已足够开测，可进第①步"**；未过 → 明确说**"还差 X 字段 / Y 项不确定，先补齐再走"**——给用户一个不含糊的"够/不够"判语。
- **总管模式**：`测试产出/` 各步已扫、状态判定基于**完成信号/闸门真实数字**（非仅文件存在）+ **精确续跑点已定位** + 《进度总览》已更新落盘并回读确认。
- **判某步"已完成"前的深度审核（§3.5）**：若该步产出由便宜/易偷懒模型跑过、或用户要求复核，**须过 §3.5 产出真实性深度审核**（机械 + 语义独立重算，不信自报）——审核**不通过**则该步判 **"未完成 · 待补救"**、出《补救提示词》，**不放行下一步**。
- 残余不确定项**诚实标注**（标"未核实+原因"，不假装确认/不假装已完成）。

## 6. 产出与位置（★统一根目录，总管的前提）
**所有三步走产出统一落在被测项目工作目录的 `测试产出/` 下**：第0步=`第0步-接案与进度/`、第①步=`第一步-全面了解项目/`、②=`第二步-编写测试用例/`、③=`第三步-执行测试/`。第0步据此根扫描总管。**各步保留了英文别名选项（根 ./test-output/、子目录 project-understanding/test-cases/test-execution）；若用了别名根或别名子目录，必须在《进度总览》里记下真实根与子目录映射**，使总管不迷路。**扫描按"中文默认名 ∪ 英文别名"双名匹配**（见 `references/progress-and-resume.md` §一）：优先读《进度总览》登记的真实路径，未登记才回退默认双名匹配。

## 7. 交接 / 路由（据进度决定去哪步）
- 信息没齐 → 先补接案（§1~§2）；
- 信息齐、未理解 → 第①步 `testing-step1-understand-project`；
- 已理解、未写用例 → 第②步 `testing-step2-write-testcases`（+ 2.1 `testing-step2-doc-conformance`）；
- 已写用例、未执行/执行中 → 第③步 `testing-step3-execute-tests`（**从《进度总览》的续跑点继续**）。
- **第③步已发现反哺漏项**（《_第②步漏项反哺清单》非空未消化）→ **先路由回第②步增补**，消化后再回第③步（见 §3 反哺回路触发）。
- **§3.5 深度审核不通过** → **不放行下一步**；把生成的《补救提示词》交执行终端（用户粘贴给执行 ag）补齐，**补完重跑审核复审**；路由 = "回第X步补救"。
路由前确保《进度总览》已记下"将进入哪步、从哪继续"。

## 8. 反模式（务必避免）
❌ 上来甩大表让用户填——应先扫再就缺口精准问。
❌ 把"扫到了/文档写了"当"已确认"、把"有产出文件"当"已完成"——必看核实/闸门/进度真实数字。
❌ **判完成只读执行者自报的《进度跟踪表》/真测率就采信、不独立重算**——便宜/易偷懒模型常自报注水（自报66%实22%）、整面空壳凑数、借图、偷换分母；要复核时总管必走 §3.5 机械+语义独立审核。
❌ **没确认 DB 连哪就放行**——连错生产库、误造数/误删的根源。
❌ **不更新《进度总览》**——新对话续不上，总管形同虚设。
❌ **只顾内部扫描/路由，把用户蒙在鼓里**——不向用户交代四步法全貌、当前在哪步、需要配合什么、怎么续跑（违 §0.5）；用户不懂这套 skill 时尤其要讲清。
❌ 收集/总管阶段就去写库/真跑用例/深入探索——越界。
