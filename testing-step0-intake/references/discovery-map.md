# 自动发现地图：各字段去哪找 + 怎么轻量核实

> 配合 SKILL.md 第 2~3 节。原则：**先从项目里读出来，再请用户确认**。专名仅举例，套到任意项目把路径换成实际的。默认只读，不改不跑业务。

## 1. 入口 URL + 部署形态
- 找：`docker-compose*.yml`(ports 如 `<宿主端口>:80`→入口端口；service/container_name)、README/部署文档、前端 dev server 配置、nginx/反代配置。
- 核实：`docker ps` 看实际在跑的容器与端口；浏览器/`curl` 打一下入口看是否可达、是 HTTP 还是 HTTPS。
- 注意：dev 与 prod 常是不同 compose（不同端口/镜像 tag），**确认当前在跑的是哪套**。

## 2. 各角色账号
- 找：README/文档里的测试账号；种子脚本(`seed*`/`fixtures`)；`.env` 里的初始管理员；需求文档的角色表。
- 核实：核心角色**真登录一次**确认可用；**特别注意角色级**——后端能登录(API 200)≠前端能进入（可能被路由守卫踢回登录页）。每个角色都要列全（管理员 + 各业务角色）。
  - **★但登录常写 session/last_login——与第0步"绝不写库"相抵**：优先**只读手段**核账号(token/API-key 打只读端点如 `GET /whoami`/`GET /current-user` 看 200、或核 token 是否失效)；仅当必须验密码且无只读替代时才真登录(只登一次、注明 session 写副作用已最小化)；核不动标 🟡。详见 SKILL.md §2"账号可登核实与'绝不写库'的调和"。
- 缺失就问用户：有哪些角色、各自账号密码。

## 3. ★数据库：连接 + 指向（本地 / 远程 / 生产）——最关键，最易错
- 找：后端 `DATABASE_URL`/`DB_HOST`（在 `.env` / `docker-compose.yml` environment / 配置文件 / `docker inspect <backend> 的 Env`）。
- **判断指向（别想当然）**：
  - 看 host 是 `localhost`/`127.0.0.1`/`db`(同 compose 服务名)/`host.docker.internal` → 多半**本地**；
  - 看 host 是某内网 IP（私有地址段，如 `<内网网段>`）→ **可能是远程/VPN/生产**，**不要默认当本地**！
  - 用 `docker network inspect <net>` 看**网关/子网**佐证：若该 IP **不是** docker 网络网关、也不在容器子网内，它就是**宿主侧/外部**地址（VPN、另一台机、生产）。
  - 必要时**只读连一次**确认连上的是哪份数据（数据量/特征是否像生产）。
- 核实怎么查库（给第①/③步用）：优先用**后端自己的连接/ORM**查（`docker exec <backend> <python> ...` 走它的 SessionLocal），避免连接配置漂移、也保证查的是后端实连的那个库。
- **必须让用户确认**：这是本地副本还是远程/生产？要不要切到本地副本再测？（连生产 = 造数/删数风险，须明确授权或先切本地。）**无交互/subagent 场景** `AskUserQuestion` 不可用时按 SKILL.md §2"无交互降级协议"处理：上游书面授权可代替问答(注明出处)、未覆盖项标 🟡 入待问清单并**设为进入写库步骤的前置闸门**——不卡死、也不假装已确认。
- **★SUT 部署在远程机、本机只有文档/截图时**：上面"从本地配置读 DB 指向"的路径失效——此时 DB 指向、连接方式只能靠文档声称/向用户问，**远程库尤其可能是生产**，务必标为最高风险危险项、写库前必须用户显式确认；本机不可达时(ping/curl 超时)如实记"远程不可达、形态/DB 指向待用户提供"，不臆断当本地。

## 4. 需求 / 产品文档
- 找：`docs/`、`文档/`、`*.md`、需求/产品/测试记录单 Word/PDF；用户丢的文档。
- 记录：路径 + 格式 + 大致章节。多份就都列。
- **无文档**：显式记"无"（面4 文档符合性不启用、由实测反推，不影响其他面、不判未完成）。

## 5. 前后端源码位置
- 找：项目根的前端/后端目录（`*-frontend`/`*-backend`、`src/`、`app/`）；compose 的 `build.context`。
- 记录：前端根、后端根、关键入口（路由表、API 路由、数据模型）。代码是判定基准（代码 > API > UI > 文档）。

## 6. 环境约束
- **VPN / 内网依赖**：DB/外部服务/内部依赖（缓存 Redis、消息队列、对象存储等）是否在内网（需 VPN 才通）——结合第 3 项的 host 判断；逐个内部依赖看 host 是否为内网/外部地址。
- **写库授权范围**：能不能造数据、能不能删、破坏性端点能不能碰；默认**只读优先**，写需用户明确授权且最小范围。
- **物理测不了的（按类对号入座，别只列媒体）**：先识别清单，第③步据此标"阻塞"不空转。任意项目按下面 5 类归类，原媒体项只是各类下的**示例之一**：
  1. **依赖外部第三方服务**（无 key / 无额度 / 不可达）：支付沙箱与生产网关、短信/邮件网关、第三方 OAuth/登录、外部 AI/LLM、语音转写 ASR、地图/风控等外部 API。
  2. **依赖真实硬件/设备**：摄像头、麦克风、打印机、扫码枪、读卡器、串口/IoT 设备等。
  3. **需特殊环境**：真机多浏览器内核、性能压测/高并发吞吐、注入式弱网/故障注入工具、特定 OS/驱动。
  4. **破坏性操作**：全表清空、不可逆迁移、删生产数据等（默认不碰，须授权且最小范围）。
  5. **前置数据不可构造**：依赖特定 schema/外部系统状态/历史数据，本地无法造出对应前置。
- 找：compose/`.env` 里的外部服务配置——通用枚举：**第三方 API key、对象存储、消息队列、SMTP/短信网关、支付/OAuth 凭据、外部 AI/LLM key** 等；以及文档里的性能指标与环境说明。
- **🟡电商/交易类危险项（若项目含支付/订单/库存等）**：① 支付/退款走的是**生产网关还是 sandbox**；② 测试是否会**产生真实资金流**（动真金须明确禁止/隔离）；③ 支付/异步**回调能否安全重放**（幂等键、签名校验）；④ 订单/库存能否**隔离造数**（独立测试账号/商品，不污染真实交易）。
- **🟡CRUD 后台旁注（若项目含管理后台/配置项）**：① 能否改**角色/权限配置**——会**影响其他在线用户**的可见性/可用性，须谨慎；② 删除是**软删还是硬删**（硬删不可逆）；③ 对**真实业务记录**操作，建议优先 QA 造数，或测前快照、测后还原。

## 7. 实时通道与协同前置（🟡条件触发 · 仅实时/协作类项目；纯请求-响应项目整项跳过、不计缺失）
仅当项目含 WebSocket/SSE 实时推送、在线协同、presence/在线状态、实时通知等形态时启用；否则一句"本项目无实时通道"整项豁免，不影响其他字段、不判未完成。
- **WS/SSE 端点与鉴权**：找 `ws://`/`wss://`/`socket.io`/`EventSource`/SSE 路由、握手鉴权方式（token/cookie/握手参数）；记录端点 + 鉴权方式。
- **实时基础设施**：是否依赖 pub-sub/消息 broker（Redis pub-sub、消息队列等）、presence/在线状态存储——这些常在内网（结合第 6 项 VPN 判断）。
- **≥2 个可同时在线账号**：测协同/presence/广播**必须**至少 2 个能同时登录的账号（最好覆盖不同角色）。**明确告知用户：这是测该形态的硬前置**——没有第二个在线会话，协同/presence/广播类用例无法真测。
- 找：compose/`.env` 的 broker/Redis 配置、前端 WS/SSE 客户端代码、文档里的实时/协同/通知章节。

## 轻量核实 vs 深入探索的边界
- **第0步只做轻量核实**：确认"信息是真的、准的"——URL 可达、账号能登、DB 连哪、文档/源码在不在。
- **深入探索（逐页跑、API 全景普查、闭环图谱、边界越权实测）属第①步**，别在第0步做，免得越界、拖长。
