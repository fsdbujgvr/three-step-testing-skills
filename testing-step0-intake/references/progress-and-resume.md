# 进度扫描 · 续跑点定位 · 《进度总览》模板

> 配合 SKILL.md §3~§4。总管模式用它判断"跑到哪、从哪继续"，并维护状态记忆。**判定按完成信号/闸门真实数字，不按"文件夹是否存在"。**

## 一、各步完成信号（怎么判 未开始 / 进行中 / 已完成）
扫 `测试产出/`（或《进度总览》记录的真实根）。**根/子目录用中文默认名 ∪ 英文别名双名匹配**（各步保留了英文别名选项，别只认中文名扫空）：`测试产出|test-output`、`第一步-全面了解项目|project-understanding`、`第二步-编写测试用例|test-cases`、`第三步-执行测试|test-execution`；**优先读《进度总览》"产出根目录"字段按登记扫，未登记才回退默认双名匹配**：

| 步 | 关键产出（在则"至少进行中"） | 判"已完成"的信号（要核实数字/闸门，不只看文件在） |
|---|---|---|
| ⓪ 接案 | `第0步-接案与进度/项目信息确认卡.md` | 6 字段全 ✅、危险三项已确认、账号已核实 |
| ① 理解 | `第一步-全面了解项目/` 项目理解文档 + 证据 | step1 完成度闸门：API N/N、**前端源码八类 X/X(不得以页面 M/M 替代)**、页面 M/M、状态机 S/S、闭环图谱 100%、数据基线已建、可复现抽检——**读文档里的分子/分母确认达标**（漏看"前端源码八类"易把只读了页面就当①完成、误放行进②）。**★旧版产出兼容：若既有①产出由旧版 step1 生成、无"前端源码八类"这一口径，不机械判未完成——接受等效证据(如带 file:line 的全页/全组件穷举基线)判达标、并在台账标注"口径差异(旧版无八类、以X等效)"；只有连等效证据都缺才判①未过、回补** |
| ② 写用例 | `第二步-编写测试用例/` 用例 + 覆盖矩阵 + 文档符合性核实 | 面×维度矩阵每格有用例或标不适用、交互方式覆盖率、回环覆盖、回链①分母、写后回读、文档逐条——**读覆盖矩阵/README 统计确认** |
| ③ 执行 | `第三步-执行测试/` 执行记录 + `执行进度跟踪.md` + 缺陷台账 | **读《执行进度跟踪表》**：各面/模块 真测/源码佐证/阻塞/截图覆盖；step3 常**部分完成**——按面/模块/TC 看哪些 done、哪些 pending；**另核两专项是否齐：《_第②步漏项反哺清单》《_无法测试项说明/汇总.md》(条数闭合，见 step3 §11)** |

- **"进行中"判据**：关键产出存在、但闸门/进度数字未达标（如 step3 真测率 60%、还有面/模块没跑；step2 矩阵有空格）。
- **冲突时以实测为准**：《进度总览》记的进度若与实际产出不符，**以扫到的真实产出为准**并纠正台账（台账可能是上次中断前没来得及更新）。
- **★按形态把 N/A 信号剔出分母（纯 API / 无前端形态）**：先读确认卡/进度总览登记的**部署形态**。若为**纯后端 API / 微服务（无前端）**，则上表中**前端专属信号**——`页面 M/M`、`前端源码八类 X/X`、step③ 的 `UI 截图覆盖`、面1 UI 闭环——**整列视为 N/A，不计入完成度分母**，改以 `契约/OpenAPI 端点 N/N`、`资源关系图`、`错误码全集`、面2/3 接口与数据为分母。**切勿因为纯后端项目没有页面/截图就把它判成"未完成"。** 含前端的项目这些信号照常计入；形态未知先确认形态再判。

## 二、续跑点怎么精确定位（重点是 step3 的部分完成）
- **第③步**：读 `执行进度跟踪.md` 的逐模块表——列出"已真跑/已截图"的面·模块·TC vs "未跑/待补"的；续跑点 = 第一个未完成的面/模块，给出"从 面X-模块Y 的 TC-zzz 继续"。注意区分：真测 / 源码佐证(需补真跑) / 阻塞(测不了,进专项) / 待补；以及 UI 截图覆盖缺口、《_第②步漏项反哺清单》《_无法测试项说明》是否齐。
  - **★跟踪表缺失时的回退定位（真中断常态）**：执行**中途中断**时 `执行进度跟踪.md` 可能根本没生成(收尾才写)。此时**别因"没有跟踪表"就判第③步未开始或扫空**——回退用三样反推中断点：① **执行记录的逐TC汇总表**(已写到第几条)；② **`_scripts/state.json` 等链路变量**(造数链跑到哪一步、已建实体ID);③ **`证据/<TC>.log` 的完整性与文件 mtime**(最后一条有收尾标记的 TC 即断点)。把反推结论写进台账并标注"反推、无第一手进度记录"。
  - **★第③步中断必做：DB 残留只读核查 + 处置决策**（续跑判定的关键一环）。中断常留下未清理的 QA 造数。**只读**核查：按 QA 前缀(配 ASCII 子串/精确 id，见 step3 pitfalls 防中文 LIKE 假阴)逐表扫残留 + 关键表与基线对账，判断残留 = **活体前置数据(可直接续跑、省去重建)** 还是 **需兜底清理**。给出处置策略写进台账(A 保留续跑 / B 放弃则按子→父精确 id 清理回基线)；**总管只判定、不亲自清理或写库**。
- **第②步**：读覆盖矩阵找空格（哪些面/维度还没用例）、哪些文档源还没做符合性核实。
- **第①步**：读完成度自检看哪条闸门未达（如某些 API 三态没测、某状态机没画）。
- **跨步**：若①未过闸门就别催着进②；②没覆盖完就别进③——**总管要挡住"地基没打好就往上走"**。
- 把定位结果写成一句可执行的话（专名换成本项目实际值）："下一步 = 第③步，从 `面1-UI/<某模块>` 的 `TC-UI-<模块>-NNN` 继续；另 面7 持久化 还剩 N 条待补真跑。"

## 三、《进度总览.md》模板（落 `测试产出/第0步-接案与进度/`）
```markdown
# 测试进度总览（总管状态记忆 · 最后更新 <时间>）

## 0. 项目信息确认卡（接案结论）
- 入口URL/部署：____
- 角色账号：____（各角色列全）
- ★数据库指向：____（本地副本/远程/生产 + 连接方式 + 是否已切本地/是否需授权）
- 需求文档：____（位置/格式，或"无"）
- 源码位置：前____ / 后____（无源码/仅镜像部署也注明——决定第②/③步用容器内源码 file:line 或文档§+API+DB 降级取证）
- 环境约束：VPN? ____ / 写库授权：____ / 物理测不了项：____
- 危险三项确认：DB指向=____、写库授权=____、测不了项=____（无交互时标"据书面授权"或🟡待确认+设为写库前置）
- 产出根目录：测试产出/（若非默认，记真实根）；**跨根/沙盒映射**：____（若本轮用例集/产出散在别的根或沙盒目录，逐条登记真实路径，否则总管下次扫不到）
- ★执行中断残留状态：____（无 / 有：列残留实体精确ID+所在表+处置策略A保留续跑或B待清理）

## 1. 各步进度
| 步 | 状态 | 完成度 | 关键产出路径 | 闸门是否过 |
|---|---|---|---|---|
| ⓪接案 | 已完成/进行中/未开始 | __% | 第0步-接案与进度/ | __ |
| ①理解 | … | __% | 第一步-全面了解项目/ | API_/_·页面_/_·状态机_/_ |
| ②写用例 | … | __% | 第二步-编写测试用例/ | 矩阵填满? 文档逐条? |
| ③执行 | … | __% | 第三步-执行测试/ | 真测率__%·截图覆盖__% |

## 2. ★当前续跑点（新对话从这里接着干）
- 下一步 = 第_步，具体：从 面X-模块Y 的 TC-zzz 继续；还剩 ____ 待办。

## 3. 环境与续跑须知
- 当前 DB 连哪、是否需先起本地库/VPN、override 文件等环境状态。
- 已知风险 / 待确认 / 上次中断原因。
```

## 四、维护纪律（写后回读）
- **每完成一步、或预感要中断前**：更新本台账→回读确认落盘。这是新对话能续上的唯一依据。
- 台账是**活文档**：进度变就改，别写一次就废；与实际产出冲突时以产出为准并纠正。
