# 全面了解项目（测试三步走 · 第一步）

## 唯一目标
产出一份**完全的《项目理解文档》**，作为第②步写测试用例的**最牢固地基**：详细到一个不懂该系统的人，仅凭它就能写出覆盖 功能 / 边界 / 状态机 / 权限与越权 / 跨模块端到端 / 文档符合性 的最完整用例。**未过第 5 节「完成度闸门」前，不算完成。** 这一步只做理解、不写用例本身。

> **投入策略：全面性 > 效率，不设投入/ token 上限**。闸门是**地板不是天花板**——过闸门后继续往穷尽里做：每个端点的每个字段/错误码/副作用、每个输入框的每条校验、每个状态的每条流转都要落到；能穷举就穷举、**必须大量扇出子代理(subagent)并行勘察——派出总数 = 业务模块数 N + 后端代码线 1 + 前端代码线 1 + 文档线 1，即 N+3 个起步（"逐模块"是乘以 N、不是一条线只派一个；纯 API 等异形态按 §5 闸门"实际勘察单元数"折算；按 §4《扇出计划表》照派、别自己判断要不要）**（三家终端 Claude Code/OpenCode/Codex 均称 subagent；支持 workflow/编排的环境用其更强并行），**宁冗长勿遗漏**。目标是"最全面"，不是"最快/最省"。

> **前置输入（优先用第0步产出·若有）**：若本项目已跑过第0步 `testing-step0-intake`，**先读 `测试产出/第0步-接案与进度/项目信息确认卡.md` 与《进度总览.md》**，直接取用其中已确认的 入口URL / 各角色账号 / ★DB指向(本地vs生产) / 写库授权范围 / 物理测不了项 / 源码与文档位置——**存在即直接用、不重复发现、不重复发问**。**没有第0步产出时，本步照常自行收集（见 §4），独立可用、不强依赖第0步**——此条只增不减、不改变本步既有能力。

## 0. 先花半分钟判断形态（form factor，决定哪些面/闸门适用 + 要不要读附录）
被测系统的"**形态**"决定哪些主线、哪些面、哪些完成度闸门适用——**这是贯穿 step0/1/2/2.1/3 的统一概念**，本步识别出的形态结论要经《项目信息确认卡》/《进度总览》向下游传递，下游各步据此开关对应闸门。常见形态：
- **① 标准 Web 应用（默认）**：有前端 + 源码 + 可登录账号——绝大多数情况，**直接走第 1 节三主线**。
- **② 纯 API / 无前端**：仅 REST/gRPC + 契约，凭据为 token/API-key/mTLS——**前端渲染主线 N/A**，改以"真实请求-响应 + schema/契约校验 + 代码 file:line"为"眼见为实"，查 `references/non-standard-forms.md` 的"纯 API"节。
- **③ 薄 UI 后端 / 交易中台**：价值在后端业务规则（公式/事务/幂等/状态机），UI 只是薄壳——前端闸门按实际 UI 体量缩放，**另加后端侧硬闸门**（见下）。
- **④ 实时协作**：标准应用叠加 WebSocket/SSE 实时通道——**三主线照常全走**，另就实时这一面挂载附录的实时双工取证（叠加，不是二选一）。
- **⑤ 含富交互（canvas / 媒体 / 可视化）**：在上述某形态基础上含拖拽/悬停/滑块/右键/滚轮/canvas 绘制/音视频——**控件确实支持该交互时即必测**（命中即必测），无则跳过。
- **⑥ 对话式 / LLM 应用（聊天机器人 / AI 助手 / Copilot / RAG）**：核心是"输入 →（流式）生成"、输出**非确定**——三主线照常，**额外**挂载"非确定输出断言范式（断属性/结构/不变量，而非精确文本）+ AI 特有维度（配置变更真生效/流式生命周期/上下文记忆/安全护栏/多模态/历史持久化）"，详见 `references/non-standard-forms.md` 对话式节。命中才加、不命中零影响（纯后端 LLM API 无聊天 UI 时并入形态②）。

> **统一开关（哪些只在"有面向用户的前端"时适用）**：面1 UI 主线、"必须浏览器截图、不可用源码佐证替代"、交互方式覆盖率、browser_snapshot 可操作元素穷举、角色级"后端 200 不算通过/必须前端可达"——**这些仅在形态①③④⑤（有前端）时适用**；**形态②（纯 API/无前端）整组标 N/A**，改以"真实请求-响应 + schema/契约校验 + 代码 file:line"为权威。形态判定后，缺失的主线/面**显式标 N/A + 原因，不计未完成**。

判定为标准 Web 应用（形态①）后就别再纠结附录，直接往下。非①形态先读 `references/non-standard-forms.md` 挂载对应降级/叠加取证，再回来走能走的主线。

> **两类常见混合/薄壳形态的额外指引（叠加而非替换）**：
> - **混合形态：标准 Web 应用 + 叠加实时通道（形态④）** → 三主线照常全走，**另就 WS/实时这一条额外面**挂载 `references/non-standard-forms.md` 的"实时双工"取证（消息/事件 schema、广播范围、协同冲突解决、房间级鉴权、重连补发、presence）——**叠加而非二选一**，实时面是新增的一面，不抵消任何原有主线。
> - **以后端业务规则为主、UI 为薄壳的交易/中台类（形态③）** → 前端八类/页面 M/M 闸门**按实际 UI 体量缩放**（UI 小则分母小，不硬凑）；**新增后端侧硬闸门**：核心业务公式 / 事务 / 幂等键 / 状态机迁移**逐条落 file:line 并标"全/缺"**，与前端闸门对等核；脚手架/框架样板代码可**批量标记"已览" + 抽样 file:line**，不必逐行。

> 这步必做：否则容易对着异形态硬套三主线、或把够不到的面默认成"不存在"、或反过来把没有的特性当强制项空转。

## 1. 核心方法：三条主线交叉印证 + 文档对照，把项目"全方面"摸透
理解一个项目，靠下面**三条主线**，缺一条都不算全面——它们分别回答"应该怎样 / 实际怎样 / 看到怎样"：
> **限定（以标准 Web 应用为默认）**：以下三主线以**形态①标准 Web 应用**为默认；**非标准形态（含纯 API/无前端、薄 UI 后端、实时协作）按 §0 形态分流**——前端渲染等缺失主线**显式标 N/A + 原因，不计未完成**，纯 API 形态以"真实请求-响应 + schema/契约校验 + 代码 file:line"替代前端渲染主线。§0 的形态分流结论必须回灌到这里与 §7 反模式。

### 主线一 · 代码（实现真相："应该怎样"）
通读前后端源码，产出**带 file:line 的穷举清单**。**前端要读"全部源码"，不止页面**——含运行期入口文件（如 main.tsx 的 DOM 文本替换层）/ 路由表 / 状态管理 / api 层 / hooks / utils / types / 本地数据；后端含模型 / 全部路由 / 鉴权依赖 / crud / services / 配置与启动 / 部署编排。逐项：
- 数据模型：每个实体的字段 / 类型 / 可空 / 默认 / 唯一约束 / 外键 / 枚举。
- **全部 API**：每个端点的 方法 / 鉴权 / **对其他实体的副作用** / **错误码与中文文案原文**。
- 前端每个页面/组件：表单字段 / 校验规则(正则·必填·范围) / 枚举选项 / 硬编码常量 / 隐藏限制(maxLength·重试·超时·分页·轮询) / 状态映射与文案 / 运行期文本替换映射。
- 数据编排：谁创建谁、创建序列、跨页 ID 如何传、缓存何时失效。
> **铁律：一切以代码为准**（代码 > API实测 > UI/渲染 > 文档）。代码与浏览器实测是事实，文档只是"声称"。
> 唯一例外：当**特性开关未开 / 端点返回 mock / 线上数据已与代码分叉**时，该处以**可复现实测**覆盖"代码字面结论"（代码说"会做X"但线上没开关→以实测"未发生X"为准；见 pitfalls 14/15/20）。即"以代码为准"指逻辑真相，"以实测为准"指线上现状，二者矛盾时**两者都记、并标明分歧与触发条件**。

### 主线二 · 浏览器（运行真相："实际怎样"）
把站点**逐页、逐按钮、逐输入框真的跑一遍**，用截图 + 网络请求作证：
- 实操每个交互（点按钮、填表单、开弹窗、翻页、搜索筛选），看真实结果与网络调用。
- 看**运行期才有的东西**（通用项前置、必看）：空态、加载态、错误态、真实数据规模、第三方回调(AI/支付/地图)、文件导出；**（若有）实时流/音视频**。
- 用 API 脚本化跑**权限矩阵**(无token/各权级)与**边界**(空·超长·特殊字符·最值±1·重复)——比纯点击快且精确。

### 主线三 · 前端渲染（所见真相："用户看到怎样"）
看 UI **实际渲染出什么**，做"代码说的"与"用户看到的"对账——**核心是对账"代码字符串" vs "渲染所见"**（文案 / 枚举 / 硬编码常量 / 未映射兜底）：
- DOM 提取渲染后的精确文案 / 枚举 / 字段值（比读源码字符串更可信，且不乱码）。
- 抓"渲染 ≠ 数据"的坑：**硬编码常量被当数据**（型号/时间/版本号固定不变）、**未映射枚举兜底**（显示原始英文/空白）、**运行期文本被改写**（源码词≠显示词，按显示文案断言、用路由/属性定位）。
- **（条件触发·若有富交互/媒体）** 若项目含视频/音频/canvas/实时流 → 再验它们是否**真工作**（如读 video.readyState 而非只看截图、canvas 是否真画出、流是否真连）；无此类则跳过本条、不计漏测。

### 再叠加 · 文档对照（声称真相："声称怎样"）
通读产品/需求文档，**逐条与上面三主线实测对照**。文档常暴露架构声明、性能指标、独有功能，且**常与实现严重不符**——这些冲突正是高价值发现。**一切冲突以可复现实测/代码为准**；无文档则显式记"无"、由实测反推（不得因此判未完成）。
> **冲突必须单独成文件**：把"代码/浏览器实测 vs 文档"的全部冲突汇集成一个专门的《产品文档对照与冲突》文件（逐条：文档声称 / 实测 / 判定，含技术栈失真、性能指标、独有功能、声称未实现），**不要散落在各模块文档里**。这是让下游一眼看清"哪些文档不可信"的关键产出。

> 三主线 + 文档的关系：**代码=应该怎样，浏览器+前端渲染=实际怎样，文档=声称怎样**。三者交叉、矛盾处以实测为准——这才叫"全方面"。证据分级：代码 > API实测 > UI/渲染 > 文档。
> 逐项检查清单见 `references/checklist.md`。

## 2. 广度优先：把所有模块连成"数据流闭环"（端到端用例的地基）
**不要只孤立记录每个模块。** 第①步的灵魂是把模块连成端到端闭环，否则第②步写不出"在 A 操作 → 去 B/统计验证记录是否生效"的跨模块用例。产出《业务闭环与数据流图谱》：
- **枢纽实体**与实体关系；**归属如何赋值与传播**（谁建/谁负责/谁可见——决定权限·派单·统计，常有"指定者≠操作者≠作者"之分）；**状态同步与级联删除**规则。
- **典型端到端 happy path**（每步操作 + ID 交接 + 副作用）；**跨模块"建X验Y"验证点清单**（直接喂给第②步）；各**汇总/统计指标的真实数据源与公式**。
- 用一条**真实数据的只读回溯**佐证（挑一条完整链路，把关联实体逐一拉出，核对 ID/owner/status 贯穿）。

## 3. 安全红线：探索期默认只读，不写部署库
对部署系统的**任何写**（建/改/删账号或业务数据）会触发护栏、污染共享状态——**默认禁止**。证闭环用真实已有数据的只读回溯。确需写才能证的（如匿名写漏洞）**仅在显式授权下**用"创建→立即删除"可逆往返 + 自清理 + 唯一前缀；无授权则只记"疑似·未实证·留授权阶段"。**若第0步已确认"写库授权范围 / DB 本地或生产 / 物理测不了项"，以其危险三项确认为准**（连生产库尤其谨慎）；未授权一律只读——与本步默认只读一致。

## 4. 执行顺序：先骨架后填充（防中断丢结论、为覆盖率立分母）
1. 锁定唯一被测入口、排除同名残留；**先落盘空骨架**（API清单/页面清单/实体清单/状态机清单，含"总数"占位）。**被测入口 URL 与各权级账号凭据：优先取第0步《项目信息确认卡》(若有，已确认即直接用)；其次用户直接给；再次依次从 compose/env/README/前端配置/种子或导入 SQL 自取并记入《环境部署与数据现状》；确实无处可得才向用户索取**（与确认卡冲突时以可复现实测为准并回写《进度总览》）——不要因缺这些在执行中途反复发问（那会破坏"一次性无提醒"）。
2. 走三主线（代码→浏览器→前端渲染）回填骨架；通读文档并对照。**代码线与文档线只读、分两波扇出子代理(subagent)——共 N+3 个起步**（纯 API/无前端形态前端线 N/A、按 §5 勘察单元折算）。**第①波·勘察建图（先派 3 个）：后端代码线 1 + 前端代码线 1 + 文档线 1，各自读完本侧全部代码/文档**——产出带 file:line 的全局穷举基线（数据模型全集 / API全集清单 / 路由表 / 页面清单 / 鉴权 / 基础设施 + 文档对照），**并圈定业务模块边界与清单**（前端导航菜单 + 路由分组 + 后端模块目录 + 需求文档章节 多源对照，对不齐处以"用户视角的业务模块"为准）。**规划：主循环据①波产出落盘《扇出计划表》**（放第①步产出目录）——得出模块清单 N（**基于①波已读透的代码、不是浮于表面的踩点**；表头登记四来源各自计数 导航=a/路由=b/目录=c/文档=d + 最终 N + 一句取舍理由，把 N 取粗必须留痕、§5.5 审计会核对 N 与各来源计数）→ 逐行登记"序号 | 负责模块(含其页面/路由/表/接口范围) | 预期落盘文件"。**第②波·逐模块深挖（照表派 N 个）：每个业务模块各派一个 ag**，深挖该模块 页面/路由/services/crud/hooks 的每字段/校验规则/错误码原文/副作用/状态流转——**严禁一个 ag 囫囵多个模块**（那样每块必然只剩一两段）、模块多可分批；①波代码线 ag 已建的全局共享层（utils/公共组件/types/鉴权中间件）模块 ag 引用即可、不重复产出（切分主轴=共享层 vs 模块层）。**★两波都必做、不许只做①波就跳过第②波**（只有全局骨架没逐模块深度 = 写不出最完整用例）；**别自己判断"要不要派/够不够"，照表派**，鼓励复杂模块多拆几个、加专项 ag，上不封顶。每子代理产出落盘——能写盘则自落盘到《扇出计划表》指定路径、否则返回纯 markdown 由主循环拆分落盘（各终端均称 subagent，支持 workflow 的环境用其更强并行）；**环境不支持子代理则单线程但分波、分模块逐个深做、每块写完即落盘防赶工**（判"不支持"须在《进度总览》记一句判定依据，如试派失败的报错原文/工具列表确无子代理类工具，无《进度总览》时记入本步《完整性自检与覆盖矩阵》）。**浏览器线单会话、留主循环串行**——起步：用 Playwright MCP 起单一会话 → resize 1920×1080 → 用上面凭据登录；**切身份 = 清 localStorage + 重新登录，或注入匹配用户档案结构的 token**（见 pitfalls 9/25/26）；读文本/枚举用 DOM evaluate，验证媒体用 video.readyState。总览/闭环/对照/自检/就绪等集成文档主循环亲写以保一致。
   - **子代理任务书（防"派而不深"——子代理看不到本 SKILL，全靠任务书带要求）**：给每个子代理的任务 prompt 必须写明：① **边界**——**①波代码线 ag = 本侧全部（后端 ag 全部后端源码 / 前端 ag 全部前端源码 / 文档线 全部文档）；②波模块 ag = 单个业务模块及其页面/路由/表/接口范围（取自《扇出计划表》）**，附主循环踩点已知的起点目录/关键文件提示；"别碰邻块"指**别替邻块产出条目**：共享 utils/公共组件/types 由①波代码线子代理统一产出，②波模块子代理为兑现任务书③(产出)**可顺藤阅读共享文件**、但不为其单独产出，只在条目处标"引用共享文件X(file:line)"；② **方法**——先用 glob+grep（路由注册、import 追踪）建立块内全部源文件清单、再逐文件过，不许"抽几个代表文件看看"；并写明**本任务为纯静态阅读：不启动服务、不发请求、不写任何数据**（§3 红线子代理看不到，必须带上）；③ **产出**——按 §1 主线一逐项带 file:line，落盘到任务书指定文件（取自《扇出计划表》"预期落盘文件"列，写**绝对路径**）；**§1 主线一的逐项清单须原文展开抄进任务书**（含数据模型字段细粒度、数据编排、状态映射与运行期文本替换），不许只写章节号或缩略括号；④ **投入策略原话带给子代理**——"全面性 > 效率、宁冗长勿遗漏、不设 token 上限"（不写这句，子代理会按省 token 的默认习惯干活）；⑤ **收尾自报《文件清点》**——"块内源文件共 X 个（文档线为文档文件）、已读 X/X，略过的逐个标原因"，**附建清单用的 glob/grep 命令原文 + 完整文件列表**（让分母可复查）；确因上下文物理写不下时方可分批落盘、如实报"已读至第X个、建议主循环续派余下"（分批=续派补全、不是缩量,别拿"超载"当少读借口）——**宁可报没读完，不许压深度凑完整**；正当略过仅限 纯样式/构建生成物/框架样板（样板按 §0 批量"已览"+抽样 file:line），**含业务逻辑的源文件不得以"与测试无关"为由略过**，可如实标"已读·无本块相关条目(纯样板/仅 re-export 等)"；块清单与实际源码不符（多/少/找不到）时照实勘察、在产出开头列《与任务书清单的差异》；任务书全文随该块产出落盘一份；返回值只报"落盘路径 + 条目计数 + 清点摘要"、正文以落盘为准（子代理不能自己写文件的终端按 pitfalls 24 返回纯 markdown、主循环拆分落盘）。《文件清点》与产出供 §5.5 审计抽核。
   - **凭据字段泛化**：上面所说"账号密码"泛指**各权级凭据**（账号密码 / token / API-key / 证书 mTLS）——按形态取实际凭据类型。
   - **（形态②纯 API/无前端分支）** 跳过浏览器会话 / resize / 用 localStorage 切身份这套；**认证改为"取/签发各角色 token 或 API-key/mTLS"**，**切身份 = 换 Authorization 头 / 换证书**而非清 localStorage 重登；前端渲染主线整条标 N/A，回填骨架时以"真实请求-响应 + schema/契约校验 + 代码 file:line"取证（见 `references/non-standard-forms.md` 纯 API 节）。
3. API 数据全景普查（各实体计数/状态/主数据/脏数据）建真实基线——**用 node(全局fetch)/curl 取数，别用易乱码的 PowerShell 或常被劫持的 python**；可在浏览器页面内用已登录态 fetch 取证。
4. 边界 / 权限 / 越权 实测；建闭环图谱 + 真实链路回溯；文档逐条对照成**专门的冲突文件**。
5. 过完成度闸门 → 整理冲突/待确认 → 收尾。
> 编码/工具/认知/编排陷阱见 `references/pitfalls.md`（中文乱码与取数工具选型、文案运行期改写、特性开关让代码结论失效、mock 端点≠真实数据、工具结果过大致中断、子代理扇出结果落盘、浏览器实例锁、截图落点、代码暴露≠入口可达等）。

- [ ] 三主线都走到：代码穷举清单 / 浏览器逐页取证 / 前端渲染对账，各有产出（缺前提时按 `references/non-standard-forms.md` 降级并标注）。
- [ ] **API 覆盖**：已枚举 N 条，逐条标[无token/各权级 三态已测] = N/N。

## 5.5 产出后对抗审计（派独立子代理复核，别自我宣布"完成"）
过完 §5 闸门 ≠ 真达标——**自评易美化**（实战中"自报 90%"经独立一审常缩水）。**必做：过闸前必派一个独立子代理(subagent)做对抗审计**（干净上下文、立场怀疑、默认不通过），逐项复核并**举反例**；**未做这轮独立审计 = 本步未完成、不许进第②步**：
- **空壳/赶工**：有没有"建了空目录却不填"（代码级基线 / 证据 / 冲突待确认）、"自检打钩却无对应文件落盘"（声称有截图却 0 张、声称有基线却空目录）；
- **深度**：抽 2~3 个模块详解，反推"凭它能否写出 3 条精确用例"——写不出 = 深度不足；
- **真实性**：抽 3 条 file:line / 计数 / 错误文案，回源码或重查核对真伪；并抽 1 个块的《文件清点》——从"已读"文件里任挑 1 处肉眼可见的校验/枚举/错误文案，核产出是否收录，漏收 = 该块回炉（标"已读·无本块相关条目"的核其原因真伪）；
- **结构**：文件编号有无重号、各业务模块是否真逐个独立成文；核《扇出计划表》——四来源计数与最终 N 的取舍是否留痕、计划行数(N+3 起步)与实际落盘文件是否对得上。
审计须**结构化输出**（逐项 达标/不达标 + 反例证据）；**不达标项必须回填并重做，禁止"记一笔就过"**。
> 诚实边界：此审计由主进程派出、结论也由主进程采纳——**它提升质量但不替代你的诚实**，最终是否认账仍取决于执行方（这是纯 prompt 的天花板）。环境不支持子代理时，改由主循环对照本清单亲自逐项复核（同样举反例、不达标重做）。

## 6. 产出：一套"能直接写用例"的文档
**产出落在当前被测项目的工作目录下**（如 `./测试产出/` 或 `./project-understanding/`，跟随当前项目，不要写死任何绝对路径）。按三步走组织，第①步产出放在「第一步-全面了解项目/」下，关键文件：
> **隶属统一根（供第0步总管扫描续跑）**：本步产出隶属测试三步走统一根 `测试产出/`（见 step0 §6），第 5 节完成度闸门(API N/N·**前端源码八类 X/X**·页面 M/M·状态机 S/S·闭环图谱)即第0步总管 `testing-step0-intake` 判定本步"完成/进行中"的信号。默认子目录 `第一步-全面了解项目/`。**仍可用别名根/子目录（如 `./project-understanding/`，原选项保留不变）**，但用别名时**必须在《进度总览.md》登记真实路径**，否则总管按默认名扫不到。
总览与架构 / 环境部署与数据现状 / 认证与权限体系 / 数据模型与校验速查 / API接口清单导读 / **各业务模块详解（★每个模块各一份独立文件、逐个详尽，不合并成单个"模块详解"）** / **业务闭环与数据流图谱** / **产品文档对照与冲突** / 完整性自检与覆盖矩阵 / **测试就绪补全（账号凭据·可复现造数payload·统计口径公式·schema精确边界）** / 代码级穷举基线 / 证据(按模块) / 冲突清单 / 待确认清单。
> **★产出方式对齐 skill 出现前那种"充分展开"**：上面"各业务模块详解"指**逐模块各一份**；"代码级穷举基线（A数据模型/B API与安全/C前端页面/D契约）/ 证据 / 冲突清单 / 待确认清单"同样**各自独立落盘、不为省事合并或建空目录占位**。判据不是文件数，而是"内容完整到没见过的人能据此写最完整用例"——而单个文件装不下逐模块深度+逐条证据，所以充分产出自然展开成 20+ 份而非压成 10 份。这是**明确产出要求**（非机械数字闸门）：内容够充分，结构自然就回来了。
> 完整目录结构与各文件用途见 `references/output-structure.md`。这四块——闭环图谱 / 文档对照 / 完整性自检 / 测试就绪补全——是让"新模型立刻能写用例"的关键拼图，务必齐全。

## 7. 反模式（务必避免）
> **限定（呼应 §0/§1 形态分流）**：以下反模式以**形态①标准 Web 应用**为默认。**非标准形态（含纯 API/无前端、薄 UI 后端、实时协作）按 §0 分流**：缺失主线/面**显式标 N/A + 原因即可，不算违反、不计未完成**——例如纯 API 项目没有前端渲染主线、没有页面 M/M，不构成"只走一两条"或"用页面冒充全覆盖"的反模式。
❌ 三主线只走一两条（只读代码不跑浏览器、或只截图不看渲染对账）→ 理解不全面。
❌ 前端**只读了页面组件**就当"代码线已完成"，漏掉 hooks/utils/api层/状态管理/运行期入口 → 用页面 M/M 冒充前端全覆盖（必须过第5节"前端源码覆盖 八类 X/X"）。
❌ 模块各记各、不连闭环 → 写不出端到端用例。
❌ 跳过/延后读文档、或把文档结论当事实 → 漏架构失真与独有功能。
❌ 探索期往部署库造数据"跑一遍" → 触发护栏/污染库。
❌ 用自答"是/否"过闸门、用"每个/全部"却无分母 → 覆盖率空转、自以为完成。
❌ **把十几个业务模块挤进单个"各业务模块详解"文件**（每模块只剩一两段）→ 撑不起"没见过的人能写最完整用例"，必须逐模块单独成文、各自详尽。
❌ **把 证据 / 冲突与待确认 / 代码级勘察基线 合并省略、乃至只建空目录却不填内容、自检却打钩"已完成/有截图"** → 产出看似齐全实则空壳；自检里凡声称"有截图/有基线/已浏览器"，对应文件/截图必须真落盘，否则属虚报（诚实纪律）。

> 被测对象不是带前端的标准 Web 应用（纯 API/无源码/无账号/Electron/移动端/WebSocket/Canvas/二进制/无界面）时，查 `references/non-standard-forms.md` 挂载降级取证——目标不变，仍要产出能写完整用例的理解。
