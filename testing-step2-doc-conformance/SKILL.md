# 文档逐条(原子级)符合性核对（测试三步走 · 第②步 · 文档符合性专项 / 2.1）

## 唯一目标
把需求/技术/规格文档拆到**原子级**——每一个被声称的最小单元(功能点 / 权限项 / 操作 / 字段 / 列 / 数值规格 / 顿号并列子项)，**逐一对照真实代码 + 真实网页**核定 **有 / 无 / 部分 / 待确认**，并给出**可复核证据**。产出《文档逐条(原子级)符合性对照表》。
> **命脉 = 准确性**：每一条判定必须可被复核。**判"有"必须给代码 file:line 或 网页实操证据；判"无"必须给"找过了"的证据（grep 无果 + 网页无入口）；拿不准只能标"待确认"，绝不臆断。** 这是本 skill 与"凭印象对文档"的根本区别。

## 0. 为什么独立成 skill（与其它测试面的区别）
- **判定基准(oracle)不同**：其它面测"系统行为对不对/有没有 bug"，基准是系统自身；本面测"实现是否=文档**声称**"，基准是**外部文档**——是"实测 vs 声称"的对账。
- **粒度要求不同**：要逐词/逐并列项准确，不能句子层打包。
- 它是**官方《测试记录单/报告》、评审高亮/批注的唯一对接口**。
> 前置：需要第①步《项目理解文档》(代码级穷举基线 A-D / 页面 / API / 数据模型) 作"代码侧"证据来源；没有则先做 `testing-step1-understand-project`。

### 0.0 · 形态分流（form factor）—— 决定"网页实测"是否适用（全流程统一概念）
> 被测系统的**形态**决定哪些证据形态/闸门适用，本概念在 step0/step1/step2/step2.1/step3 **统一引用**，由 step0/step1 识别并经《项目信息确认卡》/《进度总览》向下游传递。本步从中读取形态，决定"步骤 C 网页实测"是否适用。
常见形态：
- ① **标准 Web 应用**（默认，有前端 + 源码 + 账号）。
- ② **纯 API / 无前端**（仅 REST/gRPC + 契约，凭据为 token/API-key/mTLS）。
- ③ **薄 UI 后端 / 交易中台**（价值在后端业务规则，UI 是薄壳）。
- ④ **实时协作**（标准应用叠加 WebSocket/SSE 实时通道）。
- ⑤ **含富交互**（canvas/媒体/可视化）。
> **统一开关**：本 skill 中"步骤 C 网页实测确认"、§5"面4 必须浏览器截图、不可源码佐证替代"——**仅在被测系统有面向用户的前端时适用**。**纯 API/无前端形态（②，及③的纯后端规格部分）下，把这组要求整组标 N/A**，改以「**真实请求-响应报文 + schema/契约校验输出 + 代码 file:line**」为"眼见为实"。形态在 step0/step1 识别并向下游传递；本步缺上游信息时按默认①处理，但凡发现"应有 UI 却无"的声称仍判"无"。

> **执行编排（推荐隔离）**：本任务最吃精度，**最好作为独立子 agent 在干净上下文里专做**——单一使命=准确，避免被其它测试活动的上下文带偏成"句子层打包/凭印象判有"。文档大时**按章节扇出多个子代理(subagent)**（每个包一章/一组，各自拆原子+翻代码+点网页，返回该段对照表；三家终端均称 subagent，支持 workflow/编排的环境用其更强并行），主流程汇总成全表。浏览器是单一有状态会话——若多 agent 并行，**网页核验那步要串行或分时**（代码侧 grep 可并行）。**后台长 agent 防断连**：`run_in_background` 的核实/补漏 agent 可能在收尾(写文件+返回摘要)时 socket 断连、做了一堆活却文件没落盘全丢——prompt 里明确"**先把文件写完落盘(主交付物)、分批 Write、最后只返回极简摘要**"，并在 agent 完成后**用 `ls`/`wc -l` 确认文件真落盘**，没落盘就重跑（前台并行更可控）。

## 1. 核心方法：原子化四步（对每一句话都走一遍）
### 步骤 A0 · 先枚举全章节目录（防"静默跳过非主功能章"，否则分母虚高）
拆原子前，**先把源文档的章节目录整份列出来**（含 前言/免责声明/术语表/附录/性能/安全/可靠性/维护/兼容性/数据备份 等"非主功能"章；现代/API 化 SaaS 还常见 SLA/限流配额、兼容性矩阵、数据保留与删除策略、变更日志 等章——同样登记入表），建一张"章节清单 + 已核 X 章/共 Y 章"打勾表。
> **★ 血泪教训**：核实 agent 天然倾向只核"主功能章节"而**静默跳过**版本命名/术语表/附录/性能/安全/维护等章——而这些恰恰藏着**最高价值的失真**（术语表/附录里的技术栈声称、版本规则、安全级别、审计日志）。曾有一份核实"自评基本完整"，独立审计却查出整本只核了操作章、§前言/术语表/性能/安全/附录全漏。**任何章不得不出现**：哪怕判"非软件功能/资料类"也要**留痕登记**（标待确认+理由），而不是不写。逐章打勾后，分母才真实。

### 步骤 A · 逐句拆原子
把文档每一句拆成**最小可独立判定**的声称单元：
- **顿号/逗号分隔的并列项，每个算一个原子**。例：「商家具有商品管理、订单下载、订单查阅、查看订单详情、发送通知、库存管理权限」= 6 个原子，逐个判；「按下单时间、支付状态、收货地址筛选并导出 CSV/Excel」= 5 个原子。（媒体/画布类项目的并列拆分补充示例：「支持图片放大、缩小、平移、旋转」= 4 个原子。）
- **每个字段/列/状态**各一个原子。例：「查看账号列表、最近登录时间、联系方式、状态」→ "最近登录时间""联系方式" 各一原子。
- **每个数值/规格**各一个原子(≤100ms / 最多 6 张 / 并发 10 / ≤2 分钟)。
- **契约即规格（条件触发·若有 OpenAPI/Swagger/proto/gRPC IDL）**：把每个 path/operation、每个参数、每个 schema 属性、每个枚举值、每个 required 约束、每个错误码**各拆为一个原子**。对纯 API 形态（§0.0 ②），「**契约声明 ↔ 代码实现 ↔ 实测响应**」三方对账是**主线 oracle**，散文需求文档作为**补充 oracle**；标准 Web 形态下契约原子与散文需求原子并存、互为补充。
- 给每个原子**定位**：文档章节号 + 原文片段（如 `2.1.9.1 旋转`）；契约原子定位到 path+method+字段名/枚举值。
> 拆法细则与各类示例见 `references/atom-decomposition.md`。

### 步骤 B · 找代码实现
在第①步代码基线/源码中找该原子的实现点并记 **file:line**，或确认其不存在：
- 按语义找（**源码词 ≠ UI 词**，可能有 i18n key/运行期文本替换/模板变量；按显示词 grep 会落空——用源码原词/路由/字段名/i18n key）。
- 找不到时，记录"已 grep 哪些关键词、查了哪些模块"，作为"无"的依据，而非"没看到=无"。

### 步骤 C · 网页实测确认（按形态分流，见 §0.0）
去真实网页找该功能的入口/控件/行为：看得到且能用 = 实测支持；无入口/点了无效 = 实测无。媒体/画布/实时类读运行期状态（如视频 video.readyState、实时连接 WebSocket 状态、异步加载/骨架屏态等）。**写库类只观察不破坏（造数留第③步授权）。**
> **形态分支（条件触发）**：对**后端/无 UI 规则类原子**（幂等 / 事务原子性 / 金额公式 / 状态机校验 / 契约字段）——以 **API 真发请求** 或 **代码 file:line** 为实测证据，证据形态 = **请求-响应报文 + schema/契约校验输出**。**这类原子不得因"无网页入口"判"无"；只有"应有 UI 而 UI 缺失"才判"无"。** 纯 API 形态（§0.0 ②）下，面4 的"眼见为实"= 真实请求-响应报文，而非网页截图。

### 步骤 D · 判定 + 证据 + 冲突
每原子给 **有 / 无 / 部分 / 待确认** + 代码证据(file:line) + 网页证据 + 备注。
- **冲突**（代码有但网页无 / 文档有但实现无 / 两源不一致 / 与官方批注不一致）**单列**，**以可复现实测为准**。
- 例：文档"订单可退回(打回提交者整改)"，代码有 `…/return` 端点但评审批注"网页无该功能"——必须实际去网页点一遍定论，冲突进清单。

## 2. 准确性纪律（本 skill 的命脉，违反即作废）
- **判"有"= 必附证据**（代码 file:line 或 网页实操）；只凭文档/命名/想当然 → 不可判有。
- **判"无"= 附"找过了"的证据**（grep 关键词无果 + 网页无入口/无效）。
- **拿不准 = "待确认"**，绝不猜（猜"有"会放过缺陷，猜"无"会误报）。
- **术语 ≠ 功能名**：文档措辞(如"发送通知""订单下载")可能与实现命名不同——按**语义**找对应实现，找到才判有；反之"命名像"≠"真有"，要点进去确认。
- **修订痕迹/批注**：文档**删除线**=旧版已删，**不算现行声称**(单列"旧版已移除")；**高亮/批注**是审阅者标记，作为"疑似无"线索但**仍独立核验**给自己的证据。
- **数值/规格**：能静态判的判（如"最多6图"看前端有无 maxCount）；需计时/压测的标"留第③步性能实测"，本步只记指标与判定法。

## 3. 多文档：一份源文档 → 一份独立、自包含的核实文档（不合并）
**输入几份文档就产出几份核实文档，一一对应**——目的是能拿每份核实文档去**逐份纠正它对应的那份源文档**（像审阅者在每个 Word 上各自批注）。因此：
- **每份核实文档自包含**：核该源文档的**全部**功能/规格声称，**不要因为另一文档已核过重叠项就在本文档省略**——本文档必须能独立读、独立用来改它对应的源文档。
- **不合并成跨文档总表**；每份单独成文件、文件名标明对应哪份源文档（一份很大时可分卷 part1/2，但仍属同一源文档的核实，不与别的文档混）。
- **证据可复用、但各自列出**：同一事实在多文档出现时，底层代码/网页证据只需查一次、各文档复用其结论（免重复调查），但**每份核实文档都要把该原子完整列出**(判定+证据)，保持自包含。
- **跨文档矛盾**(同一点措辞/数值不一)：在**涉及的每一份**核实文档里都标注，以可复现实测为准。
- 有官方《测试记录单/报告》则在相应核实文档里逐条 1:1 映射。
> 自检：源文档 K 份 → 核实文档 K 份(可分卷)，每份覆盖其对应源文档全部声称、可独立使用。

## 4. 完成度闸门（可计量，禁纯自答）
- [ ] **全章节枚举（步骤 A0）**：源文档**每一章都出现在核实文档里** = 已核 X 章 / 全文共 Y 章（给分母）；判"非软件功能/资料类"的章也**留痕登记**（标待确认+理由），任何章**不得不出现**。——这是防"自报覆盖率虚高"的第一道闸门。
- [ ] **原子覆盖**：文档(逐章…)每句每原子都有判定 = 已判定 X / 原子总数 X（给分母）。
- [ ] **拆分粒度抽检（硬拦"句子层打包"）**：随机抽 3 句含顿号/逗号/斜杠并列的声称，核其是否**已拆到每个并列子项**；发现把"放大、缩小、平移、旋转"当 1 个原子这类打包 → **判未完成、回退重拆**。（否则分母被自己做小，X/X 假性 100%。）
- [ ] **证据齐**：每个判"有/无"的原子都有代码 file:line 或网页证据；判"有"而无证据 → 判未完成、补证。**（形态分流：纯 API/无前端形态下，"网页证据"以「真实请求-响应报文 + schema/契约校验输出」等价替代，与网页证据并列接受；后端规则类原子不得因无网页入口而记为缺证。）**
- [ ] **冲突清单**齐，且每条注明"以实测为准"的定论。
- [ ] **评审批注/官方记录单**的每一处都已覆盖并给出本方独立判定（一致/不一致）。
- [ ] **抽检**：随机 3 个"有"的原子，能按其证据复现；随机 2 个"无"的，能复述"怎么找过的"。
- [ ] 留③项（数值/性能/需写库实证）诚实标注。
- [ ] **★ 完整性审计（"核实的核实"，多文档/高要求时务必做）**：核实文档产出后，**每份源文档配一个独立审计子代理(subagent)**（纯文本对账+代码 grep 抽查、可并行、不开浏览器；三家终端均称 subagent），逐章查五件事：① **漏核**（源文档写了、核实文档无对应原子）；② **粒度打包**；③ **证据缺失**；④ **对不上未入清单**（无/部分/冲突未进未实现或冲突清单）；⑤ **证据真伪抽查**（抽 3 个判"有"的原子 grep 其引用代码确认真存在）。**"证据真"≠"覆盖全"——抽查 32/32 全属实仍可能整章漏核，两者分开验。** 审计发现的缺口→新建分卷补漏（文件名标"与X卷不重叠"）→回总表更新原子总数与导航。

## 5. 产出
《技术文档逐条(原子级)符合性对照表》，按文档章节号排序，落在 `…/第二步-编写测试用例/面4-文档符合性(独立)/` 下（或项目对应目录）。表列：
`文档章节 | 文档原文(原子) | 判定(有/无/部分/待确认) | 代码证据(file:line) | 网页证据 | 备注/冲突`
另出：**⚠未实现清单**（判"无/部分"的原子汇总）、**冲突清单**、**官方记录单/批注映射表**。详见 `references/verdict-table-template.md`。
> 落盘到**第②步产出根下的"文档符合性"目录**（跟随当前项目的产出结构，如 `<第②步输出根>/面4-文档符合性/` 或 `./test-cases/doc-conformance/`，**不写死绝对路径**）。
> **隶属统一根**：本核实表隶属测试三步走统一根 `测试产出/` 下第②步产出（默认 `第二步-编写测试用例/面4-文档符合性(独立)/`）；随第②步走，若用别名根/子目录须在《进度总览.md》登记真实路径，供第0步总管 `testing-step0-intake` 扫描续跑。
> 与主 skill 衔接：`testing-step2-write-testcases` 的"面4 文档符合性"**委托本 skill** 产出；本表即面4 的核心交付物，第③步执行时逐原子复验。
> **★ 对第③步（执行测试）的约束（预告执行强度，按形态分流见 §0.0）**：本《原子级核实表》的每一条，在第③步（`testing-step3-execute-tests`）执行时是**最高图文并茂要求的面**——**有面向用户前端的形态下：必须在真实网页逐条看过去/测过去、每条配真实网页截图 + 位置说明**（判"有"→截图显功能在；判"无"→截图显无入口；判"部分"→截图显打折扣），**不接受用源码佐证替代浏览器实证**，且第③步须对面4 做逐条重点审计。**纯 API/无前端形态（§0.0 ②）下：该组"必须浏览器截图、不可源码佐证替代"整组标 N/A，"眼见为实"改为「真实请求-响应报文 + schema/契约校验输出 + 代码 file:line」逐条留证**；仅"应有 UI 的声称"才强制网页截图。本步只做"代码+网页（或 API 真发请求）判定"产出核实表，第③步据此做"带截图/带报文实证"。详见 step3 §1 执行强度分级。

## 6. 反模式
❌ 句子层"打包"判定、不拆到顿号并列子项（漏掉"旋转""设备管理权限"这种）。
❌ 判"有"不给证据、凭文档措辞或功能命名臆断。
❌ "没看到就算无"——未 grep 代码、未去网页找入口就判无。
❌ 把文档删除线/旧版文案当现行声称来核。
❌ 只读文档对文档、不落到真实代码 file:line + 真实网页。
❌ 遇代码与网页/批注冲突时不实测定论、不单列。
