# 把文档拆成"原子声称"的细则

> 配合 SKILL.md 步骤 A。目标：把规格文档拆到**最小可独立判定**的单元，使每个原子都能单独给 有/无/部分 判定。examples 仅示意（用通用/虚构例），按你的项目实际文档拆。

## 一、什么是一个"原子"
一个原子 = 一个**可独立验证其是否实现**的最小声称。判据：你能对它单独说"这条做了/没做/部分做"。一句话里常含多个原子。

## 二、拆分规则（逐类）
1. **并列项逐个拆**：顿号「、」逗号「，」连接的并列功能/操作/权限/对象，**每个一个原子**。
   - 例：「商家具有 A 管理、B 下载、C 查阅、D 详情、E 发送、F 权限」→ 6 原子。
   - 例：「具有人工归档、分类管理、浏览查看功能」→ 3 原子。
   - 例（媒体/画布类项目的条件性补充）：「支持放大、缩小、平移、旋转」→ 4 原子。
2. **字段/列/属性逐个拆**：列表/表单声称包含的每个字段或列各一个原子。
   - 例：「账号列表含 用户名、最近登录时间、联系方式、状态」→ 把"最近登录时间""联系方式"等各拆为一个"该字段是否存在/展示"的原子。
3. **数值/规格各一个**：每个量化指标一个原子（标"留③实测"若需计时/压测）。
   - 例：「操作响应 ≤100ms」「最多插 6 张图」「并发 10」「≤2 分钟」各一原子。
4. **能力 vs 入口分开（必要时）**：当"能不能做"和"从哪做"都被声称时，可拆"功能存在"与"入口/角色可见"两个原子（如"审核员可打回订单"= 订单打回能力 + 审核员入口）。
5. **动作+对象**：动词作用在多个对象上，按对象拆（"可编辑或打回 订单/工单" → 订单编辑、订单打回、工单打回… 分别核）。
6. **复合句拆到从句**：一句话多个分句各自独立声称的拆开。例：「登录后进入工作台，并按角色显示不同菜单」→「登录进工作台」「按角色显示菜单」两原子。
7. **嵌套 / 括号 / 斜杠 也要拆**：括号内子项、斜杠并列同样拆。例：「导出为 PDF/Word/Excel」→ 3 原子（导出PDF/导出Word/导出Excel，逐个核是否支持该格式）；「放大（2x/4x/8x）」→ 缩放能力 + 各倍率（若各倍率被单独声称）。
8. **否定式声称单独拆并反向判**（权限/安全文档极常见）：「系统不支持匿名访问」「禁止越权查看他人数据」→ 拆为原子，判定语义是"**确实没有/确实拦住=有(符合)**、能做到=无(不符合,缺陷)"。别漏掉否定声称。
9. **条件式声称：把"触发条件"也作原子**：「仅管理员可删除」「满足A时才允许B」→ 至少两原子：①该能力存在；②**条件/限制确实生效**（非管理员删→应被拒）。条件本身是可独立验证的声称。
10. **契约即规格（条件触发·若有 OpenAPI/Swagger/proto/gRPC IDL）**：把契约里**每个 path/operation、每个参数、每个 schema 属性、每个枚举值、每个 required 约束、每个错误码各拆为一个原子**。
    - 例：`POST /orders` 的请求体 schema 含 `items[]`(required)、`couponCode`(可选)、`channel`(枚举 web/app/api)，响应可能返回 `409 OUT_OF_STOCK` → 拆为：端点存在 / `items` required 校验 / `couponCode` 可选支持 / `channel` 枚举三值各一 / `409 OUT_OF_STOCK` 错误码 = 多个原子，逐个对「契约声明 ↔ 代码实现 ↔ 实测响应」三方对账。
    - **纯 API 形态（无前端）**：上述「契约 ↔ 代码 ↔ 实测响应」三方对账是**主线 oracle**，散文需求文档作为**补充 oracle**；契约原子的"实测"= 真发请求看响应报文 + schema 校验，而非网页截图（见 SKILL §0.0/步骤 C 形态分支）。

> 复杂声称示例（中性，非医疗）：「订单列表支持按 下单时间、支付状态、收货地址 筛选，并可导出为 CSV/Excel；仅商家可批量发货」→ 拆为：筛选-下单时间 / 筛选-支付状态 / 筛选-收货地址 / 导出CSV / 导出Excel / 批量发货能力 / "仅商家"限制生效 = **7 原子**。

## 三、每个原子要带的最小信息
- **定位**：文档章节号 + 原文片段（如 `2.1.5.2.1 · 最近登录时间`）。
- **归一后的声称**：用一句中性话表述"声称软件能 X / 含字段 X / 满足 X"。
- 留待判定的列：有/无/部分/待确认 + 证据（见 verdict-table-template.md）。

## 四、不该拆/合并的情况
- 纯背景叙述、术语定义、修订说明、与功能无关的套话——**不拆为原子**（标"非功能声称，不核"）。
- **套话 vs 模糊真声称 判别启发式**（降低 prose/营销化文档的分母主观性）：问"这句**隐含可验证的行为/字段/数值/限制**吗？"——有→拆为原子核实；纯主观形容("无缝体验""推荐精准""界面友好")且无可验证锚点→记"非功能声称(不可验证)，不核"。把判别写进备注，使分母可被复核、不同人拆得一致。
- 同一原子在多处文档重复出现 → **合并为一条**，核一次，注明多处出处。
- 文档**删除线**内容（旧版已删）→ 不作现行原子，单列"旧版已移除（不计声称）"。

## 五、拆分自检
- 这条还能再拆出能单独判"有没有"的子项吗？能就继续拆（尤其看有没有漏掉的顿号并列子项）。
- 评审高亮/批注里点到的每个词，是否都对应到了你的某个原子？没有就补。
- 原子总数 = 覆盖率分母；拆得越细，"漏判"越少。
