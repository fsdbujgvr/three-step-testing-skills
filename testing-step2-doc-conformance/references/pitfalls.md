# 文档符合性核对的避坑（保"逐词准确"）

> 配合 SKILL.md。这些坑会让"有/无"判错——而本 skill 的命脉就是准确，所以逐条防。

## 准确性相关
1. **句子层打包 → 漏并列子项**：最常见错。一句"放大、缩小、平移、旋转"只判一条"图片操作=有"，就漏了"旋转=无"。**强制拆到每个顿号并列项**。
2. **判"有"凭命名/措辞臆断**：文档写了某功能名，不等于真实现。必须点进代码 file:line 或网页实操确认才判有。
3. **"没看到=无"**：未 grep、未去网页找入口就判无 → 误报。判无要给"查了哪些词/模块 + 网页无入口"的证据。
4. **术语 ≠ 功能名**：文档措辞与实现命名常不同（"发送通知"可能实为"导出/打印通知"；"订单"可能实为"工单/单据/记录"）。按**语义**找对应实现，找到算有(标对应关系)，确实无对应才判无。
5. **命名像 ≠ 真有**：代码里有个名字相近的东西不代表实现了该声称——要确认其行为/可达性。
6. **源码词 ≠ UI 词**：源码标识符可能与 UI 显示文案不同（i18n key、运行期文本替换、模板变量等，如某些前端做运行期 DOM 文本替换）——按 UI 显示词 grep 源码会落空。**优先用源码原词/路由/字段名/i18n key 搜**，找不到时去入口文件/i18n 资源找替换映射。

## 文档本身的坑
7. **修订痕迹**：**删除线**=旧版已删，**不算现行声称**（单列"旧版已移除"，别拿它当原子核）；改写处以现行(未划线)文案为准。
8. **高亮/批注**：是审阅者的标记（常标"疑似软件没有"），**作为线索但仍独立核验**给自己的代码+网页证据——不能直接照抄批注当结论(批注也可能错，如把实际存在的功能误标为无)。
9. **多文档矛盾**：综述/技术要求/用户手册/操作步骤 对同一点措辞或数值不一——单列矛盾，核实际实现。
10. **数值/规格规格**：能静态判的判（"最多6图"看前端 maxCount/slice；"无某维度"看有无该字段/接口）；需计时/压测/弱网的(响应≤Xms/并发N/时延)标"留第③步实测"，本步只记指标与判定法，别凭感觉判过。

## 证据与边界
11. **前端有 ≠ 系统有 / 后端有 ≠ 用户可达**：声称的功能可能前端有壳后端没实现，或后端有端点前端无入口——这类记"部分/冲突"，分别给两侧证据，以网页实测可达性为用户视角定论。
12. **冲突必实测定论**：代码↔网页↔批注 不一致时，**实际去网页操作**给结论，并单列冲突（"以可复现实测为准"）。典型："代码有 return 端点 vs 批注说网页无该功能"——必须真去点。
13. **写库类只观察**：核对"有没有某写功能"时只看入口/控件是否存在与可点，**不真提交造数**（造数实证留第③步授权）。
14. **保持可复现**：每条"有/无"都要让别人能照证据复核（代码行号能跳转、网页路径/步骤能重走、grep 命令能重跑）。

## 覆盖完整性的坑（★ 最隐蔽、危害最大）
15. **静默跳过"非主功能章" → 自报覆盖率虚高**：核实时天然倾向只核"主功能章节"，而把 前言/免责/术语表/附录/版本命名/性能/安全/可靠性/维护/兼容性/数据备份 等**整章悄悄略过**，最后却自评"基本完整"。**而这些被略章恰恰藏着最高价值的失真**（术语表/附录里的技术栈声称、版本规则、安全级别表、审计日志可核查性、删除二次确认等可核强声称）。**对策**：拆原子前先列**全章节目录**逐章打勾(SKILL §A0)；判"非软件功能/资料类"也要**留痕登记**，任何章不得不出现。
16. **"证据真" ≠ "覆盖全"**：抽查判"有"的原子证据全部属实，**不能**推出"核全了"——证据真伪与覆盖完整是两件事，要分开验。多文档/高要求场景**务必跑"核实的核实"**（每源文档配独立审计 agent 查 漏核/粒度/证据缺失/对不上未入清单/证据真伪，见 SKILL §4）。曾出现抽查 32/32 全属实、却整本只核了一章的情形。
17. **一源文档一份、补漏开新卷不塞回**：审计发现漏核 → 新建分卷（文件名标覆盖范围+"与X卷不重叠"），再回总表更新原子总数与导航；不要硬塞回原卷打乱其自包含性。
18. **后台 agent 收尾断连丢成果**：`run_in_background` 的核实/补漏 agent 可能在写文件+返回摘要时 socket 断连，工具调用做了但**文件没落盘全丢**。prompt 里要求"先落盘(主交付物)、分批 Write、摘要极简"，完成后 `ls`/`wc -l` 验证真落盘，没落盘就重跑。
