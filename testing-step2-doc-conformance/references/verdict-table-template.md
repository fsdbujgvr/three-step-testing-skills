# 逐条(原子级)符合性对照表：模板 + 判定标准

> 配合 SKILL.md 第 5 节。这是核心交付物的格式与"有/无/部分/待确认"的判据。

## 一、对照表列定义
按文档**章节号顺序**逐原子一行：

| 列 | 含义 | 要求 |
|---|---|---|
| 文档章节 | 原子在文档中的定位 | 章节号(如 2.1.9.1)，多文档时标文档名 |
| 文档原文(原子) | 该原子的原文片段 + 归一声称 | 抄原文关键词，别概括掉细节 |
| 判定 | 有 / 无 / 部分 / 待确认 | 见下"判据" |
| 代码证据 | 实现点 file:line 或"grep X 无果" | 判"有"必填实现点；判"无"填"找过的关键词/模块" |
| 网页证据 / API实测 | 入口/控件/行为实测 或"网页无入口"；**或**（后端/无 UI 规则类原子、纯 API 形态）真实请求-响应报文 + schema/契约校验输出 | 截图路径 / DOM 片段 / 实操结果；或 请求-响应报文 + 校验结论。两者**并列**为"眼见为实"的证据来源（按形态分流，见 SKILL §0.0） |
| 备注/冲突 | 冲突、留③、术语对应、旧版删除、形态(N/A 网页) 等 | 冲突必写"以实测为准"的定论 |

## 二、判定标准（必须可复核，禁臆断）
- **有**：**默认要求代码实现点(file:line) 与 网页入口可达且生效 两侧都成立**。仅一侧成立时**不得直接判"有"**——按下面"部分/冲突"处理。
  - **形态分流（按 SKILL §0.0）**：对**后端/无 UI 规则类原子**（幂等/事务原子性/金额公式/状态机校验/契约字段）或**纯 API/无前端形态**，"网页入口可达且生效"这一侧由「**真实请求-响应报文 + schema/契约校验输出**」**等价替代**——代码实现点 + API 实测两侧成立即可判"有"。**此类原子不得因"无网页入口"而判"无"或扣证；只有"应有 UI 而 UI 缺失"才据此判无。**
- **无**：代码 grep 相关语义关键词**无果**（列出查了哪些词/模块）**且** 网页无对应入口/控件/行为。"没看到"不等于"无"——必须给"找过了"的证据。
- **部分**：原子只实现了一部分；或**后端有实现但前端无入口/不可达**（同时并入冲突清单）；或仅某角色可用而声称面更广。
- **待确认**：证据不足以定论（需运行期/写库/计时才能判，或代码语义模糊、后端真假未明）。**绝不用"有/无"代替"待确认"。** 标明"还需怎样才能定论"。

**同类证据的优先级裁决（消歧"部分/冲突/待确认"）**：
- 后端有端点/实现 + 前端有入口且实测生效 → **有**。
- 后端有 + 前端**无入口/点了无效** → **部分**（并入冲突清单："后端有、用户不可达"）。
- 后端有 + 前端入口在但**因写库不能非破坏性实测**（如下单/退款/删除）→ **部分**（注"后端有+前端可触发，提交实测留第③步授权"），不要一律压成"待确认"。
- 代码语义模糊、连"后端到底有没有"都判不准 → **待确认**。

## 三、配套产出（同目录）
1. **⚠未实现清单**：所有判"无/部分"的原子汇总（章节 + 原文 + 缺什么 + 证据）——直接对应"软件没有的功能"。
2. **冲突清单**：代码↔网页↔文档↔评审批注 不一致项，每条给实测定论。
3. **官方记录单/评审批注 映射表**：官方/批注每一处 → 你的原子判定 → 一致/不一致。
4. **数值/性能规格清单**：标"留第③步实测"的指标(原文值 + 判定法)。

## 四、汇总指标（放表头）
- 原子总数 N；判定分布 有 a / 无 b / 部分 c / 待确认 d（a+b+c+d=N，d 应尽量小）。
- 评审批注覆盖：批注 M 处，已逐处给独立判定 M/M。
- 章节覆盖：文档功能章节逐节已拆并判定。

## 五、一行示例（占位，按项目替换）
| 2.1.9.1 | "支持图片…旋转" → 声称可旋转图片 | 无 | 图片预览组件仅缩放/平移，grep `rotate/旋转/transform.*rotate` 无果(<预览组件>:行) | 网页图片查看器仅缩放+悬停平移，无旋转控件 | ⚠未实现；放大/缩小/平移见同句另3原子 |
| 2.1.5.2.4 | "管理员可打回订单" → 订单打回能力 | 待确认 | 无 `order.*return` 端点(grep 无果)；order 有 returned 状态值但无置位入口 | 管理页订单无"打回"按钮(实测) | 倾向"无"；与某批注一致；待第③步管理页复操确认 |
| 2.1.x | (反向冲突示例)某操作 → 评审批注标"软件无" | **有** | 实现点 `<文件>:行`(确有该端点/逻辑) | 网页实测该入口存在且生效(截图) | **与批注不一致**：批注误标为无，**以实测为准判有**；单列冲突 |
> 反向冲突很重要：评审批注/官方记录单也会错（把实存功能误标为无）。**两个方向都要核**——"声称有但实测无"和"批注说无但实测有"都进冲突清单，一律以可复现实测为准。
