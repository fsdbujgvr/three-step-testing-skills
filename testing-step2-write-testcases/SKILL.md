# 编写测试用例（测试三步走 · 第二步）

## 唯一目标
在第①步《项目理解文档》之上，产出一套**最多、最完善、全覆盖、可执行、可追溯**的测试用例，详细到"没做过本系统的人照着就能执行、且能回溯到每条用例的依据"。**本步只编写、不执行**（执行、性能实测、写库验证属第③步）。**判定基准恒为：代码 > API 实测 > UI/渲染 > 文档**——文档只是"声称"，用例须把它当对照对象而非事实。

> 未过第 8 节「完成度闸门」（面×维度矩阵填满 + 回链第①步分母 + 跨模块联动 N/N + 回环覆盖）**及 §8.5 产出后对抗审计**前，不算完成。

## 0. 前置：必须先有第①步《项目理解文档》
本步的输入是第①步产出（`testing-step1-understand-project` 的结果）：数据模型/全部API(含副作用与错误文案)/每页元素与校验/状态机/业务闭环/owner传播/真实数据基线/文档对照/无鉴权与越权清单/代码级穷举基线。
- **若还没有第①步产出 → 先去做第①步**（用 `testing-step1-understand-project`）。没有它，用例会浅、会漏、会写死成猜测。
- 第①步锁定的"分母"（API N 条 / 页面 M 个 / 状态机 S 个 / 实体字段 / 文档条目 / 无鉴权清单）就是本步覆盖率的**分母**，逐项回填用例。

### 0.1 形态分流（form factor）— 决定哪些面/闸门适用（统一概念，step0/1/2/2.1/3 一致引用）
被测系统的**形态**决定哪些面/闸门**适用**、哪些整组标 N/A。形态在 step0/step1 识别，经《项目信息确认卡》/《进度总览》向下游传递；本步据此**裁剪闸门**而非空转论证。常见形态：
- **①标准 Web 应用**（默认，有前端+源码+账号）——7 面全适用。
- **②纯 API / 无前端**（仅 REST/gRPC + 契约，凭据为 token / API-key / mTLS / cookie-session / OAuth，按项目）——下列"需前端"的项整组标 **N/A**：面1 UI、面4"必须浏览器截图、不可源码佐证替代"、§8「交互方式覆盖率」闸门、`browser_snapshot` 可操作元素穷举、角色级"后端 200 不算通过/必须前端可达"。改以**"真实请求-响应 + schema/契约校验 + 代码 file:line"为"眼见为实"**；面3 收敛为"契约一致性 / 超时重试 / 畸形请求"而非前后端校验对比。
- **③薄 UI 后端 / 交易中台**（价值在后端业务规则，UI 是薄壳）——重心压在面2/面7 的业务公式/事务/幂等键/状态机迁移逐条 file:line；UI 面按薄壳实际控件覆盖即可。
- **④实时协作**（标准应用叠加 WebSocket/SSE 实时通道）——7 面照常，**额外挂载"实时协同专项"**（见 facets 条件触发表）。
- **⑤含富交互**（canvas/媒体/可视化）——7 面照常，富交互按"命中即必测"逐方式穷举（见 §3/§8/§11 与 facets 面1）。
> **铁律**：以上"需前端"项**仅在"有面向用户的前端"时适用**；纯 API/无前端形态下整组标 N/A，不必逐项论证"不适用"。但**只要有前端，原有严格度一丝不减**。形态未在《进度总览》登记时，**默认按①标准 Web 应用**全量适用，不得擅自降级。

## 1. 核心架构：测试面 × 测试维度 矩阵（三轴，别用扁平清单）
覆盖最不易漏的结构是一个**矩阵**：每个"测试面 × 测试维度"格子都自问"该不该有用例"，空格就是待补。区分三轴——**面/维度是通用骨架，内容才是项目特有**（这是"通用 vs 特有"的真分界线）。

### 轴一 · 测试面（7 个稳定外壳，判定基准 = 代码+浏览器行为真相）
1. **UI 功能与闭环**（浏览器/前端渲染·所见与可用；含跨模块"建X验Y"闭环）
2. **后端接口与数据**（实现真相·返回与落库·副作用）
3. **前后端通讯**（请求方式/超时重试超限/前端态 vs 真落库）
4. **文档符合性（★独立面！）**——判定基准是**外部需求文档的"声称"**，做的是"实测 vs 声称"对账，与其它六面的 oracle 本质不同；也是官方《测试记录单/报告》的唯一对接口。**必须独立成一类/文件夹，不可拆散进各模块**，否则丢失"哪些文档不可信/声称未实现"的整体视图。
5. **权限/越权/认证/会话**
6. **安全与配置**
7. **数据持久化与端到端深度**（写后回读·原子性·级联·闭环逐步回读）

### 轴二 · 测试维度（横切各面，逐面施加，别只放一处）
功能正向 · 边界/超出 · 状态机(合法/非法流转) · **计算/业务规则正确性** · **并发与数据完整性** · **展示鲁棒性/脏数据/未映射枚举** · **错误恢复/原子性** · 权限越权 · **写后回读持久化** · 编码/本地化 · 兼容性 ·（性能多留③）。

### 条件触发（按第①步发现的"项目形态特征"勾选，命中才加对应面/子类）
有实时流/音视频/AI → 媒体与第三方专项；有 Canvas/富编辑器 → 画布深验；有导出/打印 → 导出保真；有文件/对象存储+历史迁移 → 存储多链路；**有产品/需求文档 → 启用第4面文档符合性**；多用户 → 并发加重；有统计/派生值 → 计算正确性加重。

### 轴三 · 内容（项目特有，**绝不写死进通用流程**）
具体公式 / 脏数据样本 / 技术选型 / 业务域功能 / 端点·字段·文案——全部由第①步发现**按项目填充**。
> 详细的面×维度展开、条件触发表、贯穿维度，见 `references/facets-and-dimensions.md`。

## 2. 用例规范（先定规范，再批量写，保证一致）
每条用例统一字段：**ID / 标题 / 优先级(P0-P2) / 类型 / 前置 / 步骤 / 预期 / 判定依据(追溯) / 备注**。
- **预期必须可断言**：抄错误文案原文(带引号)、HTTP 状态码、响应信封、**DB 写入/级联/回读结果**、UI·toast·跳转 原文。**禁止"应该正常"这类空话**——执行者无法据此判定通过。
- **判定依据(追溯)**：每条回指 第①步文档§ / 代码 file:line / API 实测 / 文档出处，使用例可被复核、不沦为臆测。
- ID/优先级/类型 取值与模板细节见 `references/case-conventions.md`。

## 3. 穷举法则（"最多最完善"的可操作定义）
- **每个端点** = 正向(预期返回+DB写入) + **每条错误分支**(逐状态码+文案原文) + **每个字段边界**(空/超长/特殊字符/中文/前后空格/最值±1/格式正则合法非法/唯一重复/默认) + **鉴权三态**(无token/低权/高权)，各独立成条。
- **每个状态机** = 每条合法流转 + **每条非法流转**(错误来源状态触发动作→错误码/文案) 各一条。
- **状态机回环完全统计（环路各走通一遍 · 不重复 N 次）**：单跳流转之外，**把每个状态机里所有"能回到先前状态"的环路逐条列全**（如 待审→打回→整改→待审、草稿→提交→驳回→草稿），**每条环路至少一条用例走通一遍**（一条端到端用例串完整环、可跨角色：如 审核员打回→提交者整改→审核员看到整改内容→审核员复核/再打回）；**不要求把同一环路重复循环 N 次**——同一回环跑多遍耗资源、无额外价值，走通一遍即覆盖。环路清单来源于第①步状态机图谱；无环路的状态机显式标"无回环"。**这是"完全统计整个项目所有回环"的落点**（区别于上一条"单跳流转各一条"的边覆盖）。
- **每个页面（命中即必测）** = 每个**可操作元素 × 其支持的每种交互方式**（别把"可操作"隐性等同于"可点击/可输入"）：
  - **默认硬性（恒必测）**：每个可操作元素 × 其**默认支持的交互**（单击/输入/选择/键盘可达）各有 TC——按钮点击/输入框/枚举/空态/加载态/分页/筛选/搜索/二次确认。
  - **富交互（命中即必测、命中才进分母）**：双击、右键/上下文菜单(每菜单项)、悬停(缩放预览/tooltip/预览/高亮)、拖拽绘制(canvas/标注)、拖放上传、长按、滑块/进度条拖动(含键盘±微调到边界)、滚轮缩放、弹窗内部各控件交互——**仅当第①步识别出该控件确实支持该交互时，才纳入覆盖分母、逐方式各建一条可独立断言的用例**（命中时逐方式严格穷举，一个都不能省）。
  - **整体豁免（不必逐控件标不适用）**：若本页/本系统控件仅点击/输入/选择、无任何富交互，一句**"本页/本系统控件仅点击/输入/选择，无富交互"**即整体豁免该组，不必逐控件标"不适用"。
  - **命中时的非点击交互是面1 主动穷举项（不依赖文档），不要只靠面4 文档拆原子来覆盖（文档没写就漏）。**
- **每个写端点（除边界外）= 加一条幂等/并发用例**（条件触发：凡写端点带唯一约束或操作库存/计数/金额类字段即必加；属功能正确性、必真跑，不是吞吐压测、不得归留③）：幂等键重放、并发扣减、重试副作用 至少各按其适用性补一条（配方见 `references/facets-and-dimensions.md` 条件触发表「并发/幂等」行）。
- **每个跨模块依赖（owner传播链/状态同步点/级联）= 一条"改A→去B/C/D验证→回退恢复再验"联动用例**：分母取自第①步《业务闭环图谱》（owner传播/状态同步/级联）——改上游配置/权限/数据/状态后，逐个回到依赖它的下游功能验证"是否同步生效/受影响正确"，再回退恢复确认下游随之复原。**这类用例跨多个模块、不属于任何单模块组，必须在基础用例写完后由持全局上下文者专写**（见 §6「跨模块联动专写阶段」），不在逐模块扇出那一波写。
- **已知缺陷也写成用例**（第①步发现的无鉴权/越权链/脏数据/声称未实现），标 **⚠预期暴露缺陷**——其"通过"语义是**复现缺陷**（不可记系统合格）。
- **需写库/外设/性能的用例**标"**需显式授权(第③步执行)·造数自清理·唯一前缀**"，本步只定义不执行。

## 4. 数据持久化"写后回读"铁律（最易漏、最致命）
只把"DB 应写入"写进预期、却**不回读确认**，等于默认"HTTP 200 / 前端 toast = 真入库"——这是错的（可能乐观更新、半成功、或只改了前端态）。**凡写操作（POST/PATCH/DELETE/上传/状态流转），用例必须含显式"写后回读"步骤**：
- POST → 重新 GET 资源/列表，断言新行 + 字段落库 + **服务端默认值**正确。
- PATCH → 回读断言改动持久化；验"未传字段不变 / 显式 null 是否清空"。
- DELETE → **先判软删/硬删**：软删回读 = 记录仍在但被默认查询过滤 + **唯一键是否释放（能否重建同名）** + 引用方行为（仍引用/置空/报错）；硬删按 回读 404 + 列表计数 -1 + **级联子数据回读消失**（或外键置 NULL 回读）。
- 上传 → 回读资源行 + **取静态文件字节确认文件真在盘** + 刷新仍在。
- 状态流转 → 回读 status+时间戳 + **回读连带副作用实体**。
- **重复写后回读（验幂等）** → 同一业务操作重复触发，回读断言**计数/金额是否守恒**（只建 1 行、金额不翻倍），区分"幂等"与"重复写脏数据"。
- **一次业务多表写后回读（最终一致）** → 一笔业务跨多表写入时，逐表回读断言各表状态一致、无半截（如下单需库存表-↓、订单表+1、流水表+1、优惠/积分表对应变更四表最终一致，按本项目实际表结构替换表名）。
- **实时增量写（WS/SSE 帧驱动，条件触发：有实时通道才适用）** → 既 **DB 回读持久化**（帧驱动的变更是否真落库），**又**第二在线客户端回读是否收到对应广播 / 最终态与首端一致（双 oracle，见 facets「实时协同专项」）。
- **换会话/刷新/重新登录**确认真 DB（而非前端态）；**错误分支也回读**确认无半截脏写。
- 辨"前端态 vs 真落库"：AI/建议类(返回≠已保存)、勾选/编辑类(是否真发请求且回读生效)。

## 5. 文档符合性（第4面专章）
> **★先主动判定本项目有没有专门文档（别假定有/无，自己查两处）**：① 扫项目目录(`文档/`·`docs/`·`需求/`·`README`·`*.md/*.docx/*.pdf` 等)看有没有 独立的 需求/技术/操作/用户/测试 文档；② 看第①步是否已产出《产品文档对照与冲突》。据此走分支——**有独立文档**→用专项 skill 逐条对照(下文)；**无独立文档、只有代码注释/docstring/README**→以其"声称"(如有)对照实现、抓"注释声称≠代码实际"的过时失真；**连注释声称都没有**→面4 记"无文档可对照、不启用"、其余6面照常、不判未完成。一律以 代码/可复现实测 为准。（防"跳过第0步或第①步漏看 `文档/` 目录就误当无文档"。）
> **本面委托专项 skill `testing-step2-doc-conformance`（即 2.1）产出**——因为它要求"逐句拆原子、每原子对代码+网页给 有/无/部分 判定、准确性第一"，是与其它面不同的活。需要逐词级文档对照时优先用该 skill 出《逐条原子级符合性对照表》。
> **编排建议：把这一面交给独立的子 agent 在干净上下文里专做**（甚至**按文档章节扇出多个 agent 各包一章**），主流程只负责委托与汇总——因为本面最吃精度，混在主流程里上下文一杂就容易"句子层打包/凭印象判有"。下面是其要点摘要：
1. 需求文档 **每条声称/规格 → 正向核验用例**；**每个操作步骤 → 走查用例**；**声称却疑似未实现 → 负向核验用例**（验证 UI/API 确无此能力）。**关键：拆到原子级**——顿号并列的每个功能/操作/权限/字段(如"放大、缩小、平移、旋转"=4项)都单独核"有/无"，别句子层打包。
2. **若项目已有官方测试记录单/测试报告，必须取来逐条 1:1 映射**，产出**覆盖映射表**(官方用例ID→你的TC编号→状态)。官方"纯按文档设计"的用例与本面**同源、常约 80% 重叠**；**仍要逐条核验**——易漏"使用限制/展示鲁棒性"类软条款这类真缺口。
3. **你的"代码+浏览器查证"用例 ⊋ 官方"纯文档"用例**：能多发现 架构失真/声称未实现/硬编码当数据/未映射枚举/安全头·越权 等——官方纯文档覆盖不到，是增量价值，务必保留。
4. **排除项显式标且给去向**（如代码扫描/压测被指明忽略）；但**性能指标条目仍要收录**(标留③)，别连指标一起丢。多份官方文档去重（报告的功能性表常=记录单同套）。

## 6. 多轮迭代与交叉补充（别一次成型）
- **扇出子代理(subagent)并行编写——必做、不是可选**：用例量大，**环境支持子代理时必须扇出，且派出数有公式可循：大面（面1 UI、面2 后端接口、面7 持久化与端到端）各按第①步模块清单分组（「默认一个模块一组」；仅模块极小/强相关时才合并——一组一般≤2 个模块、合并组不应占该面行数的多数，且须在《扇出计划表》"负责块"里列明所含模块并注一句合并理由）、每个"面×模块组"一个专注子代理（各自穷举该块的"面×维度"格子）；小面（面3 通讯/面5 权限越权/面6 安全配置）各 1 个、「每个命中的条件触发专项各 1 个」；面4 委托 2.1（按其编排：文档大按章节扇出、小文档单代理）——合计远多于"两三个"，粗拆成两三个≈没扇出**。**开派前先落盘《扇出计划表》（放第②步产出目录）：模块组直接取第①步模块清单（不重新发明），表头登记"第①步模块 N=… → 面1/2/7 各组数=…、合并 X 处"（组数 vs N 的差距一眼可见），逐行登记"序号 | 负责块(面×模块组) | 用例ID段 | 预期落盘文件"（面4 行按 2.1 编排登记：文档大按章节多行、小文档一行"面4×<文档名>全册"，面4 用例标识 = 2.1 对照表的章节号/原子行定位、不占本表 TC ID 段；用例ID段按预估上浮留余量——「段用尽≠写完」,顺延编号并在《格子清点》说明,顺延前核对计划表同类码+模块无其它行占用、被占则另开续段——回写《扇出计划表》由主循环串行统一做,子代理只在《格子清点》申报实际用到的号段、避免并行写竞争），照表逐行派遣，别自己判断"要不要拆/够不够"**，回收后再汇总去重；**各块《格子清点》非满格（暂缺/没写完/建议续派）的,主循环须续派补齐或转为正当标注（真不适用/留③）——"暂缺"不是 §8 可接受的终态**——比单上下文囫囵全写更深、且防"写到后面赶工变浅"。**（以上是必做下限——不许判断成不拆；在此之上鼓励按项目特点拆得更细/更多，上不封顶。）** 三家终端 Claude Code/OpenCode/Codex 均称 subagent；支持 workflow/编排的环境用其更强并行；环境不支持子代理才单线程（判"不支持"须在《进度总览》记一句判定依据,如试派失败的报错原文/工具列表确无子代理类工具,无《进度总览》时记入本步 README 或覆盖矩阵），但分块逐个写、每块写完即落盘防赶工；文档符合性面本就委托独立子代理按章节扇出，见 §5。
- **子代理任务书（防"派而不深"——子代理看不到本 SKILL，全靠任务书带要求）**：给每个编写子代理的任务 prompt 必须写明：① **边界**——负责的"面×模块组"+ 分配的用例ID段 + 落盘文件（取自《扇出计划表》,写**绝对路径**）,别替邻块写用例；② **输入**——第①步该模块详解/代码基线/真实数据基线的文件路径（绝对路径） + **已先落盘的《00-用例编写规范》**（模板/ID规则/优先级/追溯/标注约定——跨子代理一致全靠它,开派前必须先写好） + **本块适用的"面×维度"格子清单与穷举法则要点**（格子清单=清点分母,**须由主循环逐项列明贴入任务书**、不得只甩文件指针——"本块适用哪些维度"是形态裁剪/条件触发命中后才知道的主循环判断；穷举法则细则可给 references/facets-and-dimensions.md 对应节的绝对路径；穷举要点如：每端点=正向+逐错误分支+逐字段边界+鉴权三态、写操作必配写后回读——**《格子清点》的分母以此为准,不许子代理自定义维度**）；③ **产出格式**——每条 TC 按规范模板,预期结果必须含**机器可核对锚点**（状态码/错误文案原文/DB回读/精确值）,追溯回链第①步 §/file:line；本任务**只读**（读第①步产出与源码即可写用例,不操作被测系统、不写库）；④ **投入策略原话带给子代理**——"全面性 > 效率、宁冗长勿遗漏、不设 token 上限"（不写这句，子代理会按省 token 的默认习惯干活）；块超载时分批落盘、如实报"写到第X格、建议主循环续派余下"——**宁可报没写完,不许压质量凑数量**；⑤ **收尾自报《格子清点》**——"该块'面×维度'格子共 X 个、已写 X/X,空格子逐个标原因（真不适用/留③需授权/暂缺）",随产出落盘；任务书全文随产出落盘一份（与《扇出计划表》同目录或附于该块用例文件末尾）；返回值只报"落盘路径 + 用例计数 + 清点摘要"、正文以落盘为准。《格子清点》与产出供 §8.5 审计抽核。
- **分多轮**：第①轮按"面×维度矩阵"铺基础；第②轮对照覆盖矩阵与第①步分母补**薄/缺**的格子；第③轮做**交叉补充**——一类的发现催生另一类用例。
- **★跨模块联动专写阶段（必做·后置·依赖前两轮）**：跨模块"改A→验B/C/D→回退"联动用例**不在逐模块扇出那一波写**（单模块子代理没有跨模块视野、上面任务书①明令"别替邻块写用例"），而是**等第①②轮基础用例全部落盘后单独一阶段写**（**可开始的硬条件：《扇出计划表》每行的《格子清点》均非"暂缺"**——基础未全落盘则联动阶段不算可开始，别在残缺基础上写联动）——此时各模块已摸透、基础用例可直接复用引用（"改 A 后原 TC-xxx 的预期是否仍成立"），联动用例才写得准、写得全。四步走：① **开派前先锁分母**——②一开始就读第①步《业务闭环图谱》（owner传播/状态同步/级联），把每条跨模块依赖枚举成**《联动点清单》共 N 条**（每条="改 A → 影响 B/C/D"）落盘锁死（**先锁分母、暂不写用例**，防"做完就忘"被静默跳过）；② **专人来写**——由**主循环（持全局上下文）或一个专门子代理**写，喂给它《联动点清单 N》+ **全部已落盘的基础用例文件** + 第①步闭环图谱，**严禁让任一单模块子代理顺带写**；③ **独立落盘**——联动用例**独立成目录/文件**（比照面4 独立成面的待遇，不并进各模块文件、不按模块拆散），每条含"改A→去B/C/D验证→回退恢复再验"三段，**且"验B/C/D"段必须含 ≥1 机器可核对锚点（下游 DB 回读值 / 状态码 / 错误文案原文 / 精确数值），"回退恢复"段必须回读确认下游随之复原——禁止"改A→打开B看一眼没报错→回退"这种无断言空壳**（范例见 `references/case-conventions.md` §8）；④ 过 §8「跨模块联动 N/N」闸门（见 §8）。
- **类与类相互喂**（举例）：持久化↔并发（前端态≠真落库→并发下回读是否一致）；异常数据↔持久化（未映射枚举白屏→后端真返回该枚举时崩不崩）；安全↔原子性（公开可写→批量滥用/多步失败留孤儿）；错误文案↔文档符合（toast 类型误用 既是 UX 缺陷又是差错防御符合性点）。
- **每轮收尾问三句**：①矩阵里还有哪个格子空？②已覆盖的格子，是否只写了正向、缺边界/异常/越权/回读？③本项目独有特性(实时流/存储/AI/画布/多角色)是否各有专项？答得出"还缺X"就再来一轮。

## 7. 诚实性纪律
- **区分"原本已覆盖" vs "看到对照材料后才补"**：做覆盖映射时给每条标 `pre-existing / added-after`；**绝不把"补完后 100%"说成"原本就 100%"**。
- **覆盖性结论证据驱动**：用 文件时间戳 / grep 实证 回答"是否已包含"，而非凭记忆断言。
- 被追问时如实拆分：直接已有 / 他处已测后补成专条 / 真缺口后补——不粉饰。

## 8.5 产出后对抗审计（派独立子代理复核用例集，别自评"全覆盖"）
过完 §8 闸门后，**必做：必派一个独立子代理(subagent)做对抗审计**（干净上下文、立场怀疑、默认不通过），逐项举反例；**未做这轮独立审计 = 本步未完成、不许进第③步**：
- **可断言性**：抽 5 条，预期是否含机器可核对锚点（状态码/错误文案原文/DB回读/精确值）——"应该正常/功能可用"空话即不合格；
- **覆盖真实性**：矩阵是"真填满"还是"空格充数"？抽几个"已覆盖"格子核对其用例真存在且对题（可用各块《格子清点》定位抽样；标"真不适用/留③"的核其原因真伪）；
- **追溯真伪**：抽 3 条追溯（第①步§/file:line），回查是否真存在、非编造；
- **可执行性（反推）**：抽 2~3 条，判断"第③步执行者凭它能否直接执行"——步骤/预期含糊到没法执行 = 不合格；
- **漏面**：对照第①步分母（API N/N、状态机 S/S、页面 M/M、**跨模块联动 N/N**），有无整面/整维度漏写；**联动尤其要回查保真：抽 2~3 个 step1《业务闭环图谱》的 owner传播/状态同步/级联点，核其是否都进了《联动点清单》N——图谱里有依赖、N 却被报低/报零（声称"无跨模块依赖"）的，记为审计发现**；顺带对照《扇出计划表》行构成与 §6 公式（大面×组数/小面各1/命中专项各1）及合并理由——明显少行、借口合并的记为审计发现（核对即可,非证据闸）。
审计**结构化输出**（逐项 达标/不达标 + 反例）；**不达标必须回炉重写，禁止"记一笔就过"**。
> 诚实边界：审计由主进程派出、结论也由主进程采纳——提升质量但最终看执行方诚实（纯 prompt 天花板）。环境不支持子代理时，主循环对照本清单亲自逐项复核（**判"不支持子代理"须在《进度总览》记判定依据**：试派失败的报错原文 / 工具列表确无子代理类工具——比照 §6 同款留痕，防"自评全覆盖=已复核"的自我宽免）。

## 9. 产出结构
产出落在被测项目工作目录的 `第二步-编写测试用例/`（或 `./test-cases/`，**不写死绝对路径**）：用例规范说明 + 按测试面分目录的用例文件 + README(各方向数量/优先级统计) + 覆盖矩阵与追溯(+官方记录单映射表)。完整结构见 `references/output-structure.md`。
> **隶属统一根（供第0步总管扫描续跑）**：本步产出隶属测试三步走统一根 `测试产出/`（见 step0 §6），完成信号(§8 面×维度矩阵/回链分母/回环覆盖**/跨模块联动 N/N/§8.5 对抗审计报告已产出**)由第0步总管 `testing-step0-intake` 扫描判定续跑；默认子目录 `第二步-编写测试用例/`。**仍可用别名根/子目录（如 `./test-cases/`，原选项保留）**，但用别名时**必须在《进度总览.md》登记真实路径**，否则总管扫不到。
> **换项目时勿照搬上一项目的"用例规范/覆盖矩阵/模块码/校验弹药/状态机"当模板**——它们是上个项目的"内容(轴三)"，会写死成错误结论。一律从**本项目第①步分母重新派生**；起草时用占位符 `<模块码>/<校验弹药>/<状态机>/<错误文案>`，再用第①步发现填充。

## 10. 对第③步（执行测试）的约束（防偏）
本步的**面/维度/TC编号/覆盖矩阵就是第③步执行与报告的骨架**。第③步及其 skill 必须对照本步用例集执行（每条 TC 有执行结果、1:1 对应），不得另立体系而偏离；⚠用例复现即记缺陷；写库/性能用例在③授权执行；报告可对齐官方《测试记录单/报告》格式。
- **★反哺回路（双向，非只单向下发）**：骨架由 ②→③ 单向下发，但**漏写由 ③→② 回流增补**。第③步执行时对照 `browser_snapshot` 穷举可操作元素，会撞见"页面上存在、本步用例集却漏写"的控件/交互，登记进第③步《_第②步漏项反哺清单.md》——**本步应把该清单当作下一轮迭代/交付前增补的直接输入**：按 `case-conventions` 把这些漏项补成正式 TC、回填覆盖矩阵，使"第②步可能漏写"的回路被第③步发现、再由第②步闭合。这也呼应 §6"多轮迭代"——第③步现场发现是本步再迭代一轮的来源之一。

## 11. 反模式（务必避免）
❌ 只写 CRUD 正向，漏边界/状态机/越权/并发/计算正确性/回读 → 不叫"全覆盖"。
❌ **对存在富交互的页面，只穷举"点得动的按钮/填得了的输入框"，漏掉"拖/悬/滑/拽/长按/双击/右键/快捷键/弹窗内交互"这类可操作但非简单点击的交互** → 第③步无对应用例可执行 = 结构性漏测（可视化/媒体/上传类页面若存在此类交互最易中招；本质把"可操作"误等同于"可点击/可输入"）。**前置条件：仅当页面确实存在该类富交互时才算结构性漏测；若页面只点击/输入/选择、无此类交互，整体豁免、不构成反模式（见 §3 命中即必测）。**
❌ 预期写"应该正常/功能可用" → 执行者无法判定，等于没写。
❌ 写操作只验 200/toast、不回读 → 测不出"是否真入库"。
❌ 把文档声称当事实、不做实测对照；或把文档符合性拆散不独立成面。
❌ 把"第二轮才补的维度"当成"项目特有"而不进通用考量 → 换项目漏掉并发/计算/会话/兼容等普适必测项。
❌ 一次成型、不分轮不交叉补充；用"每个/全部"却无矩阵分母 → 覆盖率空转。
❌ 在本步真去执行写库/压测 → 越界（属第③步、需授权）。
