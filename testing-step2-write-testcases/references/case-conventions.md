# 用例规范：模板 / ID / 优先级 / 追溯

> 配合 SKILL.md 第 2 节。先把这套规范落成一个"00-用例编写规范"文件放进产出目录，并在批量编写(含子代理扇出)时要求严格套用，保证跨文件/跨 agent 一致。

## 1. 用例字段模板（每条统一）
```
### TC-<类>-<模块>-<NNN> 简明标题
- 优先级: P0 / P1 / P2
- 类型: 功能正向 | 输入边界 | 状态机(合法/非法流转) | 状态机回环(环路各走通一遍) | 越权/数据隔离 | 跨模块闭环 | 跨模块联动(改A→验B/C/D→回退) | 文档符合性 | 请求异常 | 计算正确性 | 并发 | 展示鲁棒性 | 写后回读 | 安全 | 可用性/UI | 兼容性 …
- 前置条件: 账号(角色)/所需数据/入口URL/环境
- 测试步骤: 1) … 2) …（UI 写"点哪个/填什么"；接口写 `METHOD 全路径` + 关键 body/headers）
- 预期结果: 含【状态码 / 响应信封 / 错误文案原文(带引号) / DB 写入或回读结果 / UI·toast·跳转 原文】
- 判定依据(追溯): 第①步文档§x / 代码 file:line / API实测 / 文档出处
- 数据清理(写库/删除/上传类用例必填，其余写"无写入"): 造数用 `<PREFIX>` 占位前缀(见 §6)、写后回读、用完按 子→父 顺序清理、回基线核对——**把清理方案提为独立字段、别只塞进备注**(铁律核心，散在备注易被执行者漏看)
- 备注: 已知缺陷预期(标 ⚠预期暴露缺陷)/需显式授权的写操作(留③·自清理·唯一前缀)/数据隔离对照/交叉引用相关 TC
```

## 2. ID 规则
`TC-<类>-<模块>-<三位序号>`。
- **类码**：按测试面/维度取一组稳定缩写（示例，按项目语言可调）。
  - 默认常用：`UI / API / REQ(前后端通讯) / DOC(文档符合性) / AUTH(权限会话) / SEC(安全) / PERSIST(持久化) / BIZ(计算正确性) / CONC(并发) / ROBUST(鲁棒性) / RECOV(恢复) / COMPAT …`。
  - **条件触发面命中才用**：`MEDIA(媒体/RTC) / EXPORT(导出) / IMPORT(批量导入) / RT(实时协同/WS) …`——未命中对应形态/特性时不引入。
- **模块码**：取项目模块短名（由第①步模块清单来，**项目特有**）。
- 序号在"类+模块"内递增；跨文件保证不撞号（同类多文件时分配号段）。
- 好处：可按类/模块聚合、去重、与覆盖矩阵对账、被第③步执行引用。

## 3. 优先级定义
- **P0**：核心主流程冒烟 + 安全/越权 + 数据正确性(写入/级联/隔离/计算) + 阻断后续的功能。
- **P1**：主要功能分支、关键边界、状态机非法流转、重要文档符合性、并发/恢复要点。
- **P2**：次要/边角文案、UI 细节、低风险边界、提示态、兼容性细节。

## 4. 预期结果怎么写才"可断言"
执行者要能仅凭预期判定通过/失败。务必包含可机器或肉眼核对的锚点：
- HTTP 状态码 + 响应信封形态（成功/业务错误/校验错误 各自结构）。
- **错误文案抄原文并带引号**（来自第①步 API 基线）。
- **DB 效果**：写入了什么行/字段、级联影响、**回读如何确认**（见 pitfalls「写后回读」）。
- UI 表现：toast 原文、跳转目标、状态徽章文案/颜色、列表是否出现该行。
- 反例（禁止）："应该正常"、"功能可用"、"显示正确"——无法判定。

## 5. 追溯（判定依据）
每条回指证据来源，使用例可复核、不沦为臆测：
- `第①步文档 §x`（模块详解/闭环/数据模型/权限/文档对照）。
- `代码 file:line`（来自第①步代码级穷察基线 A/B/C/D）。
- `API实测`（第①步数据全景/权限矩阵实测）。
- `文档出处`（需求文档条款，用于文档符合性用例）。
- **追溯要指到具体位置，别写粗指针**。范例（占位符按本项目替换）：`判定依据: 第①步《<模块>详解》§状态机 + 代码 <api路由文件>:<行> + 数据全景与权限矩阵实测.md §<节>`——抽检时要能据此回到第①步具体文件/行，不能只写"见第①步文档"。
- **前置条件应复用第①步"测试就绪补全"**：账号凭据、可复现造数 payload、统计口径公式、schema 精确边界 —— 用例前置直接引用它（指针或贴值），让执行者不必回查。

## 6. 标注约定
- **⚠预期暴露缺陷**：对应第①步发现的已知问题（无鉴权/越权链/脏数据/声称未实现）。其"通过"=复现缺陷，不可记系统合格。
- **需显式授权(第③步执行)**：所有写库/删除/上传/重置/批量/性能 用例——本步只定义，执行时用"造数→自清理"可逆往返 + 唯一前缀 + 隔离环境。
- **★造数前缀写成 `<PREFIX>` 占位符、别把字面量写死**：用例文里(label_config、造数 name/编号、断言文本、清理 SQL)凡涉前缀一律用 `<PREFIX>` 占位，并在文件头注明"执行时由第③步替换为**实例级唯一前缀**(如 `QA_<模块>_<时间戳/uuid>`)"。**别写死成具体串**(如 `QA小测0609`)——否则第③步换前缀执行需全局心算替换、易错漏，且固定前缀二次重跑会撞唯一约束。**含中文的前缀额外提醒**：第③步经 docker/shell 传中文易编码损坏致回基线假阴，宜配 ASCII 安全标记(见 step3 pitfalls)——故 `<PREFIX>` 最好预留一个 ASCII 子串位。
- **pre-existing / added-after**：做覆盖映射(尤其对照官方记录单)时标明该用例是原本就有还是事后补——诚实性要求（见 pitfalls「诚实纪律」）。

## 7. 完整范例用例（领域中性 · 后台管理通用 — 示范"深度与可断言性"，不是抄内容）
> 下面这条用"后台-用户管理-编辑用户"这类**几乎所有管理后台都有**的写操作，示范一条**达标用例**长什么样：预期可断言到能机器核对、追溯到代码、写后回读、含易漏的"未传字段保持"与"错误分支零脏写"、自带数据清理。**学的是这个深度与结构，不是抄它的字段/文案**——换项目按你自己的领域、实体、错误文案原文替换。

```
### TC-API-USER-014 编辑用户·部分字段更新且未传字段保持（PATCH 语义 + 写后回读）
- 优先级: P1
- 类型: 功能正向 + 写后回读
- 前置条件: admin 登录取 token；已造一条 QA 用户 <PREFIX>_user_01（id 由前置造数步骤返回）
- 测试步骤:
  1) PATCH /api/v1/users/{id}  body {"phone":"13900000000"}（只传 phone，不传 name/role/status）
  2) GET /api/v1/users/{id} 回读
  3) 直查 DB: SELECT name,phone,role,status,updated_at FROM users WHERE id={id}
- 预期结果:
  - 步1 → 200，信封 {code:0,data:{...}}，data.phone="13900000000"
  - 步2/步3 → phone 已更新；**name/role/status 三个未传字段保持原值不变（验 exclude_unset，非被清空成 null）**；updated_at 已刷新
  - 边界对照：phone 传非法 "abc" → 422，错误文案原文 "手机号格式不正确"（抄自①步 API 基线）；**DB 无任何变更（错误分支零脏写，回读确认）**
- 判定依据(追溯): 第①步《用户管理详解》§编辑接口 + 代码 app/api/v1/users.py:88(PATCH handler) + schema app/schemas/user.py:21(UserUpdate, exclude_unset)
- 数据清理(写库): 仅改 QA 用户、不碰真实用户；测后按精确 id 删除 <PREFIX>_user_01 并回读=404、users 计数回基线、前缀残留扫描=0
- 备注: added-after。同模块同质变体（改 email/改 status）可共享造数前置、各自断言；非法值/越权改他人/超长 name 等另立条。
```
> 对照看它**为什么达标**：① 预期里每一句都能机器核对（状态码 / 错误文案原文带引号 / DB 回读 SQL / 具体字段值），没有"应该正常"这种空话；② 覆盖了"未传字段保持"这个 PATCH 最易漏的点；③ 错误分支也回读验"零脏写"；④ 追溯同时给了①步文档§与代码 file:line；⑤ 写库用例自带清理+回基线。**你的用例每条都该达到这个可断言/可追溯/可清理的标准。**

## 8. 跨模块联动范例（改A→验B/C/D→回退 · 领域中性）
> 联动用例（SKILL §6「跨模块联动专写阶段」）最易写成"改A→打开B看一眼没报错→回退"的空壳。下面这条示范一条**达标的真联动**：改的是 A(上游权限)、却去 B/C/D 三处**独立下游**各验一遍，**每段都有机器可核对锚点**，且有回退段回读确认下游随上游复原。**学结构与断言硬度，不是抄内容**——换项目按你自己的依赖链(owner传播/状态同步/级联)替换。

```
### TC-LINK-PERM-007 收回角色权限后·下游入口/接口/数据范围同步失效（改A→验B/C/D→回退）
- 优先级: P0
- 类型: 跨模块联动(改A→验B/C/D→回退)
- 前置条件: admin 登录取 token；已造 QA 操作员 <PREFIX>_op_01（角色含"导出报表"权限）、其名下 1 条 QA 报表
- 测试步骤:
  1) 改A：admin PATCH /api/v1/roles/{opRoleId} 收回"导出报表"权限
  2) 验B（菜单级）：以 <PREFIX>_op_01 登录，进工作台 → 看"导出"入口是否消失
  3) 验C（接口级）：该操作员带 token 直打 POST /api/v1/reports/{id}/export
  4) 验D（数据范围）：GET /api/v1/reports 列表，看是否仍只见自己范围
  5) 回退：admin 重新授予该权限，重复步2/3 确认恢复
- 预期结果:
  - 步1 → 200，**回读 role_permissions 表已无该权限行**
  - 步2 → 前端"导出"按钮/菜单不可见（DOM 断言：该元素不存在/disabled）
  - 步3 → **后端 403，错误文案原文 "无权限"**（不能只前端隐藏、后端仍 200 放行 = 越权缺陷）
  - 步4 → 仍只返回该操作员数据范围内报表（隔离未被连带破坏）
  - 步5 回退后 → 步2 入口重现、步3 改判 200 导出成功（**下游随上游变更复原**）
- 判定依据(追溯): 第①步《业务闭环图谱》§owner传播/权限同步 + 代码 <角色接口文件>:<行>(改权限) + <鉴权中间件文件>:<行>(下游放行)
- 数据清理(写库): 测后撤销 QA 角色改动回基线、删 <PREFIX>_op_01 及其报表并回读=404、前缀残留扫描=0
- 备注: 跨模块联动核心样例——"验B/C/D"每段都有机器可核对锚点(状态码+文案原文/入口可见性/数据范围)，非"看一眼没报错"；回退段回读确认下游复原。
```
> 对照看它**为什么是"真联动"而非空壳**：① 改的是 A(权限)、却去 B(前端入口)/C(后端接口)/D(数据范围) 三处**互相独立的下游**各验一遍，真跨了模块；② "验B/C/D"每段都有**可断言锚点**(403+文案原文 / 入口可见性 / 数据范围)，不是"打开看一眼"；③ 有**回退恢复段**、回读确认下游随上游复原；④ 追溯回链第①步《业务闭环图谱》的具体传播链。**空壳反例（禁止）**：`改A → 打开B页面没报错 → 回退`——无任何下游断言锚点、无回退回读，过不了 SKILL §8 联动闸门与 §8.5 审计。
