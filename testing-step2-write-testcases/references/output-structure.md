> 配合 SKILL.md 第 9 节。产出落在被测项目工作目录的「第二步-编写测试用例/」下（或 `./test-cases/`，**不写死绝对路径**）。按"测试面"组织顶层，文档符合性独立成面。

```
第二步-编写测试用例/
  00-用例编写规范与字段说明.md      ← 模板/ID规则/优先级/追溯/标注约定（批量编写前先定，供各子代理(subagent)共享）
  README.md                          ← 用例体系总览 + 各方向数量统计 + 优先级分布 + 与①③衔接
  <面1>-UI功能与功能闭环/             ← 可按模块组分多个文件
  <面2>-后端接口可用性与超出数据/
  <面3>-前后端通讯（请求异常·前端态vs真落库）/
  <面4>-文档符合性/                   ← ★独立面：声称核验 + 官方测试记录单/报告 1:1 映射表
  <面5>-权限越权与认证会话/
  <面6>-安全与配置/
  <面7>-数据持久化与端到端深度/
  跨模块联动与变更影响/              ← ★独立目录(比照面4不拆散)：改A→验B/C/D→回退恢复；在基础用例写完后专写，分母=第①步闭环图谱《联动点清单 N》(SKILL §6)
  [按项目特性条件触发的专项目录]      ← 媒体/画布/导出/存储多链路/计算正确性/并发/鲁棒性/恢复/兼容 等(命中才建)
  附-用例覆盖矩阵与追溯.md            ← 面×维度矩阵 + 回链第①步分母(API/页面/状态机/字段/文档/无鉴权) + 官方记录单覆盖行
  附-扇出计划表.md                    ← ★开派子代理前落盘：逐行"序号|负责块(面×模块组)|用例ID段|预期落盘文件"（SKILL §6）
  附-联动点清单.md                    ← ★②开头从第①步《业务闭环图谱》锁定：N 条跨模块依赖(改A→影响B/C/D)，作联动覆盖分母（SKILL §6/§8）
```

## 命名与衔接要点
- **文档符合性必须独立成面/目录**（oracle 不同：对账"声称"而非测行为），不可拆散进各模块。
- **跨模块联动必须独立成目录**（同理不拆散进各模块文件）：联动用例="改A→验B/C/D→回退恢复"，**在基础用例写完后由持全局上下文者专写**（SKILL §6「跨模块联动专写阶段」）；《联动点清单 N》是其分母（②开头从第①步闭环图谱锁定、落盘）。
- **覆盖矩阵是闸门的落地**：行=模块/资源，列=维度（或反之），格子填覆盖来源 TC 编号；空格=待补。另起一节回链第①步分母逐项给分子。
- 有官方测试记录单/报告 → 在面4 放**逐条覆盖映射表**(官方ID→TC→状态)，并在覆盖矩阵加"官方记录单 X/X"行。
- 顶层用稳定序号/语义名，避免重命名破坏内部相对链接与 TC 交叉引用。
- 各面/块的用例文件**末尾附该子代理《格子清点 已写X/X》**（SKILL §6 任务书⑤）；《扇出计划表》《格子清点》《任务书》是**过程产物**（供 §8.5 审计追溯；**非 step0 进度完成信号、第③步不当成 TC 执行**），按上面目录树落盘、任务书全文与计划表同目录留存（如 `附-子代理任务书/`）。
- **本步是第③步的骨架**：面/维度/TC编号/覆盖矩阵 → 第③步逐条执行并记结果、1:1 对应；README 末尾写明"留待第③步"项（写库执行/性能实测/需外设手动触发）。

## README 应含
- 用例总量 + 各方向(面)数量 + 优先级分布(P0/P1/P2) + ⚠预期暴露缺陷条数。
- **跨模块联动覆盖 M/M**（《联动点清单》N 条 → 各有联动用例的条数）+ **产出后对抗审计结论**（§8.5：已派/达标项/回炉项）——这两行必填，防"做完逐模块+文档就当完成"。
- 一句话方法论（面×维度矩阵 + 多轮迭代 + 代码+浏览器为准 + 写后回读）。
- 输入来源（第①步哪些文档）与覆盖矩阵指针。
- 与官方记录单(若有)的覆盖结论 + 排除项去向(留③)。
