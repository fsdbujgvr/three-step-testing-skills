# 编写测试用例的避坑与方法（持久化/文档映射/诚实/编排/多轮）

> 配合 SKILL.md。这些是真实编用例时最容易绊倒、且与具体项目无关的点。

## 一、数据持久化"写后回读"（最易漏、最致命）
只把"DB 应写入"写进预期、却不回读确认 = 默认"200/toast = 真入库"（错）。每个写操作配显式回读步骤：
- **POST**→重新 GET 资源/列表：新行存在 + 输入字段落库 + **服务端默认值**(默认状态/时间戳/uuid)正确。
- **PATCH**→回读改动持久化；验"未传字段不变 / 显式 null 是否清空"(框架 exclude_unset 语义)。
- **DELETE**→回读目标 404 + 列表计数 -1 + **级联子数据回读消失**；外键置 NULL 型回读引用方置空。
- **上传**→回读资源行 + **取静态文件字节确认文件真在盘** + 刷新页面仍在(非仅前端预览)。
- **状态流转**→回读 status+时间戳 + **回读连带副作用实体**(如"开始"动作把上游单据置进行中)。
- **跨会话**：换 token/刷新/重新登录后数据仍在 = 真 DB；只在当前前端可见 = 前端态。
- **错误分支也回读**：报错后确认无半截脏写(回滚干净)。
- **辨前端态 vs 真落库**：AI/建议类(返回≠已保存，需再保存)、勾选/编辑类(是否真发请求且回读生效)。
> 落地：单列一类"数据持久化与端到端深度验证"，对每个写端点配"写后回读"用例 + 复杂闭环逐步回读；"回读"应是高频动词。

## 二、跟随需求文档做"文档符合性"（含官方记录单映射）
- 三类映射：声称/规格→正向核验；操作步骤→走查；声称却未实现→负向核验。
- **有官方测试记录单/测试报告 → 必取来逐条 1:1 映射**，产出覆盖表(官方ID→你的TC→状态：✅已覆盖/🟡他处已测/➕缺口补/⛔排除)。官方"纯按文档设计"用例与你的文档符合性面**同源、常约 80% 重叠**。
- **逐条核验防真缺口**：即便大面积重叠，仍易漏"使用限制/展示鲁棒性"类软条款——对官方 ID 逐条过是兜底。
- **你的 ⊋ 官方**：你的代码+浏览器用例能发现 架构失真/声称未实现/硬编码/未映射枚举/安全·越权，官方纯文档覆盖不到——保留为增量。
- **排除项显式标 + 给去向**（代码扫描/压测被忽略时，性能指标条目仍收录留③）；多份官方文档(报告功能性表常=记录单同套)去重只映射一次。

## 三、诚实性纪律（覆盖性结论别夸大）
- 覆盖映射给每条标 `pre-existing / added-after`；**绝不把"补完后 100%"说成"原本就 100%"**。
- 回答"是否已包含"用 **文件时间戳 / grep 实证**，而非凭记忆（记忆易错——宁可现场 grep 核对）。
- 被追问时如实拆分：直接已有 / 他处已测后补成专条 / 真缺口后补。不粉饰。

## 四、编排（子代理(subagent)扇出，规模化编写）
- **按 测试面×模块组 扇出**：大面(UI/后端接口/持久化端到端)按**第①步模块清单**分组拆多个 agent（默认一模块一组,不重新发明分组单位），小面(请求异常/权限/安全/各专项)各 1 个；开派前按 SKILL §6 落盘《扇出计划表》、照表派遣。
- **先写共享"用例规范"文件**，agent prompt 要求严格套用，保证一致。
- **返回契约以 SKILL §6 任务书为准**：子代理直接落盘至《扇出计划表》指定文件、返回只报"路径+用例计数+清点摘要"；**子代理无法写盘的环境才回退**"返回纯 markdown(从首个 `#` 截取、去寒暄)、主循环按 label 拆分落盘"（Claude Code Workflow 取 `.result`；OpenCode/Codex 等按其子代理返回结构类推）；`grep -c "^### TC-"` 统计数量与优先级分布喂 README。
- **集成产物(README/覆盖矩阵/官方映射表)主循环亲写**，保跨文件一致与追溯。
- 取接口/解析 JSON 用 `node`(全局fetch)/`curl`，别用易乱码的 PowerShell 或常被劫持的 python（沿用第①步工具经验）。

## 五、多轮迭代与交叉补充（别一次成型）
- 第①轮按"面×维度矩阵"铺基础；第②轮对照覆盖矩阵+第①步分母补**薄/缺**格子；第③轮**交叉补充**——一类发现催生另一类。
- 类间相互喂(举例)：持久化↔并发、异常数据↔持久化、安全↔原子性、错误文案↔文档符合、媒体↔第三方。
- 每轮收尾三问：①矩阵还有空格？②已覆盖格只写了正向、缺边界/异常/越权/回读？③本项目独有特性各有专项？答得出"还缺X"就再来一轮。

## 六、通用认知陷阱
- **前端校验 ≠ 系统校验**：前端 rules 是 UX，真约束在后端；两端都要测、说清一端缺失的后果。
- **未映射枚举/无兜底**：字典外取值→前端可能白屏/报错；凡按字典渲染处都构造字典外取值并验崩不崩。
- **硬编码常量被当数据**：型号/时间/水印——换数据看是否变化即可判定，应写成"恒定不变"用例。
- **计算正确性 ≠ 有显示**：统计/派生值要喂已知输入断言公式结果，别只验"有数字"；注意脏数据致异常值。
- **mock 端点 ≠ 真实数据**：有的看板/统计写死假数据，换调用/换数据证伪。
- **默认分页可能静默截断**：真实总数 > 默认页大小时，UI 是否只显一页而不提示——专门验。
- **写库/性能是第③步**：本步只定义、标授权；在本步真去造数/压测 = 越界。
