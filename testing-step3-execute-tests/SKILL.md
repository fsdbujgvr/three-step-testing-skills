# 执行测试（测试三步走 · 第三步）

## 唯一目标
对第②步《用例集》**逐条真实执行 + 取证**，产出对齐官方《测试记录单/报告》、**图文并茂、可视化复核、诚实不夸大**的测试报告 + 缺陷台账 + 失败用例集。
**判定基准恒为「真实证据」**：浏览器真操作 / 接口真发请求 / DB 写后回读 / 代码 file:line——**"读代码推断它应该对" 不等于 "执行了"**。
> 未过第 11 节「完成度闸门」（含执行后审计：真测率 + 截图覆盖率 + 反例核查）前，**不算完成**。

## 0. 前置：必须有第②步《用例集》
输入是第②步产出（`testing-step2-write-testcases` + `testing-step2-doc-conformance`）：面×维度矩阵 / 每条 TC(ID·步骤·可断言预期·判定依据) / 官方记录单映射 / 面4 文档符合性《原子级核实表》 / ⚠预期暴露缺陷用例 / 需授权写库与性能项。
- **没有第②步用例集 → 先做第②步**。第②步的 TC 编号 + 面/维度矩阵就是本步执行与报告的**骨架与分母**——逐条 1:1 执行、不另立体系。**（条件触发：第②步若另产出独立《跨模块联动与变更影响》目录/《联动点清单 N》，它同为本步执行分母的一类——独立成执行单元、单独统计联动 N/N，既不并入也不漏出面/维度矩阵；第②步无联动产出则本项 N/A。防"分母只认面/维度矩阵、整个联动套件被静默漏跑"。）**
- 执行中若需**第①步《项目理解文档》**的真实数据基线 / owner 传播 / 无鉴权与越权清单作证据(如越权用例要知道哪些端点本应鉴权、统计用例要知道真实数据规模)，**可回查第①步产出**(`testing-step1-understand-project` 的结果)。

### 0.1 ★被测系统"形态"决定哪些面/闸门适用（form factor·step1/2/2.1/3 统一引用）
执行前先确认被测系统的**形态**（应已在第0步《项目信息确认卡》/第1步识别并经《进度总览》传递下来；若下游产出未带，本步开跑前先确认一句）。形态决定下面一组"前端相关闸门"是否适用：
- **① 标准 Web 应用**（默认，有前端 + 源码 + 账号）：本 skill 全部闸门按原文适用。
- **② 纯 API / 无前端**（仅 REST/gRPC + 契约，凭据为 token/API-key/mTLS）：**无面向用户前端**，下列"前端相关组"整组标 **N/A**，改以 API 形态闸门替代（见 §1 形态适用性、§11）。
- **③ 薄 UI 后端 / 交易中台**（价值在后端业务规则，UI 是薄壳）：后端面按原文严格；UI 面按"实际控件"裁剪，反哺闭环等 UI 专项对薄 UI 降为推荐（见 §1 表尾、§11）。
- **④ 实时协作**（标准应用叠加 WebSocket/SSE 实时通道）：在 ①/③ 基础上**额外**挂载实时通道真测（通道2.5）+ 协同写双 oracle（§3），条件触发。
- **⑤ 含富交互**（canvas/媒体/可视化）：在 ① 基础上，富交互项"命中即必测"（§1 表尾、§11）。
> **"前端相关组"（仅"有面向用户的前端"时适用，否则整组 N/A）**：面1 UI、面4"必须浏览器截图、不可源码佐证替代"、交互方式覆盖率、`browser_snapshot` 可操作元素穷举、角色级"后端 200 不算通过"。**纯 API/无前端形态下整组标 N/A**，"眼见为实"改判据为"**真实请求-响应报文 + schema/契约校验 + 代码 file:line**"。这是**条件化、不是弱化**——只要被测系统有前端，这组闸门一丝不减地继续生效。

## 1. ★执行强度分级（本步最关键的防夸大约束：不同面，不同强制级别）
**不是所有面都能"读代码推断"过关。** 按面强制不同的真实执行级别——这是本步与"对着代码写'通过'"的根本区别：

| 面 | 强制级别 | 含义 |
|---|---|---|
| **后端接口与数据 / 权限越权 / 安全 / 持久化（API 类）** | **必须完全真跑一遍** | 每条用例**接口真发请求**(node/curl 带 token)拿真实状态码+响应体 + **写操作 DB 写后回读**。**源码佐证只能作补充说明，绝不替代真跑**——API 是可程序化的，没有"跑不了"的借口。 |
| **文档符合性（面4，独立面）** | **三重最高标准（不打折、不受"只测核心"影响）：浏览器图文并茂 + 逐条审计 + API/函数真检测** | 每条文档声称**一条一条在真实网页看过去、操作过去**，**每条配真实网页截图 + 网页位置说明**（判有→截图显功能在；判无→截图显无入口；判部分→截图显打折扣）；**而且对每条声称做 API/函数调用真实检测**（背后接口真发请求 / 前端函数复刻执行，与截图互证）；**重点审计必须逐条复核完成、一条一条看过去**（见 §9）。**不接受源码佐证替代浏览器实证。面4 永远全量最高标准，不因省 token 缩水。** |
| **UI 功能与闭环（面1 等纯浏览器面）** | **能测必测 + 尽量 API/函数化 + 运行前先问范围** | **铁律：凡能真测的 UI 都必须真测一遍，不得用源码佐证冒充"能测但没测"。** 运行前用 AskUserQuestion 问范围：**完整全跑** 还是 **只测核心(省 token)**（**★无交互/subagent，或范围已由上游预答/书面授权时**：按上游已确认的范围执行 + 在产出登记"范围据上游预答/授权"，**不卡死、也不借此默认缩水**——上游没说就按"全跑"从严，不能自行缩成核心）。**选"只测核心"时必须同时与用户确认/回填一份"核心清单"，不得由执行者私自把核心缩成三两条**——核心判据：① 第②步矩阵里标 P0/核心的用例；② 主业务流上的写操作与状态流转；③ 每个角色的登录入口及到达其目标界面的路径。核心清单进 §11 闸门可复核。无论选哪种：**① 核心交互必须调起浏览器真操作 + 截图保存 + 做到非常完善（图文并茂）**——核心在任何范围下都不省、不退化；**② 非核心：能用背后 API/请求函数 或 前端纯函数复刻验证的，尽量用 API/函数调用真实检测（不只看代码）**——**只有连 API/函数都无法验证的纯前端表现，才退到源码佐证**。真测的配真实证据（截图/请求-响应/函数输出），没真测的诚实标"源码佐证/阻塞"——绝不标✅冒充。 |
| **并发/幂等正确性（功能正确性维度，非性能）** | **必须真跑（程序化并发发请求 + DB 回读最终态）** | 防超卖/重复扣款/支付回调幂等/丢失更新这类是**功能正确性缺陷、不是性能项**——**用少量并发请求（如 `Promise.all` 同时打同一资源 / 幂等键重放）真发 + 并发后 DB 回读最终一致/守恒断言**真验。**绝不归入"留③性能/阻塞/源码佐证"**：只有"N 用户吞吐/时延"才属性能留③；"少量并发验最终一致"属功能必测。配方见 `references/execution-channels.md` 通道2.5。 |

> **★形态适用性（呼应 §0.1）**：上表"前端相关组"（**面4"必须浏览器截图、不可源码佐证替代"** + **面1 UI**）仅在"**有面向用户的前端**"时按原文适用。**纯 API/无前端形态下**：面4 行的"眼见为实"改为"**每条声称配真实请求-响应报文 + schema/契约校验 + 端点检索证据（代码 file:line）**"，浏览器截图项标 N/A；面1 整面标 N/A；但 **"API 类完全真跑、源码佐证=0"作为该形态的主闸门保留并强化**。**有前端的形态**这组闸门一丝不减。
>
> **★富交互/canvas 适用性（条件触发，呼应 §0.1 形态⑤）**：表格"UI 功能与闭环"里的**非点击交互穷举（canvas/拖拽/滑块/悬停/右键/滚轮/canvas 绘制）= 条件触发**——**仅当第①步识别出控件确实支持该交互时全量适用、命中即必测**（命中时的严格度一丝不减，page.mouse 真实轨迹真测配方见 §1 末铁律与 `references/execution-channels.md` 通道1.6）；**表单-列表类系统按实际控件裁剪，本页/本形态根本无此类交互时一句话整体豁免、不逐控件标不适用**。

> 为什么分级：API 可程序化、必须穷尽真跑；面4 是对外承诺的符合性、必须眼见为实的截图+接口检测；UI 量大且浏览器串行慢、有卡死风险，故让用户先定范围而非默默缩水。**任何缩水都必须显式、经用户同意或诚实标注，不得静默用源码佐证填充。**
>
> **★ 全局铁律「能测必测」**：凡是**能程序化真测的用例（接口可发请求 / 前端纯函数可复刻执行 / 浏览器可触发）都必须真测一遍**——源码佐证**只允许用于运行期物理上无法触发的项**（外部依赖不可达、无真实外设、并发压测、破坏性全表、schema 不可构造的前置、纯前端且无任何 API/函数可验），**绝不用源码佐证冒充"能测却偷懒没测"**。判断顺序永远是：**能 API/函数测 → 真发请求/真调函数；必须看渲染交互 → 浏览器真操作+截图；二者都不行且物理受限 → 标阻塞+进专项说明（见 §10、§11）；只有纯前端无 API 无函数可验 → 才源码佐证。**
>
> **★ 角色级判定硬规则（防"后端 200 掩盖前端不可用"）**：凡按角色判定"某角色能否用某功能"，**后端接口 200 不能单独支撑该角色"通过"——必须先在浏览器用该角色真实登录、验证它能进入前端并到达目标界面**。实战踩过：某角色后端可登录(API 全 200)、前端却被路由守卫踢回登录页/白屏（账号实质不可用）；遇此记为该角色"前端不可用"缺陷，绝不因 API 200 判通过。通道1.5 的"API 等价验证后端效果"只代表后端，不代表该角色前端可用。**（形态边界：本规则仅当被测系统含面向用户的前端且按角色分权时适用；纯 API/服务间调用形态下，角色判定最终证据 = 该角色凭据下的真实响应 + 数据隔离回读，后端响应即为权威，不要求前端可达。）**
>
> **★ 可操作 UI 穷举真操作铁律（仅"有面向用户前端"形态适用；纯 API/无前端形态整组 N/A，见 §0.1）（UI 真测分母 = 第②步用例 ∪ 页面上每个可操作元素，不止用例）**：进入任一界面后，**对照 `browser_snapshot` 主动枚举页面上所有可操作元素/交互——按钮点击 / 表单输入 / 下拉选择 / 单选复选（这些是任何控件默认支持的交互，命中即必测）；以及 拖拽绘制 / 拖放排序 / 悬停展开(tooltip·浮层) / 滑块拖动 / 快捷键 / 右键菜单 / 滚轮 / 弹窗开关与弹窗内控件 / 二次确认 / 分页筛选排序 / tab 切换 等（这些**富交互项条件触发**：控件确实支持时命中即必测、严格度一丝不减；本页根本无此交互则整体豁免、不逐项标不适用）——逐个真实操作一遍并记录结果**。截图看到≠测过、源码有 handler≠测过。**第②步用例没写到的可操作元素，要做两件事、缺一不可**——因为第②步可能漏写，第③步对照 `browser_snapshot` 穷举时是发现并闭合这些漏写的最后一道关：**① 当场补测**（真操作+截图，记入本步执行记录）；**② 登记为"第②步漏项"反哺回去**——凡"页面上存在、却不在第②步用例集"的可操作元素/交互，**其中能测的**逐条写进 `_第②步漏项反哺清单.md`（控件/交互 + 所在页/角色 + 第③步怎么补测的(证据) + 建议补成的 TC 编号 + 追溯），使下次单独复用/交付第②步用例集时不再缺这一条；**其中环境/物理受限测不了的，改入 `_无法测试项说明/汇总.md` 并标"第②步亦漏写"、不入本清单**（本清单只收"能测且已补测"的漏写，两专项加总 == 穷举发现的"不在第②步"控件总数）。**只补测不反哺，第②步的漏会一直留着；只登记不补测，本步就漏测了——两件事都做**（清单格式见 `references/report-structure.md`，闸门见 §11）。**即使某交互背后有 API，只要它涉及交互渲染态/前端 handler 行为，就必须在浏览器真操作一遍**（API 等价验证只对"纯后端效果"计数，**不抵该控件"被真正操作过"**）。这是防"画布绘制/缩放平移这类非点击交互靠人事后指出才补"的根本约束——非点击交互的真实操作配方见 `references/execution-channels.md`。

## 2. 四条证据通道（执行手段）
1. **浏览器**（Playwright MCP）：真点按钮/填表单/开弹窗/切页/拖拽 → 截图 + 运行期 `evaluate`(DOM 文案 / `video.readyState` / 元素位置 / localStorage)。UI 与面4 的主通道。
2. **接口**（node 全局 fetch / curl）：登录取 token → 带 `Authorization: Bearer` 打端点；边界(空/超长/非法/最值±1)、越权三态(无token/各权级)真发请求看真实状态码+错误文案原文。API 类主通道。
3. **数据库写后回读**（区分"前端态 vs 真落库"，最易漏）：写操作后**直查 DB** 确认真入库/默认值/副作用/级联——**接口 200 / 前端 toast ≠ 真入库**。
4. **代码佐证**（grep/read file:line）：仅作**补充定位与解释**；运行期确实难触发的(并发 schema/底层时序)可代码佐证，但**必须显式标"源码佐证(未真跑)"**，且 API 类不得以此替代真跑。
> 证据分级：接口/DB 实测 > 浏览器实测 > 代码佐证。判"通过"必附证据，判"不通过"必附差异原文，环境受限标"阻塞+原因+源码佐证逻辑"。

## 3. 数据持久化"写后回读"铁律（承接第②步面7）
**为什么必须回读**：接口 200 / 前端 toast 只证明"请求被接受"，不证明"真落库"——可能是乐观更新、半成功、或只改了前端态；实战中曾抓到 GET 触发写副作用、PATCH 绕过状态机落脏数据，都是回读才发现。所以每个写操作后必做 DB 回读：POST→回读新行+服务端默认值；PATCH→回读改动持久化+未传字段不变(exclude_unset)；DELETE→回读404+列表-1+**级联子表回读消失**；上传→回读资源行+**取静态文件字节确认在盘**；状态流转→回读 status+时间戳+**连带副作用实体**。**错误分支也回读**确认无半截脏写。换会话/刷新确认是真 DB 而非前端态。
- **★协同写双 oracle（条件触发：实时协作形态，见 §0.1 形态④）**：写操作除 DB 回读外，**再开第二个在线会话**（第二浏览器上下文 或 第二 WS 连接），断言它在**限定时间内**收到对应变更帧 / UI 更新——验"写不仅落库、还按实时通道广播给了其他在线会话"。非实时形态此条 N/A。WS 真连真测配方见 `references/execution-channels.md` 通道2.5。

## 4. 带截图的 UI 实证（图文并茂 = 每条真跑的 UI 操作配专属截图 + 位置说明）
- **"图文并茂"的真实含义**：截图必须是**本用例、本页面、本操作态**的专属截图，旁边说明"操作在页面哪里、点了什么、结果可/不可"。**借用他页/他用例的截图冒充 = 造假**（典型反例：拿"用户注册页"的校验截图冒充"登录页"校验）。
- **业务流程贯穿截图法**：登录→主业务流一条龙走，每界面/关键操作截图，admin 端与各角色端分别走。
- 截图工程（必避坑）：文件名**禁含英文括号 `()`**(截断 markdown 链接，用 `-`)；截图默认落 cwd 根目录需移到证据目录并**即时校验落盘**；产出后跑断链校验脚本验 0 断链；react-konva/canvas 合成事件**不触发**绘制，真实拖拽需 browser_drag(易卡死，见 §7)。
- **面4 文档符合性**：是"图文并茂"要求最高的面——见 §1 表格，逐条声称配真实网页截图，不可源码佐证替代。
- **★瞬时 toast/提示的取证（易漏证）**：成功/失败 toast 常有 TTL、几秒就消失，"提交完再截图"必然截空 → 漏掉用例的关键可断言预期。**触发写操作前先布好捕获**（如先 `browser_evaluate` 注册 MutationObserver 盯 toast 容器、或 `browser_wait_for` 文本与点击并发），抓到原文再断言；**确实没抓到时**以"Modal 关闭+列表变化+DB 回读"为主证据判通过、把 toast 子断言单独标"未取证"，**不冒充"文案已核"**。配方见 `references/execution-channels.md` 通道1。
- **UI 同质重复用例的逐条覆盖追溯（合并不漏）**：UI 模块常有大量同质重复用例(同表单不同字段/同组件不同入口)。原则：**每条逐条列出 + 标覆盖来源；同一功能点的同质重复可共享同一截图但每条明确归属；不同功能点/状态/边界/异常变体必须各自覆盖、绝不合并漏掉**；用 `追溯行数==用例总数` 硬闸门防漏。"列出"≠"真跑"、"共享截图"≠"专属图"——覆盖来源要如实标。详见 `references/ui-coverage-traceability.md`。

## 5. ⚠预期暴露缺陷用例的执行
第②步标"⚠预期暴露缺陷"的用例（越权/无鉴权/脏数据/无兜底）：**必须真发请求/真操作复现该缺陷行为** = 用例通过(系统行为符合预测) + 登记缺陷台账。**绝不只读代码就断言"缺陷已复现"**——越权写、匿名删、篡改归属这类必须真发匿名/越权请求看真返回(典型反例：用 admin 带权请求冒充"匿名可删"的验证)。缺陷分级 P0/高/中/低，每条附 TC + 证据 + file:line + 影响。

## 6. 写库类用例的安全执行（守则）
- **优先 QA 造数、不碰真实实体**：`QA_<模块>_<时间戳或uuid>` **实例级唯一前缀**（防跨次重跑唯一键/业务唯一约束冲突）造数 + 写后回读 + 用完**按子→父顺序清理** + 回基线核对。**回基线一律按 QA 前缀/独立标记列精确计数，不用全表计数**（全表会被其他并行模块/他人写入扰动，产生假阳/假阴）；**name 为空/NULL 的造数行 LIKE 前缀匹配不到，须按已知主键 id 清理**（曾因此漏删残留）。**★中文前缀的致命假阴**：经 Windows 宿主 shell → `docker exec mysql -e "...LIKE 'QA小测%'"` 传中文参数会**编码损坏、匹配不到刚插入的行**(总数却照样 +1)，照此"按前缀回基线"会得到**假阴残留=0、漏清误判**——含中文的前缀务必**配 ASCII 子串(如 mrn/编号列带 `QA…0610`)或 HEX/精确主键 id 双查**核验，别只靠中文 LIKE(见 `references/pitfalls.md`)。**独占静默库**(本地副本、单租户)时可**额外叠加全表计数对账**作更强证据(此时无并行写扰动)，与前缀精确计数并用、非二选一。
- **★必须对真实业务实体写时：测前快照 → 测后还原 → 回读确认还原**：能 QA 造数就别动真实实体；若用例必须对真实记录做 PATCH/状态流转/绑定，**先 SELECT 快照原值，测后改回原值并回读确认已还原**——这类改动非 QA_ 前缀，清理脚本扫不到、回基线发现不了，不还原会留下被测试污染的真实业务数据。
- **★破坏性端点守则**：批量/级联删除/全表清空类端点，**只对完全自建的隔离 QA 链操作，严禁复用任何真实业务 ID**（实战中曾误复用真实主数据 ID，致其子资源被级联删除、不可恢复）。无法隔离的破坏体只验鉴权 403 对照、不执行破坏主体。
- 部署库默认护栏：写类需显式授权边界；性能/压测/真机/外部依赖项标"阻塞+环境受限"，诚实标注不假装执行。
- **★删前只读确认 + 写库串行（血泪教训）**：任何 DELETE/清理前先 `SELECT` 确认精确删除集合（**禁 collation 转换的盲目 NOT EXISTS 删除**——曾误删真实数据，靠备份才挽回）；**大规模写库改主控亲自串行**（逐个造数→写后回读→立即清），**不用多 agent 并行写库**（并行易留残留+清理时互相干扰/误删）；只读/鉴权类才并行。详见 `references/pitfalls.md`。

## 7. 编排：后端并行 + 浏览器串行（关键约束 + 卡死教训）
- **后端面(接口/DB/代码类)可扇出子代理(subagent)并行**(多子代理各跑一模块，互不抢资源；三家终端均称 subagent，支持 workflow/编排的环境用其更强并行)。
- **浏览器是单一有状态会话 → UI/面4 类必须串行**(两个 UI agent 同时操作会互相覆盖页面)。
- **★UI agent 卡死教训**：UI agent 在媒体/canvas/长等待上会 **MCP 层 hang**(agent 层超时拦不住)。对策：① 浏览器密集模块**主控亲自串行驱动**；② 用 agent 则**小粒度 bounded**(每≤30条+"单操作>15s 标阻塞跳过不 churn"+每5-8条增量落盘)；③ 卡死用 `TaskStop` 停 agent；④ **恢复卡死浏览器只 kill `chrome.exe`，绝不 kill `node`**(含 MCP 服务进程，误杀致 MCP 断连)。
- **后台 agent 防断连**：先把文件写完落盘(主交付物)、分批 Write、最后只返回极简摘要；agent 完成后 `ls`/`wc -l` 确认真落盘，没落盘就重跑。
- **面4 全量"三重"的批量编排（解决"全量不缩水"与"浏览器串行慢/卡死"的冲突）**：面4 几百条全量三重会与浏览器串行慢冲突。可行解——① **API/函数检测那一重可程序化并行**(不抢浏览器、先批量跑完)；② **浏览器截图那一重主控亲自串行或小粒度 bounded agent 分批**(每条数秒、卡死跳过)；③ **卡死的面4 条目标"阻塞-浏览器卡死"并进专项**(不算静默缩水)。即"面4 不缩水"指三重都要做、可拆到不同通道/节奏，不是必须一条条纯浏览器串行。

## 8. ★防夸大纪律（本步命脉，违反即作废）
执行记录里**每条用例的"执行方式"必须如实归入四类之一，不得混淆/美化**：
- **真测**：浏览器真操作 / 接口真发请求 / DB 写后回读——真跑过。
- **源码佐证**：只读代码 file:line 推断，**没真跑**——**必须显式标注，不得记成"已执行/✅通过"**（它是"代码看起来对"，不是"测过了"）。
- **阻塞**：环境受限未执行——标原因 + 源码佐证逻辑。
- **待补**：漏跑/未补。
铁律：
- **不把"源码佐证"写成"✅通过/已执行"**；判定列要让读者一眼看出这条到底真跑没有。
- **不把"共享/借用他页截图"写成"专属截图/图文并茂"**；共享要标"(共享 TC-X / 借自 Y 页)"，借错页面=造假。
- **不把"逐条列出了条目"说成"逐条执行了"**——列出 ≠ 真跑，覆盖来源是源码就如实说源码。
- **真测率、截图覆盖率用真实分子/分母报告**，不拿"条目数"冒充"真跑数"。

## 9. ★执行后对抗性审计（强制"审计是否完成"，防自我美化）
执行完成后**必须做一轮独立审计**（不是自我感觉良好地宣布完成）——尤其面4 文档符合性**重点审计必须逐条复核完成**：
- **分模块对抗核查**（派独立子代理(subagent)并行，每模块一个；支持 workflow/编排的环境用其更强并行）：逐条分类 真测/源码佐证/阻塞/待补；UI/面4 模块再分 专属截图/共享/无图；**每模块举 3-5 个反例**（被记✅却实为源码佐证、或涉 UI 操作却无专属截图、或借错图）——反例是防自我美化的核心。
- **面4 文档符合性逐条审计**：一条一条看过去 + 对照浏览器截图，确认每条声称都有真实网页截图佐证、判定与截图一致(截图与被测点对应，不是"合法态图"冒充"脏数据兜底")。
- **跨模块联动专项审计（条件触发：第②步有联动套件）**：把第②步《联动点清单 N》作**独立分母**（不并入 totalTC 拍平池），核对**联动 N/N 是否逐条真执行**；反例尤其抓——**整个联动目录被跳过没跑 / 只改A没真去下游 B/C/D 验证 / 回退段无回读确认复原 / 联动写成"改A打开B看一眼没报错"的空壳**；联动 N 被报低/报零（声称"无跨模块依赖"）的，回查第①步《业务闭环图谱》核其真伪。
- **交叉核对分母**：各类计数加总 = 用例总数(无错配)；真测率 = 真测/总数，截图覆盖率 = 专属图/UI总数——**用真实数字报告，戳破"全执行/图文并茂"的夸大**。
- 审计发现的缺口→如实回填执行记录(改标源码佐证/补截图/标阻塞)，**不留"✅"假象**。
- **★汇总计数必须客观化 + 隔离补测/复现轮（防自检自身失准）**：真测率/请求数/断言数等汇总指标**用脚本 grep 客观计数**(别手数——实战中手记反复 ±N 漂移)；且**补测轮/差异复现轮的 log 要单独计、不并入主轮再"另计"一次**(否则同一批请求被双重计数、指标虚高)。**产出落盘后必回读自查**：grep 残留的工具调用标记(`</content>`/`</invoke>` 等)、抽核引用的源码行号是否随代码漂移(执行时现场重核、别照抄第②步可能已过时的行号)——"写后回读"铁律同样适用于本步自己的产出。
> 审计方法、判定标准、可复用脚本范式见 `references/audit-and-honesty.md`。

## 10. 测试报告产出（对齐官方 + 诚实口径）
- **骨架**沿用第②步面/维度 + 官方《测试报告》结构(前言/概述/范围/环境/过程/结果/评价结论)；逐用例记录用官方《测试记录单》密集表样式(操作|期望|实际含证据|判定)。
- **官方记录单 1:1 对照表**：官方N条 vs 本方真实执行结果——凸显"代码+浏览器+接口+DB 四证" 相对"纯文档走查" 的增量价值(安全/数据正确性/健壮性)。
- **失败用例**单独成 `_失败用例/` 专夹(按面分)；**缺陷台账**(分级+TC+证据+影响)；**进度跟踪表**(各模块 用例数/真测/源码佐证/阻塞/截图覆盖)用**诚实口径**。
- **★第②步漏项反哺清单 `_第②步漏项反哺清单.md`**：第③步穷举可操作元素时"页面上存在、第②步用例集却没写"的控件/交互，逐条登记（控件/交互 + 所在页/角色 + 第③步补测方式与证据 + 建议补成的 TC + 追溯）并**反哺回第②步**，闭合"第②步可能漏写"的回路——既证明第③步不是只闷头补测、也让第②步用例集可被增补。格式见 `references/report-structure.md`、闸门见 §11。
- **★测不了项单独成专项 `_无法测试项说明/汇总.md`**：环境/物理受限真测不了的用例集中成册，**逐条举例说明"为什么测不了 + 缺什么才能测 + 源码佐证逻辑"**（外部 ASR/LLM、无真实麦克风、真机/多浏览器内核、破坏性全表清空、注入式弱网工具、schema 不可构造 FK、性能压测等）。**铁律——绝不剔除**：这些条目**仍保留在各面执行记录里**（标"阻塞+原因"、计入该面分母），专项是**额外汇总+举例**，两处并存、条数一致；**绝不为覆盖率/真测率好看，把没测的从各面（面1/2/3…）删除或藏进专项（或反之只放专项、各面留白）**。格式见 `references/report-structure.md`。
- **执行真实性与截图覆盖审计报告**(§9 产出)作为报告的诚实性背书。
> 产出落在被测项目工作目录的 `第三步-执行测试/`(或 `./test-execution/`，**不写死绝对路径**)，镜像第②步面/模块结构。完整结构见 `references/report-structure.md`。
> **隶属统一根（供第0步总管扫描续跑）**：本步产出隶属测试三步走统一根 `测试产出/`（见 step0 §6）；**《执行进度跟踪.md》(各面/模块 真测/源码佐证/阻塞/截图覆盖) + 真测率 + 《_第②步漏项反哺清单》《_无法测试项说明/汇总.md》两专项**就是第0步总管 `testing-step0-intake` 判定本步"进行中/续跑点(从哪个面/模块/TC 接着跑)"的信号——**请确保进度跟踪表如实落盘**。默认子目录 `第三步-执行测试/`。**仍可用别名根/子目录（如 `./test-execution/`，原选项保留）**，但用别名时**必须在《进度总览.md》登记真实路径**，否则总管扫不到。
> **★中断与续跑契约（长任务常中断，必须能被接住）**：
> - **《执行进度跟踪.md》随执行增量落盘**：**每条 TC 断言记录完成即更新一次**（不要攒到收尾才写）——否则中断时它根本不存在、续跑者只能反推。汇总表如实标"随执行增量更新"，不预占未执行行。
> - **中断时现场保全 + 残留登记**：写库类执行中途若中断，把**已造数实体的精确ID清单 + 已执行到第几条**留在执行记录头部(或单独 `_中断现场.md`)，供续跑者/总管只读复核——别让残留实体ID只存在脑子里。残留处置口径(子→父精确 id 清理)随执行记录走，不只写在第②步用例文里。
> - **续跑者侧规程**：承接中断现场时——① **接手前先归档中断时原稿**(改名留档，别直接覆盖写，否则上轮记录被冲掉、无法自证)；② 只读复核残留与台账一致后再决定复用残留(活体前置)还是清理重建，**留痕处置过程**；③ 上轮已执行 TC 若采信其 log 不重跑，**必须显式披露"非本会话亲测、采信上轮证据"**，不冒充本轮真跑；④ 收尾后回写《进度总览》(或提示总管更新)，闭合续跑回路。

## 11. 完成度闸门（可计量，禁纯自答）
> **★形态适用性（见 §0.1）**：下列标 **(前端相关)** 的闸门仅在"有面向用户的前端"形态适用；**纯 API/无前端形态下整组标 N/A**，改以下面的 **(API 形态主闸门)** 替代——但 N/A 必须显式登记形态、不得用来逃避能真跑的项。**(条件触发)** 的闸门仅当被测系统命中该特性（富交互/实时通道/并发语义/上游有产出）时适用，没有则一句话整体豁免、不卡首轮。
- [ ] **API 类完全真跑** **(API 形态主闸门：纯 API/无前端形态下这是主闸门、强化适用)**：每个 API 端点用例 接口真发请求(状态码+文案原文)+写端点 DB 写后回读 = N/N 真测；**源码佐证条数在 API 类 = 0**（除非运行期确不可触发——**须给客观判据：已尝试构造前置到何种程度仍触发不了、并留痕**，不得把"需复杂前置/依赖外部回调"轻易归入"确不可触发"来逃避真跑）。**纯 API/无前端形态下，每条声称还须配真实请求-响应报文 + schema/契约校验 + 端点检索证据（代码 file:line）。**
- [ ] **并发/幂等正确性已真跑** **(条件触发：被测系统有并发/幂等语义——超卖/扣减/支付回调/状态流转/唯一约束)**：用少量并发请求(`Promise.all`/幂等键重放)真发 + 并发后 DB 回读最终一致/守恒断言；**这类按功能正确性计入真测分母，绝不归入"留③性能/阻塞/源码佐证"**（只有 N 用户吞吐/时延才属性能留③）。无并发语义的系统此条豁免。
- [ ] **面4 文档符合性浏览器图文并茂** **(前端相关；纯 API/无前端形态改为"每条声称配真实请求-响应+schema/契约校验+端点检索 file:line"，浏览器截图项 N/A)**：每条文档声称 = 真实网页截图 + 位置说明 + 判定；**逐条审计已完成**(§9)、截图与被测点对应。
- [ ] **UI 范围已询问用户** **(前端相关；纯 API/无前端形态 N/A)**：运行前 AskUserQuestion 确认"全跑/部分跑"；按确认范围执行，真跑配专属截图，未跑诚实标注。**无交互/subagent/范围已由上游预答或书面授权时**：按确认范围执行 + 在产出登记"范围据上游预答/授权"即视为满足本闸门(不强制现场弹问)——但**既不卡死、也不得借此私自缩水**(上游没指定则按全跑从严)。
- [ ] **写后回读 + 真实数据还原**：每个写端点至少一条"写后回读确认真入库"已执行；**对真实业务实体的改动已还原并回读确认**；DB 回基线**按 QA 前缀/独立标记列精确计数**(QA残留=0、非全表计数)。
- [ ] **⚠预期缺陷用例**：真发请求/真操作复现(非只读代码)，缺陷入台账。
- [ ] **执行后审计已做**(§9)：真测率、UI 截图覆盖率有真实分子/分母；每模块有反例核查；缺口已回填(无"✅"假象)。
- [ ] **防夸大自检**：随机抽 5 条标"✅通过"的，核其证据是真测(浏览器/接口/DB)而非源码佐证；抽 3 条"有截图"的，核其是专属、对页、对题(非借用)。
- [ ] **官方记录单**(若有)：X/X 逐条映射本方真实执行结果。
- [ ] **能测必测**：凡能程序化真测的用例（接口可发/前端函数可复刻/浏览器可触发）真测率趋近 100%；**源码佐证仅限运行期物理受限或纯前端无 API/函数可验**——抽查若发现"能 API/函数测的却只看代码"=未过闸门、补真测。
- [ ] **跨模块联动 N/N 已执行** **(条件触发：第②步产出了《跨模块联动与变更影响》套件/《联动点清单 N》；无则一句话 N/A、不卡首轮)**：分母 N = 读第②步《联动点清单》；每条联动用例**真执行**——**改A 真发请求/真操作 → 去 B/C/D 下游各自真发请求/真操作取证（每段含机器可核对锚点：状态码/文案原文/入口可见性/数据范围）→ 回退后回读确认下游随之复原**；**源码佐证不抵真执行**（联动恰最不能只读码）；联动套件**独立统计 N/N、不并入也不漏出面/维度矩阵分母**（防整目录被静默漏跑）；空缺显式标原因（阻塞/留③）。
- [ ] **UI 核心已浏览器真测 + 核心清单经确认 + 角色前端可达** **(前端相关；纯 API/无前端形态 N/A，角色判定改以"该角色凭据真实响应+数据隔离回读"为权威)**：用户选"只测核心"时，**核心清单已与用户确认/回填**（非执行者私自缩水）、核心交互逐条浏览器真操作+专属截图（非常完善）；非核心**优先 API/函数真测**、仅纯前端无 API 者源码佐证；**按角色判定的功能已用该角色浏览器真登录、验证前端可进入到达目标界面（后端 200 不单独判该角色通过）**。
- [ ] **可操作元素穷举真操作**（防非点击交互靠人指出才补）**(前端相关；纯 API/无前端形态 N/A)**：每界面对照 `browser_snapshot` 列"可操作元素清单"，**已真操作清单 vs 可操作元素清单 = 覆盖率 + 漏操作清单**；**非点击交互（拖拽绘制/悬停放大/滑块/快捷键/右键/滚轮/拖放/弹窗内控件/二次确认）= 条件触发**：控件确实支持该交互时逐个**真实操作过**（非源码佐证、非只截图、命中即必测、严格度一丝不减）；**本页/本形态根本无此类交互时一句话整体豁免**（如表单-列表类系统）。审计硬反例：**snapshot 可见该可操作元素（且其支持该交互）、但执行记录无任何真操作/截图证据 = 漏测**（§9）。
- [ ] **第②步漏项反哺闭环**（防"漏写只补测、不回流第②步"）**(前端相关；纯 API/无前端、薄 UI 形态对 UI 控件这部分降为推荐——契约/端点的漏写仍按 API 形态反哺)**：第③步现场新发现（`browser_snapshot` 可见、却不在第②步用例集）的可操作元素/交互，已按去向全部归位、可追溯反哺回第②步：**反哺清单条数（能测·已补测的漏写，登记进 `_第②步漏项反哺清单.md`：控件+所在页/角色+补测方式与证据+建议TC+追溯）+ 新发现但测不了项条数（分流进 `_无法测试项说明/汇总.md` 并标"第②步亦漏写"）== 可操作元素穷举发现总数 − 第②步已有 TC 覆盖数**——即"页面上存在却不在第②步"的发现按"能测/测不了"两去向加总闭合、无漏算；"能测"的当场补测且登记反哺（无漏写则显式记"0 项、第②步无遗漏"）。
- [ ] **测不了项专项 + 不剔除核对**：环境/物理受限项已集中成 `_无法测试项说明/汇总.md` 并**逐条举例"为什么测不了 + 缺什么才能测"**；**按 TC 编号计数**（跨面同一物理受限项：各面各记其 TC、专项合并 1 行并列出涉及的各面 TC），且**专项覆盖的 TC == 各面执行记录里标"阻塞/留③"的 TC**（无丢条）——证明没测的两处并存、各面未剔除、未为覆盖率删/藏。
- [ ] 阻塞/留③项**诚实标注**(环境受限原因)，不假装覆盖。

## 12. 反模式（务必避免）
❌ **API 用例只读代码判"通过"、不真发请求** → 没执行，违 §1 硬约束。
❌ **面4 文档符合性用源码佐证替代浏览器截图** → 违 §1，文档符合性必须眼见为实。
❌ UI 默认缩水(只跑一部分)却不问用户、不标注 → 静默夸大。
❌ **把"源码佐证"记成"✅已执行"、把"共享/借错截图"记成"专属图"** → 造假，违 §8。
❌ ⚠预期缺陷用例只读代码就说"已复现"(尤其越权写/匿名删，用带权请求冒充) → 假阳性。
❌ 写操作只验 200/toast、不 DB 回读 → 测不出"是否真入库"。
❌ 执行完不做对抗审计、自我宣布"全执行/图文并茂" → 夸大失真(本 skill 因此而生)。
❌ 破坏性端点复用真实 ID、写库不清理/不回基线 → 污染部署库。
❌ 浏览器多 agent 并行抢会话、卡死时 kill node 杀掉 MCP → 编排事故。
❌ **能用背后 API/请求函数 或 前端纯函数复刻验证的 UI 用例，却只读代码判"通过"** → 违"能测必测"，应优先 API/函数真测。
❌ **UI 选"只测核心"时把核心也只源码佐证、不调浏览器截图** → 核心必须浏览器真操作+截图+完善。
❌ **面4 省截图 / 只截图不做 API 函数检测 / 不逐条审计** → 面4 是三重最高标准，任何一重缺失都不合格。
❌ **为覆盖率/真测率好看，把测不了的用例从各面删掉或藏进专项（或反之只标各面不汇总）** → 删/藏没测项=造假，违"不剔除"铁律；必须各面保留+专项汇总两处并存、条数一致。
