# 防夸大判定标准 & 执行后对抗性审计

> 本文件是本 skill 的"良心"。它存在的原因：一次真实执行里，"读代码推断"被包装成"已执行✅"、"共享/借错截图"被包装成"图文并茂"，自报"~93%全执行"，独立审计后真测率仅 68.9%、UI 专属截图仅 7.6%。**列出条目 ≠ 真跑；看起来对 ≠ 测过了。**

## 一、每条用例的"执行方式"四分类（判定列必须让人一眼看出真跑没有）
| 分类 | 定义 | 能否记"✅已执行" |
|---|---|---|
| **真测** | 浏览器真操作 / 接口真发请求 / DB 写后回读 — 真跑过，有运行期证据 | ✅ 可 |
| **源码佐证** | 只读代码 file:line 推断，**没真跑** | ❌ **必须标"源码佐证(未真跑)"**，不得记✅ |
| **阻塞** | 环境受限未执行(无麦克风/断流/多内核/性能压测/破坏性) | ❌ 标"阻塞+原因+源码佐证逻辑" |
| **待补** | 漏跑/未补 | ❌ 标⏳ |

**截图三分类**（UI/面4 用例）：
| 分类 | 定义 | 能否算"图文并茂" |
|---|---|---|
| **专属截图** | 本用例、本页面、本操作态的截图 + 位置说明 | ✅ 可 |
| **共享截图** | 复用他用例/他页截图 | ⚠ 必须标"(共享 TC-X)"；**借错页面=造假** |
| **无图** | 仅源码佐证，无截图 | ❌ 不算有图 |

## 二、执行后对抗性审计（强制，§9）——防自我美化的核心
执行完**不能自我宣布完成**，必须独立审计。推荐派独立子代理(subagent)并行(每模块一个，纯文本+代码 grep，不开浏览器；三家终端均称 subagent，支持 workflow/编排的环境用其更强并行)：

每个审计 agent 对一个模块：
1. 读第③步执行记录 + 第②步用例文件(数出分母 totalTC)。
2. 逐条分类：realTested / srcOnly / blocked / pendingNoEvidence（四数加总应≈totalTC）。
3. UI/面4 模块再分：uiWithOwnShot / uiSharedShot / uiNoShot。
4. **举 3-5 个反例**(防美化的关键)：标✅却实为源码佐证的 TC、涉 UI 操作却无专属图的 TC、借错页面的 TC、⚠缺陷只读代码当"已复现"的 TC。
5. schema 强制返回精确数字 + 反例 + honestVerdict。

综合 agent 汇总：全局真测率 = Σ真测/Σ分母；UI 截图覆盖 = Σ专属/ΣUI分母；**用真实分子分母回答"是否全执行/是否都有图"，戳破夸大**。

**★汇总计数三条防失准纪律（自检本身也会出错——实测三例：复现轮被双重计数、产出残留 `</content>` 工具标记、引用行号 ±2 漂移，全是自检没抓到、靠外审才发现）：**
- **隔离补测/复现轮**：差异复现轮、补测轮的 log **单独计**，**不能并入主轮 grep 再"另计一次"**——否则同批请求被双重计数、请求/断言数虚高(实测某次主轮 35 被报成 40)。grep 客观计数时明确"主轮 N + 复现轮 M = 全量"，别让 `--- 请求` 跑在含复现轮的全部 log 上还另加复现轮。
- **图片引用计数要排除字面记号**：grep `![](...)` 统计断链/图覆盖时，**审计正文里举例写的 `![]()` 字面记号会被误计**(实测把 12 处图报成 14/14)——按真实 markdown 图片引用计，复核数字。
- **落盘后回读自查**：产出文件写完**回读一遍**：grep 残留的工具调用标记(`</content>`/`</invoke>`)清掉；抽核引用的源码 `file:line` 是否随代码漂移(执行时现场重核、别照抄第②步可能已过时的行号)。"写后回读"对本步自己的产出同样适用。

### 审计脚本范式
**覆盖核查**（第②步用例集 vs 第③步执行记录的 TC 比对，归一化防编号格式差异）：
```js
// 关键：②用 API-LOGIN-002 / ③用 TC-API-LOGIN-002，机械比对会假低 → 归一化去前缀
function norm(tc){ return tc.replace(/^TC-/,'').replace(/^UI-/,'').replace(/^API-/,'').toUpperCase(); }
// 提取两侧 TC 集合，算覆盖率 + 列未覆盖
```
**判定行数 vs 用例数**：最硬的"是否逐条"信号——执行记录判定行数≈用例数=逐条；远小于=仅关键项/抽样。
**截图断链校验**：扫所有 `![](路径)` 验文件存在，0 断链才算图文并茂落地。
> ⚠ 关键词统计(grep "源码佐证"/"实测")只能给方向，**会混算执行记录与追溯表、错配多列**；精确分类靠 agent 读懂语义。两者结合：脚本给客观计数(断链/判定行数)，agent 给语义分类(真测 vs 源码佐证)。

## 三、面4 文档符合性的逐条审计（三重最高要求，不打折）
面4 必须**一条一条看过去**，每条同时满足**三重**：
- **① 浏览器图文并茂**：每条文档声称都有**真实网页截图**佐证(不是源码佐证)；判定与截图**一致且对应**(不拿"合法态截图"冒充"脏数据兜底"、不拿"用户注册页"冒充"登录页")；判"无"也要有"网页无此入口"的截图。
- **② API/函数真检测**：对每条声称**再做接口真发请求 / 前端函数复刻执行**，与截图互证(如"导出PDF"既截图又真发导出接口/查产物)。
- **③ 逐条审计完成**：逐条复核、无遗漏(对照第②步原子级核实表的原子总数)。
- **面4 不受"只测核心"影响，永远全量三重最高标准**；缺任一重(省截图/只截图不API检测/不逐条)都不合格。

## 四、诚实纪律（被追问时如实拆分）
- 区分"真测 / 源码佐证 / 借用截图 / 阻塞"，被问"是否全测完"时给真实分子分母，不含糊。
- **绝不把"读代码推断"说成"已执行"、"列出条目"说成"逐条执行"、"共享截图"说成"专属图文并茂"。**
- 修正夸大表述要落到文档(执行记录改标、进度表改口径、出审计报告)，不只口头认。
- **★不剔除原则（没测的不删不藏）**：环境/物理受限"测不了"的用例，**绝不为覆盖率/真测率好看从各面(面1/2/3…)删除或藏匿**；必须**如实留在各面执行记录**(标"阻塞+原因+源码佐证逻辑"、计入该面分母)**且额外集中到 `_无法测试项说明/汇总.md` 逐条举例"为什么测不了"**，两处并存、条数一致。审计核对"专项汇总条数 == 各面阻塞条数"，丢条=不诚实，回填补齐。
- **★能测必测原则**：能 API/函数/浏览器真测的用例都必须真测；审计时抽查"标✅但实为源码佐证"，尤其**能 API/函数验证却只看代码的 UI**——发现即补真测，不得以源码佐证冒充。
- **★并发/幂等正确性不得误归性能留③**：防超卖/重复扣款/支付回调幂等/丢失更新等**最终一致性缺陷是功能正确性、必须真跑**（程序化少量并发请求 + 并发后 DB 回读守恒/最终态断言）；**审计时若发现这类被标成"留③性能/阻塞/源码佐证·未跑"，即判违规、回填补真跑**——只有"N 用户吞吐/时延"才允许留③性能。
- 真测扎实的部分(如 API+DB 写后回读)也要如实肯定，不矫枉过正一律否定。

## 五、典型夸大反例（自查清单，命中即改）
- API 用例标"通过"但只读了代码、没发请求 → 改"源码佐证"或补真跑。
- ⚠越权/匿名缺陷标"已复现"但用的是带权请求 → 假阳性，补真发匿名/越权请求。
- 写操作标"通过"但没 DB 回读 → 补回读或降级。
- UI 用例标"图文并茂"但截图是借的他页/他用例 → 标共享或补专属图。
- "逐条覆盖追溯 0 漏"被当成"逐条真跑" → 澄清：条目无遗漏 ≠ 逐条真执行。
- 进度表"~X% 全执行" → 换成"真测 a% / 源码佐证 b% / 阻塞 c%"的诚实口径。
- **能 API/请求函数 或 前端纯函数验证的 UI，却只读代码判"通过"** → 违"能测必测"，补 API/函数真测。
- **测不了项被从各面删掉/藏进专项(或只标各面不汇总)** → 违"不剔除"，回填使两处并存、条数一致。
- **并发/幂等正确性（防超卖/重复扣款/幂等去重/丢失更新）被标成"留③性能/阻塞·未跑"** → 误归性能、漏掉最致命的功能缺陷；改为程序化并发真跑 + DB 回读最终态断言（只有 N 用户吞吐/时延才属留③）。
- **面4 只截图没做 API/函数检测、或省了截图、或没逐条审计** → 违面4 三重标准，补齐。
- **(条件触发:第②步有联动套件) 跨模块联动套件整目录被跳过没执行 / 只改A没真去下游 B/C/D 验证 / 回退段无回读确认复原 / 联动写成"改A看一眼没报错"空壳 / 联动 N 报低报零却①步《业务闭环图谱》里有依赖** → 联动被静默漏执行或跑成空壳，违"全部能测的真执行"；审计须把第②步《联动点清单 N》作**独立分母**（不并入 totalTC）逐条核、补真执行。
