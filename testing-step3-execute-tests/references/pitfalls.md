# 执行避坑（实战血泪教训）

> 下列实例来自一次真实项目实战，**原则通用、专名仅为举例**——套用到任意 Web 项目时把业务专名换成你的即可。

## 浏览器 / MCP
- **UI agent 会在 MCP 层 hang 卡死**：媒体(getUserMedia)、canvas 拖拽、长等待操作会让 Playwright MCP 阻塞，**agent 层的"15s 超时"拦不住 MCP 层 hang**（曾卡死 53min / 3.7h）。对策：① 浏览器密集模块**主控亲自串行驱动**（单次 hang 能感知、换策略）；② 用 agent 则**小粒度 bounded**（每≤30条 + "单操作>15s 标阻塞跳过、不 churn" + 每 5-8 条增量落盘）。
- **恢复卡死的浏览器：只 kill `chrome.exe`，绝不 kill `node`**——MCP server 是 node 进程，误杀会断开所有浏览器工具，需用户 `/mcp` 重连。卡死的 background agent 用 `TaskStop` 停。
- **浏览器是单一有状态会话**：UI/面4 类必须串行，多 agent 并行会互相覆盖页面。后端(接口/DB/代码)无此限，可并行。
- **合成事件普遍不触发真实 handler（不只 canvas）**：`dispatchEvent`/合成 PointerEvent 不走 React 受控组件/Konva/拖拽/滑块/悬停的真实事件管线（konva 派发后 shapeCount 不变、combobox onChange 也不触发）。**一切非点击交互必须用真实指针轨迹/真实输入工具**（`page.mouse` 逐点轨迹 / browser_hover / browser_drag / select_option / press_key，见 execution-channels 通道1.6）；**canvas/拖拽默认用 page.mouse 真实轨迹真测，禁止默认标"源码佐证·待复核"**——仅确证 MCP hang 卡死且小粒度/主控串行仍不可行时才标阻塞进专项。（画布自由绘制/区域缩放就是被旧的"默认标源码佐证"兜底漏掉的。）
- **门禁/上下文**：需上下文的页面(如某页需某业务 ID/会话态)走真实业务流进入，别硬闯 URL。

## 截图
- **文件名禁含英文括号 `()`**：markdown `![](路径含))` 会在第一个 `)` 处截断 → 断链。用 `-` 替代(`+`、中文括号不影响，但仍建议简洁)。
- **截图默认落 cwd 根目录**（非 .playwright-mcp）：截完用脚本移到 `证据/面X/`，并**即时校验落盘**(曾因 bash mv 报错丢了 3 张图却没察觉)。**注意截图实际落的是 MCP server 进程的 cwd，可能 ≠ 你执行 shell 的 cwd**——**先截一张测试图、全盘定位它真正落在哪个目录**，再批量截+移，别假设落在当前目录(否则移图脚本扫空)。
- **产出后必跑断链校验**：扫所有 `![](路径)` 验文件存在，0 断链才算图文并茂落地。
- **路径深度**：执行记录在 `面X/子模块/` 两层 → `../../证据/面X/`；实证文档在 `面X/` 一层 → `../证据/面X/`。

## 脚本 / shell（环境无关原则）
- **用脚本文件而非超长内联命令**：把多行/含中文/含转义的逻辑写成 `.mjs`/`.py` 脚本文件再执行，别用长 `node -e`/内联——避免各 shell 的转义/编码/引号踩坑（变量名别用中文，部分解析器会把 `const面1dir` 当一个标识符解析失败）。
- **后台任务先落盘再返回**：后台命令/agent **先把文件写完落盘(主交付物)、分批 Write、最后只返回极简摘要**——background agent 常在收尾(写文件+返回)时 socket 断连、活干了文件没落盘全丢；完成后 `ls`/`wc -l` 确认真落盘，没落盘重跑。
- **多语句易碎**：`;;`+单引号混用在某些 shell 易碎，复杂查询分多次调用。

## Windows / Docker 环境注意（若你也在此环境）
> 以下是 **Windows + git-bash + Docker Desktop + Node** 这一具体环境的特化贴士；其它环境（Linux/macOS、托管/远程）按对应方式调整，不必照搬。
- **git-bash 剥反斜杠/中文易碎**：`node -e` 内联在 git-bash 下尤其易碎 → 写 `.mjs` 文件再 `node x.mjs`。
- **DB 查询 `docker exec` 前加 `MSYS_NO_PATHCONV=1`**(防 git-bash 改写路径)；`-i` + `< dbq.py` 传脚本。
- **★中文前缀经 `docker exec ... -e "...LIKE '中文%'"` 会编码损坏(致命假阴)**：Windows 宿主 shell→容器传中文参数常乱码，`LIKE 'QA小测0610%'` **匹配不到刚插入的行**(而 `COUNT(*)` 总数却照样 +1)——若用它"按前缀回基线/清残留"会得到**假阴残留=0、漏清**。对策：① 造数时**额外带一个 ASCII 安全标记**(如编号/mrn 列写 `QA…0610`)，回基线/清理按 **ASCII 子串** LIKE；② 或用 **HEX**(`LIKE CONCAT('%',UNHEX('…'),'%')`)/**精确主键 id** 双查；③ 把含中文的 SQL **写进 `.sql`/`.py` 脚本文件**(`-i < file`)而非 `-e` 内联，减少 shell 编码损耗。务必用"已知能命中"的对照(如某真实中文串)验证扫描方法有效、排除假阴，再下"残留=0"结论。
- **Docker Desktop 自启**：电脑重启后 Docker Desktop 需启动、容器多按 restart 策略自启（详见下方"环境"小节）。
- **后端解释器路径**：容器内调后端虚拟环境解释器（如 `/app/.venv/bin/python`）按镜像实际路径调整。

## 写库安全
- **★破坏性端点误删真实数据(EXEC-NOTE)**：曾对某批量/级联删除端点**复用真实主数据 ID**，连带把它的子资源级联删除，**不可恢复**。守则：破坏性端点只对**完全自建的隔离 QA 链**操作，**严禁复用任何真实业务 ID**；无法隔离的破坏体(全表清空类)只验鉴权 403 对照、不执行破坏主体。
- 造数 QA_ 唯一前缀 → 写后回读 → 子→父清理 → DB 回基线(QA残留=0、计数复原、无孤儿)。
- **★清理/删除前必须只读 dry-run 确认精确集合(血泪教训)**：任何 DELETE(含清 QA 残留、清子表孤儿)前，先 `SELECT` 查出**将删的精确行数 + 样本**、确认确实是 QA/目标数据再删；**机械验收线：核对所有待删行均带 QA 标识或在已知 id 集合内，抽 K 行——任一行无 QA 标识立即中止、转人工复核**(曾误删数百条真实业务记录,就因没逐行验收就盲删)；**禁止用模糊/隐式类型转换的比较来批量删除**——优先按**明确 QA 标识/已知主键 ID** 删,不靠模糊比较(例：MySQL 跨表 collation 不一致——子表 `utf8mb4_unicode_ci` vs 主表 `utf8mb4_0900_ai_ci`——会让 `CONVERT`/隐式转换的 `NOT EXISTS` 把真实数据误判为孤儿删掉)；删后核对目标表回基线。
- **大规模并行写库风险高**：多 agent 并行造数易留残留(失败 agent 不清理)、清理时跨模块互相干扰基线判断、且事后批量清理本身易误删。**写库类用例改主控亲自串行**(逐个造数→回读→立即清,小批可控、每步验证);只读/鉴权类才并行。
- **共享部署库批量 raw-SQL/关 FK 写入**会被安全机制拦截(理应如此)——这类高危写需用户显式授权,且应是最小范围(只动目标表、用 `INSERT IGNORE` 不覆盖现存)。

## 诚实性 / 记录
- **反向漏报也是不诚实**：实际跑过且有图的用例，却在执行记录里记成"⏳待补/未截图"(两份记录不同步)——执行记录要及时同步真实状态。
- **不要把"源码佐证/共享截图/列出条目"包装成"已执行/专属图/逐条真跑"**(见 `audit-and-honesty.md`)。

## 环境
- **重启后先确认被测系统可达 + 数据基线完好再继续**（环境无关原则）：机器/容器重启后，**先确认后端服务可达、DB 基线完好**再继续测，否则会把"环境没起好"误判成缺陷。注意后端实连的 DB host(可能是宿主 IP 如 `<宿主IP>:3306` 而非某个 Exited 的 db 容器——以后端能查到基线为准)。（Docker Desktop 自启等 Windows/Docker 特化细节见上方"Windows/Docker 环境注意"。）
- 性能/压测/真机/多浏览器内核/外部依赖(第三方 ASR/LLM 等不可达服务)项：标"阻塞+环境受限"，诚实标注、源码佐证逻辑，不假装执行。
- **性能/规格指标判定（别把可静态判的也一律标阻塞）**：① **能静态判**的——"有无并发上限/限流配置""有无最大上传尺寸/分页上限"等，grep 配置/中间件直接判，不算阻塞；② **需计时/压测**的——响应≤Xms、时延≤Xms、登录/报告耗时≤Xmin、并发 N 用户，标"留③性能实测"；③ **依赖真机/外设/外部服务**的(第三方 ASR/LLM 等不可达服务、真实外设、多浏览器内核)，标"阻塞+环境受限"。只有 ②③ 才标留③/阻塞。

## 真跑常见执行意外（API/UI 真跑必然遇到，不防会污染结论/浪费时间）
- **token 过期/会话失效**：量大时一次取的 token 中途过期 → 后半段全 401。遇 401 **先排查是不是自己 token 失效**（重新登录重取再试）再判缺陷；区分"401=越权拦截正确"与"401=自己 token 过期"。登录脚本可定期重取 token。
- **限流 429 / 账户锁定**：穷举式高频连发（每字段边界+鉴权三态）易触发 WAF/限流/锁定 → 429/403/验证码。**把 429/锁定识别为环境保护机制、不是端点缺陷**；被限流后的请求结果不可信，须退避节流后重跑；登录暴力边界用例**隔离执行**避免锁定影响全套登录。
- **时区 / 跨日**：容器 UTC vs 本地时区、或执行跨午夜 → "按 created_at 当天查""查今日新建"回读不到刚写的行被误判未入库。**统一时区基准**、跨日时用绝对时间范围而非"当天"。（本项目曾因 created_at 跨到次日 00:20，按 DATE='当天' 查不到自己造的数。）
- **重跑造数唯一键冲突**：QA 前缀只到模块级、跨次重跑遇唯一约束（身份证/编号/name）冲突 → INSERT 报错。造数用**实例级唯一后缀（时间戳/uuid）**；留意清理后自增/序列残留对"行数/ID 连续"类断言的影响。
- **外部依赖偶发可用（flaky）**：外部 ASR/LLM 等第三方服务时通时断 → 同用例两次结果不同。依赖外部服务的用例**多跑几次取结果分布、标 flaky、记跑了几次**，不把一次偶成记成稳定通过、一次偶败记成确定缺陷。
- **报告图片移动后断链**：embed/mv 生成嵌图后若文档目录层级变动，相对路径整体错位。**目录结构变动后必须复跑断链校验**（不能只在生成时跑一次）。
