# 报告与产出结构

> 产出落在被测项目工作目录的 `第三步-执行测试/`（或 `./test-execution/`，**不写死绝对路径**），**镜像第②步的面/模块结构**，让第②步用例与第③步执行 1:1 对应。

## 目录结构
```
第三步-执行测试/
├── 00-测试报告.md                 # 主报告，对齐官方《测试报告》结构 + 评价结论
├── 01-测试环境与说明.md            # 四通道、账号、DB 基线、护栏守则(破坏性端点规则)
├── 02-官方记录单覆盖对照(执行结果).md  # 官方N条 vs 本方真实执行结果(若有官方记录单)
├── 执行进度跟踪.md                 # 各模块 用例数/真测/源码佐证/阻塞/截图覆盖(诚实口径)
├── 缺陷清单.md                     # 缺陷台账(分级+TC+证据+影响)
├── _执行真实性与截图覆盖-诚实审计.md  # §9 审计产出(真测率/截图率/反例)——诚实性背书
├── 面1-UI功能与闭环/<模块>/<模块>-执行记录.md
├── 面2-后端接口与数据/...           # 各面镜像第②步,密集执行表
├── 面4-文档符合性(独立)/实证-<源文档>.md  # 带截图实证(每条文档声称配真实网页截图)
├── 面5/面6/面7/...
├── _失败用例/面1-7/<模块>-失败.md    # 失败/缺陷用例专夹(用例多时按面分)
├── _第②步漏项反哺清单.md           # ★第③步现场发现、第②步漏写的可操作元素/交互逐条登记,反哺回第②步(闭合"第②步可能漏写"回路)
├── _无法测试项说明/汇总.md           # ★测不了项专项:环境/物理受限未执行的用例集中成册+逐条举例"为什么测不了"(各面不剔除,此处仅额外汇总)
└── 证据/面1../面X/*.png             # 截图(文件名禁英文括号)
```

## 执行记录表（逐用例，对齐官方《测试记录单》密集表样式）
`| TC编号 | 操作 | 期望(可断言) | 实际(含证据:截图![]/请求-响应/DB回读/file:line) | 判定 |`
- 判定明确：✅通过 / ❌不通过 / ⚠通过(暴露缺陷) / 阻塞 / 源码佐证(未真跑) / ⏳待补——**让读者一眼看出真跑没有**(防夸大 §8)。
- 实际列附**真实证据**：截图内嵌 `![](../../证据/面X/xx.png)` + 位置说明、接口真实状态码+文案原文、DB 回读 JSON、必要时 file:line。
- ⚠预期缺陷用例：实际列写"真发请求/真操作复现了该缺陷"(非只读代码)。

## 00-测试报告.md（对齐官方《测试报告》）
前言 / 概述 / 测试范围(面×维度) / 测试环境 / 测试过程(四通道+编排) / 测试结果(各面通过/不通过/阻塞统计，**诚实口径**) / **评价结论**(总体+核心功能+重大缺陷+与官方报告对照+修复建议) / 测试副作用如实披露(若有)。

## 官方记录单 1:1 对照表
列定义与第②步映射表对齐（`| 官方用例ID | 本方TC编号 | 本方真实执行结果(通过/不通过/阻塞) |`），使②的"官方ID→TC→状态"映射表与③的对照表列一一对应、不错位。凸显增量价值：官方"纯文档正向走查"覆盖不到的 安全/数据正确性/健壮性，本方"代码+浏览器+接口+DB 四证"能发现(如官方标"通过"实则未实现的功能、系统性无鉴权、schema 漂移)。

## 进度跟踪表（诚实口径）
每模块列：用例数 | **真测** | 源码佐证 | 阻塞 | 待补 | 不通过 | (UI)专属截图/共享/无图 | 执行记录链接。
- 合计行用**真实分子分母**：真测率 = 真测/总数，UI 截图覆盖 = 专属/UI总数。
- **禁止"~X% 全执行"这类模糊夸大**；API 类应趋近 100% 真测，UI 类按用户确认范围如实报。

## _无法测试项说明/汇总.md（★测不了项专项，不剔除铁律）
**为什么单独成专项**：环境/物理受限"测不了"的用例若只散在各面执行记录的"阻塞"列里，读者难一眼看全"哪些没测、为什么没测、缺什么才能测"；故额外集中成册、逐条举例，接受方一看便知覆盖边界与剩余风险。
**铁律——绝不剔除**：测不了的用例**条目必须仍保留在各面执行记录里**（标"阻塞 + 原因 + 源码佐证逻辑"、计入该面分母），本专项是**额外汇总+举例**，与各面记录**两处并存、条数一致**；**严禁为覆盖率/真测率好看，把没测的从面1/面2/…删掉或藏进专项（或反之只放专项、各面留白）**——删/藏没测项 = 造假（违 audit-and-honesty 不剔除原则）。
**汇总表格式**（逐条，一条都不少）：
`| TC编号 | 所属面/模块 | 受限类别 | 具体原因(为什么测不了) | 缺什么才能测 | 源码佐证逻辑(file:line) | 各面记录链接 |`
**受限类别 + 逐条举例"为什么测不了"**（按项目实际增减）：
- **外部 ASR/LLM 等第三方服务**：依赖外部语音识别/大模型在线服务，环境无可用 key/额度或服务不可达，无法发真实调用看真实返回。
- **无真实麦克风/摄像头等外设**：用例需 `getUserMedia` 采真实音视频，测试机/无头浏览器无真实外设，合成流不等价。
- **真机 / 多浏览器内核**：需 iOS Safari / Android / 多内核真机真浏览器，环境只有单一 Chromium，无法覆盖兼容性差异。
- **破坏性全表清空 / 不可隔离的级联删除**：端点会清空全表或级联删真实主数据且无法只对自建 QA 隔离链操作，执行即不可恢复污染部署库，故只验鉴权 403 对照、不执行破坏主体。
- **注入式弱网/故障注入工具缺位**：需弱网/丢包/限速/中断注入或代理，环境未部署，无法构造真实弱网/中断分支。
- **schema 不可构造的前置（FK/约束无法满足）**：用例需要的前置行受外键/唯一/CHECK 约束限制，不污染真实数据前提下无法合法构造满足条件的链，走不到该分支。
- **性能/压测（吞吐·时延）**：需计时/压测 **N 用户吞吐·响应≤Xms·耗时≤Xmin** 等，属"留③性能实测"。**⚠注意区分：验最终一致性的"少量并发"（防超卖/重复扣款/幂等去重/丢失更新）是功能正确性、必须真跑（程序化并发+DB 回读最终态），绝不归入本类留③/阻塞/源码佐证**——只有"N 用户吞吐/时延"才进本类。
> 每条都要写"具体原因"（不止类别，要写清这条具体卡在哪）+"缺什么才能测"（补什么外设/服务/授权/工具就能测）+ 源码佐证逻辑。**能静态判的（有无并发上限/最大上传尺寸配置）不算测不了，不进本专项**（见 `pitfalls.md` 性能/规格指标判定）。

## _第②步漏项反哺清单.md（★把第③步现场发现的漏写反哺回第②步，闭合回路）
**为什么要有**：第③步执行强度分级里"UI 真测分母 = 第②步用例 ∪ 页面每个可操作元素"——这意味着第③步对照 `browser_snapshot` 穷举时，**必然会撞见第②步用例集没写到的可操作元素/交互**（第②步可能漏了）。这些控件第③步会当场补测（记入执行记录），但**若不同时登记反哺，第②步用例集里这一条就一直缺着**，下次单独复用/交付第②步成果时仍会漏。本清单就是那条回流通道——第③步是"第②步可能漏写"的最后一道发现关。
**铁律——补测必反哺（仅限能测的漏写）**：凡"页面上存在、却不在第②步用例集"的可操作元素/交互，若**能测**，则**当场补测（真操作+截图，记执行记录）+ 逐条登记进本清单**，两件事都做；只补测不登记 = 第②步的漏没闭合，只登记不补测 = 本步漏测。无漏写时显式记"本次 0 项、第②步无遗漏"（而非留空）。
**分流——测不了的漏写不进本清单**：若新发现的漏写控件因环境/物理受限**测不了**，**改入 `_无法测试项说明/汇总.md` 并标"(第②步亦漏写)"**，不进本清单。两专项各管一类、互不混：**本清单 = 第②步漏写且可测（已补测）；无法测试项 = 测不了（含其中第②步亦漏写的）**；二者条数加总 == 第③步穷举发现的"不在第②步"控件总数（见 SKILL §11 闸门公式），无漏算、无重复。
**清单表格式**（逐条）：
`| 序号 | 可操作元素/交互 | 所在页/角色 | 交互方式 | 第③步补测方式与证据(截图![]/API/函数) | 补测结果(功能可用✅/不可用❌即缺陷/部分⚠) | 建议补成的TC编号 | 追溯(执行记录链接) |`
> "补测结果"列填的是**补测后该功能的表现**(可用/不可用即缺陷/部分)，**不是"测得了/测不了"**——能登进本清单的都是已补测过的；测不了的不在此表(已分流到无法测试项)。
> 与第②步衔接：本清单是第②步"下一轮迭代/交付前增补"的**直接输入**——把这些条目按 `case-conventions` 规范补成正式 TC、回填覆盖矩阵，第②步用例集即闭合该漏写。可计量闸门见 SKILL §11"第②步漏项反哺闭环"。**注意**：本清单收的是"第②步漏写、第③步能测且已补测"的项；与 `_无法测试项说明/汇总.md`（环境/物理受限、测不了的项）是两回事，别混。

## 与三步衔接
- 输入 = 第②步用例集(面/维度/TC编号/官方映射/面4原子核实表/⚠缺陷用例/留③项)。
- 第②步 TC 编号 = 本步执行与报告骨架，1:1 对应，不另立体系。
- 面4 文档符合性：第②步《原子级核实表》的每条，本步**三重最高标准**——真实网页截图逐条实证(判有→功能在/判无→无入口/判部分→打折扣) + **API/函数调用真实检测** + 逐条审计完成；不打折、不省截图。
- 测不了项：第②步标"需授权/留③/环境受限"的，本步执行不了的，**各面逐条标"阻塞+原因"(计入分母、不剔除) + `_无法测试项说明/汇总.md` 额外集中举例**，两处并存、条数一致。
