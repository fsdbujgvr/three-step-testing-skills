# UI 用例的逐条覆盖追溯（合并同质重复，但绝不漏）

> 解决一个真实矛盾：UI 模块常有**大量同质重复用例**（同表单不同字段、同组件不同入口、同 statusMap 不同值），逐条每条单截一张图既慢又冗余；但"合并"一不小心就变成"漏测/搞错"。本法 = **逐条列出 + 标注覆盖来源 + 同质共享、异质各自覆盖**，让"合并"可复核、不漏任何不同的功能点。

## 核心原则：列出 ≠ 删减，共享 ≠ 漏掉
- 第②步**每一条 UI 用例都在追溯表里占独立一行**——绝不因"和别条像"就删掉条目。
- **同一功能点的同质重复**（如"姓名/年龄/电话三字段各自的必填校验"）→ 可共享同一张截图/同一执行结论，但**每条仍各占一行**，覆盖来源标注 `(共享 TC-X / 同 Y 截图)`——共享是"指向同一证据"，不是"省掉这一条"。
- **不同的功能点 / 状态 / 边界 / 异常变体**（如"年龄=0 vs 年龄=150 vs 年龄=非数字"是三个不同边界）→ **必须各自有覆盖**（各自截图，或至少各自真跑/源码佐证并标明），**不得合并成一条**。
- 判不准"算同质还是异质"时，**按异质处理**（各自覆盖），宁可多一条，不可漏一个变体。

## 追溯表列式
```
| TC编号 | 文档/用例声称(原子) | 覆盖来源 | 判定 |
```
- **覆盖来源**必须如实写明这条到底靠什么坐实，且与防夸大四分类(`audit-and-honesty.md`)一致：
  - `专属截图 ![](...)` — 本用例本页本操作态的图（最强）
  - `共享 TC-X / 借自 Y 截图` — 复用他条证据（必须标，且**确认借的是对页对题的图**，借错页面=造假）
  - `接口/DB 真测` — 真发请求/写后回读
  - `源码 file:line(未真跑)` — 仅代码佐证，**显式标"未真跑"**
- **判定**：✅可 / ❌不可 / ⚠部分 / 阻塞 / ⏳待补——让读者一眼看出这条真跑没有、有没有专属图。

## 防漏的硬闸门
- **`追溯行数 == 第②步用例总数`**：用脚本/schema 强制（`tracedRows == totalTC`），少一行就是漏。本次实战靠 workflow agent 的 schema 卡死此条，杜绝"列着列着少了几条"。
- **异质变体清单自查**：对每个"看起来可合并"的用例组，列出它包含的不同功能点/状态/边界，逐个确认在追溯表里都有归属。

## 与执行后审计(§9)打通：覆盖核查 → 回填追溯表
两件事必须连起来，否则各自为政：
1. **覆盖核查**（`audit-and-honesty.md`）算的是"TC 编号集合差"——②用例集 vs ③执行记录，归一化去前缀比对，找出**哪些 TC 编号根本没出现**。
2. **追溯表**要的是"每条 UI 用例对应到哪张图 / 哪个执行项 / 哪段源码"——**细粒度归属**。
- 打通方式：**覆盖核查的输出（未覆盖/未真跑的 TC）→ 回填追溯表对应行**（标⏳待补或源码佐证），使"编号无遗漏"和"逐条有归属"两个维度都成立。
- **铁律**："逐条覆盖追溯 0 漏" 指**条目无遗漏**，**不等于"逐条真跑"**——追溯表里大量行的覆盖来源可能是"源码佐证/共享截图"。报告时必须把"追溯行数"与"真测行数/专属截图行数"分开报，不得用前者冒充后者（这是本 skill 防夸大的核心，见 `audit-and-honesty.md`）。

## 编排建议
- UI 追溯是文档密集型（读用例→映射已有证据），**可不开浏览器、扇出子代理(subagent)并行**（每模块一个子代理，强制 `追溯行数==用例总数`；三家终端均称 subagent，支持 workflow 的环境用 workflow + schema 卡死此条）。
- 大模块（上百条）**强制增量落盘**：agent 先写文件骨架（头部+表头）→ **分批 `Edit` append** → 防"攒到最后一次性写"时 socket 断连全丢。**落盘频率按场景**：纯文档追溯 agent（不开浏览器、风险低）可每 ~25 条；浏览器密集执行 agent（易卡死）每 5-8 条（见 `pitfalls.md`）。
- 追溯只解决"覆盖归属与不漏"；**真执行缺口（源码佐证→浏览器真跑）和截图缺口（无图→补专属图）仍需按 §1 强度分级和用户确认的 UI 范围去补**，追溯表如实标注这些待补项，不替代真执行。
